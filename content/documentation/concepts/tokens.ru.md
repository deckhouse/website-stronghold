> Предупреждение  
> Токены представляют собой непрозрачные значения, поэтому их структура не задокументирована и может изменяться. Скрипты и автоматизации, зависящие от внутренней структуры токена, могут перестать работать.

Токены могут использоваться напрямую, либо можно использовать [методы аутентификации](../user/auth/overview/), которые позволяют динамически генерировать токены на основе внешних идентификаторов.

Все внешние механизмы аутентификации привязываются к динамически создаваемым токенам. Эти токены обладают теми же основными свойствами, что и обычный токен, созданный вручную.

Внутри Stronghold токены сопоставляются с набором данных. К наиболее важным из них относится набор прикреплённых [политик](./policy/). Эти политики определяют действия, доступные владельцу токена в Stronghold. Также с токеном связываются метаданные, которые можно просматривать и которые попадают в аудит-логи, например время создания и время последнего продления.

## Формат токенов

Токены состоят из двух частей:

- **префикс** — указывает тип токена;
- **тело** — случайно сгенерированная строка.

Типы префиксов:

| Тип токена | Префикс |
| --- | --- |
| Служебный (`service`) | `s.` |
| Пакетный (`batch`) | `b.` |
| Токен восстановления (`recovery`) | `r.` |

Тело представляет собой случайно сгенерированную строку из 24 или более символов.

Примеры:

```text
b.n6keuKu5Q6pXhaIcfnC9cFNd
r.JaKnR2AIHNk3fC4SGyyyDVoQ9O
s.raPGTZdARXdY0KvHcWSpp5wWZIHNT
```

## Типы токенов

Существует два основных типа токенов:

- служебные (`service`);
- пакетные (`batch`).

Ниже приведена разница между ними.

| Возможность | Служебные токены | Пакетные токены |
| --- | --- | --- |
| Могут быть root-токенами | Да | Нет |
| Могут создавать дочерние токены | Да | Нет |
| Могут продлеваться | Да | Нет |
| Могут быть отозваны вручную | Да | Нет |
| Могут быть периодическими | Да | Нет |
| Могут иметь явный максимальный TTL | Да | Нет |
| Имеют accessors | Да | Нет |
| Имеют `cubbyhole` | Да | Нет |
| Могут быть отозваны через родительский токен | Да | Перестаёт работать |
| Поддерживают динамическое назначение аренды секретов | Самостоятельно | От родителя, если токен не orphan |

### Служебные токены

Служебные токены — это стандартные токены Stronghold, которые поддерживают все основные функции:

- продление;
- отзыв;
- создание дочерних токенов;
- участие в иерархии;
- работу с accessor;
- `cubbyhole`.

### Пакетные токены

Пакетные токены — это зашифрованные blobs, которые содержат достаточно информации для использования в операциях Stronghold и при этом не требуют места в хранилище для полноценного отслеживания.

Они хорошо масштабируются, но не поддерживают многие возможности служебных токенов.

## Хранилище токенов

В документации и справочных материалах можно встретить понятие «хранилище токенов». Оно совпадает с [методом аутентификации Token](../user/auth/token/).

Это особый backend, поскольку он отвечает за создание и хранение токенов и не может быть отключён. Также это единственный метод аутентификации, который не имеет обычной процедуры входа. Для выполнения любых действий требуется наличие уже существующего токена.

## Root-токены

Root-токены — это токены, к которым прикреплена root-политика.

Root-токенам доступны любые действия в Stronghold. Это единственный тип токенов, который может быть настроен так, чтобы срок его действия никогда не истекал без необходимости продления.

Существует только три способа создания root-токенов:

- во время выполнения команды `d8 stronghold operator init`;
- с использованием другого root-токена;
- с помощью команды `d8 stronghold operator generate-root`, для выполнения которой требуется кворум держателей ключей для распечатывания хранилища.

> Важно  
> Root-токены следует использовать только для начальной настройки и аварийных административных действий. В production-окружениях рекомендуется ограничивать их использование и отзывать исходный root-токен после завершения базовой настройки.

## Иерархия токенов и orphan-токены

Обычно, когда держатель токена создаёт новые токены, они создаются как дочерние по отношению к исходному токену. Токены, созданные из них, также будут дочерними и так далее.

При отзыве родительского токена отзываются и все его дочерние токены, а также связанные с ними аренды. Это предотвращает обход механизма отзыва через бесконечную генерацию дочерних токенов.

Иногда такое поведение нежелательно. В этом случае можно использовать orphan-токены. У orphan-токенов нет родительского токена.

Их можно создать несколькими способами:

- через доступ уровня `write` к endpoint'у `auth/token/create-orphan`;
- через доступ уровня `sudo` или root к `auth/token/create` с параметром `no_parent=true`;
- через роли хранилища токенов;
- после аутентификации с помощью любого метода, кроме `Token`.

Также существуют операции, которые позволяют отзывать токен, не отзывая всех его потомков, а превращая непосредственные дочерние токены в orphan-токены. Использовать такие сценарии следует с осторожностью.

## Accessors токенов

При создании токена Stronghold также создаёт accessor токена.

Accessor — это значение, которое работает как ссылка на токен и может быть использовано для выполнения ограниченного набора действий:

- поиск свойств токена без раскрытия самого токена;
- поиск возможностей токена по заданному пути;
- продление токена;
- отзыв токена.

Accessors полезны для аудита и быстрого отзыва токенов в случае инцидента.

> Примечание  
> Возможность видеть accessors в аудит-логах упрощает оперативное реагирование, но требует аккуратного обращения с журналами, так как эти значения также чувствительны с точки зрения безопасности.

## TTL токенов

У каждого не-root-токена есть определённый срок действия (`TTL`), который определяет, как долго токен остаётся действительным.

После истечения TTL токен перестаёт работать, а связанные с ним аренды отзываются.

Если токен допускает продление, можно запросить продление через:

```bash
d8 stronghold token renew <token>
```

### Общий случай

Если токен не является периодическим и не имеет явно заданного максимального TTL, его срок жизни определяется комбинацией нескольких факторов:

1. системного максимального TTL;
2. максимального TTL точки монтирования;
3. ограничений, наложенных методом аутентификации, который выдал токен.

Важно понимать, что итоговый допустимый TTL может меняться между продлениями, поэтому после продления следует проверять возвращаемое значение TTL.

### Явный максимальный TTL

Для токенов можно явно задать максимальный TTL. Это жёсткое ограничение времени жизни токена.

Даже если остальные механизмы допускают более долгую жизнь токена, он не сможет существовать дольше явно заданного предела.

### Периодические токены

В некоторых сценариях отзыв токена не должен происходить по обычной схеме. Например, это может быть длительно работающий сервис. В таком случае применяются периодические токены.

Их можно создать:

- с использованием `sudo` или root-доступа через `auth/token/create`;
- через роли хранилища токенов;
- через методы аутентификации, поддерживающие выпуск периодических токенов, например `AppRole`.

Для периодического токена TTL при каждом успешном продлении сбрасывается до настроенного периода. Пока токен продлевается, он может использоваться неограниченно долго.

> Примечание  
> Периодические токены удобны для сервисов и автоматизации, но обычно не рекомендуются для обычных пользовательских сценариев.

## Токены с привязкой к CIDR

Некоторые токены могут быть привязаны к CIDR-диапазонам. Это ограничивает список клиентских IP-адресов, с которых разрешено использовать токен.

Такое ограничение применяется ко всем токенам, кроме бессрочных root-токенов. Если у root-токена есть срок действия, привязка к CIDR также может использоваться.

## Аренды служебных и пакетных токенов

### Служебные токены

Аренды, создаваемые служебными токенами, отслеживаются вместе с токенами и отзываются по истечении срока действия токена.

### Пакетные токены

Аренды, создаваемые пакетными токенами, ограничены оставшимся TTL пакетного токена и, если токен не orphan, отслеживаются его родителем.

Аренды аннулируются:

- когда истекает TTL пакетного токена;
- когда отзывается родитель пакетного токена.

## Что важно запомнить

- Stronghold использует токены как основной механизм доступа после аутентификации.
- Токен определяет, какие политики и возможности доступны клиенту.
- Root-токены имеют полный доступ и требуют особенно осторожного обращения.
- Токены могут иметь TTL, продление, иерархию и ограничения.
- Служебные и пакетные токены предназначены для разных эксплуатационных сценариев.
- В автоматизации нельзя опираться на внутренний формат токена.
