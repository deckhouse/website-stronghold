Для каждого динамического секрета и сервиса аутентификации Stronghold создаёт _аренду_ (`lease`), в которой содержатся метаданные, такие как срок действия, возможность продления и другие параметры.

Stronghold гарантирует, что данные будут оставаться актуальными в течение установленного периода, известного как время жизни (`TTL`). После окончания срока аренды Stronghold может автоматически аннулировать данные, после чего потребитель секрета не может быть уверен в их актуальности.

Потребители секретов должны регулярно обращаться к Stronghold, чтобы:

- продлить аренду, если это разрешено;
- либо запросить новый секрет.

Это повышает ценность журналов аудита Stronghold и значительно упрощает процесс ротации ключей и учётных данных.

## Для чего нужна аренда

Аренда — один из базовых механизмов управления жизненным циклом чувствительных данных.

Она позволяет:

- ограничивать срок действия динамических секретов;
- принуждать потребителей периодически перепроверять актуальность данных;
- автоматически отзывать устаревшие секреты;
- сокращать окно компрометации в случае утечки;
- упростить отзыв секретов и токенов.

Аренда должна быть у всех динамических секретов в Stronghold. Даже если предполагается, что данные будут действовать «вечно», аренда необходима, чтобы заставить потребителя регулярно подтверждать необходимость их использования.

## Что происходит после истечения аренды

Когда срок аренды истекает, Stronghold автоматически отзывает её. После этого секрет или выданные данные считаются недействительными.

Это означает, что:

- потребитель не должен продолжать использовать такие данные без повторной проверки;
- для дальнейшей работы может потребоваться продление аренды или получение нового секрета;
- поведение после истечения срока зависит от конкретного механизма секретов и сценария использования.

## Продление аренды

Если аренда допускает продление, потребитель может запросить продление срока действия.

При продлении аренды пользователь может указать параметр `increment`, чтобы запросить изменение срока её действия.

Например:

```bash
d8 stronghold lease renew -increment=3600 my-lease-id
```

Эта команда запрашивает увеличение TTL аренды на 1 час (`3600` секунд).

Важно учитывать:

- приращение срока действия происходит не по завершении текущего TTL, а в момент запроса;
- пользователь может не только продлевать, но и сокращать срок аренды, если секрет больше не нужен на весь возможный период;
- запрашиваемое значение `increment` носит рекомендательный характер;
- backend, управляющий секретом, может ограничить или проигнорировать это значение.

После продления аренды рекомендуется проверять возвращаемые данные, чтобы уточнить фактическое новое значение TTL.

## Отзыв аренды

Помимо продления, аренда может быть _отозвана_.

Когда аренда отзывается, секрет сразу становится недействительным и не подлежит дальнейшему продлению.

Отзыв может происходить:

- вручную через API;
- с помощью CLI-команды;
- через веб-интерфейс;
- автоматически, при истечении срока действия;
- автоматически, при отзыве токена, с помощью которого была создана аренда.

Пример отзыва аренды через CLI:

```bash
d8 stronghold lease revoke <lease_id>
```

Если отзывается токен, Stronghold также отзывает все аренды, которые были созданы с его помощью.

## Идентификаторы аренды

При чтении динамического секрета, например с помощью команды `d8 stronghold read`, Stronghold возвращает `lease_id`.

`Lease ID` — это идентификатор, который используется для управления арендой секрета.

С его помощью можно:

- продлить аренду;
- отозвать аренду;
- работать с группами секретов по префиксу идентификатора аренды.

## TTL аренды

Вместе с идентификатором аренды Stronghold возвращает срок действия аренды (`TTL`) — значение в секундах, определяющее период действия секрета или результата аутентификации.

Потребитель секрета должен учитывать это значение и заранее решать, нужно ли:

- продлить аренду;
- получить новый секрет;
- прекратить использование секрета.

## Отзыв на основе префикса

Помимо отзыва единичного секрета, пользователи с соответствующими правами могут отзывать сразу несколько секретов на основе префикса их идентификатора аренды.

В префиксе `lease ID` всегда содержится путь, указывающий на место, откуда был запрошен секрет. Это позволяет отзывать целые группы секретов.

Например, чтобы отозвать все логины `Userpass`, можно выполнить:

```bash
d8 stronghold lease revoke -prefix auth/userpass/
```

Такой подход полезен, если необходимо быстро отозвать все секреты, связанные:

- с конкретным backend'ом;
- с определённой конфигурацией backend'а;
- с потенциально скомпрометированным сценарием доступа.

## Важное исключение: KV

Механизм секретов `KV`, который хранит произвольные секреты, не выдаёт аренды, хотя иногда может возвращать продолжительность аренды.

> Примечание  
> Backend `KV` не использует аренду в том же смысле, что и динамические секреты. За дополнительной информацией обратитесь к разделу [Механизм секретов KV](/products/stronghold/documentation/Руководство пользователя/Механизмы секретов/KV/Обзор/).

## Что важно запомнить

- Аренда применяется прежде всего к динамическим секретам и результатам аутентификации.
- TTL определяет, как долго Stronghold считает данные действительными.
- Аренду можно продлить, если это разрешено.
- Аренду можно отозвать вручную или автоматически.
- При отзыве токена Stronghold отзывает и связанные с ним аренды.
- Не все механизмы секретов работают с арендой одинаково.
