Политики предоставляют декларативный способ разрешать или запрещать доступ к определённым путям и операциям в **Deckhouse Stronghold**. Этот раздел описывает, как политики работают, как устроен их синтаксис и как они участвуют в процессе авторизации.

По умолчанию политики работают по принципу запрета, поэтому пустая политика не предоставляет никаких прав в системе.

## Как политики участвуют в аутентификации

Прежде чем человек или машина смогут получить доступ к Stronghold, администратор должен настроить Stronghold с помощью одного из методов аутентификации. Аутентификация — это процесс, при котором предоставленная человеком или машиной информация проверяется внутренней или внешней системой.

Политики подключаются к этому процессу после успешной проверки подлинности.

### Типовой сценарий

Рассмотрим типовой процесс на примере корпоративной службы `LDAP` или `Active Directory`. Несмотря на то, что в примере используется `LDAP`, общая схема применима ко всем методам аутентификации.

1. Команда безопасности настраивает Stronghold для подключения к выбранному методу аутентификации.
2. Создаётся политика или используется уже существующая, которая предоставляет доступ к путям в Stronghold.
3. Политика загружается в Stronghold и закрепляется за именем.
4. Настраивается сопоставление данных метода аутентификации с политикой. Например:
   - группа `dev` получает политику `readonly-dev`;
   - группа `ops` получает политики `admin` и `auditor`.

После этого при успешной аутентификации Stronghold:

- сопоставляет результат аутентификации с внутренними политиками;
- создаёт токен;
- прикрепляет к нему соответствующие политики.

Для пользователя это выглядит так:

1. пользователь проходит аутентификацию;
2. Stronghold проверяет его через внешний backend;
3. Stronghold сопоставляет результат с политиками;
4. пользователь получает токен с прикреплёнными политиками;
5. дальнейшие действия разрешаются или запрещаются на основе этих политик.

## Синтаксис политик

Политики пишутся на языке **HCL** или **JSON** и описывают, к каким путям в Stronghold пользователю или машине разрешён доступ.

Пример простой политики, предоставляющей доступ для чтения данных по пути `secret/foo`:

```hcl
path "secret/foo" {
  capabilities = ["read"]
}
```

При назначении этой политики токен сможет читать данные из `secret/foo`, но не сможет их обновлять или удалять, так как соответствующие возможности не предоставлены.

### Более подробный пример

```hcl
path "secret/*" {
  capabilities = ["create", "read", "update", "patch", "delete", "list"]
}

path "secret/super-secret" {
  capabilities = ["deny"]
}

path "secret/restricted" {
  capabilities = ["create"]
  allowed_parameters = {
    "foo" = []
    "bar" = ["zip", "zap"]
  }
}
```

Этот пример показывает:

- широкое разрешение на путь `secret/*`;
- явный запрет для `secret/super-secret`;
- ограничение допустимых параметров для `secret/restricted`.

## Сопоставление путей

При обработке запроса Stronghold использует сопоставление на основе путей для определения набора возможностей.

В политике можно использовать:

- точное совпадение;
- префиксную маску `*`;
- сегментный шаблон `+`.

### Примеры

```hcl
path "secret/foo" {
  capabilities = ["read"]
}

path "secret/bar/*" {
  capabilities = ["read"]
}

path "secret/zip-*" {
  capabilities = ["read"]
}
```

```hcl
path "secret/+/teamb" {
  capabilities = ["read"]
}

path "secret/+/+/teamb" {
  capabilities = ["read"]
}
```

> Примечание  
> Символ `*` не является регулярным выражением. Он используется как маска и поддерживается только в конце пути.

## Возможности

Каждый путь в политике должен определять одну или несколько возможностей (`capabilities`), которые задают разрешённые действия.

### Основные возможности

- `create` — создание данных по пути;
- `read` — чтение данных;
- `update` — изменение данных;
- `patch` — частичное обновление;
- `delete` — удаление;
- `list` — перечисление значений по пути.

### Дополнительные возможности

- `sudo` — разрешает доступ к путям, защищённым правами root;
- `deny` — полностью запрещает доступ и всегда имеет наивысший приоритет.

> Примечание  
> Возможности обычно сопоставляются с HTTP-методом, а не с логическим смыслом действия. Например, генерация учётных данных базы данных может требовать `read`, если underlying API использует `GET`.

## Шаблонные политики

Stronghold позволяет подставлять переменные в политиках с использованием identity-данных токена.

Это полезно, если нужно создавать динамические правила доступа, зависящие от сущности, группы или псевдонима пользователя.

### Примеры параметров

Поддерживаются, в частности, следующие значения:

- `identity.entity.id`
- `identity.entity.name`
- `identity.entity.metadata.<metadata key>`
- `identity.entity.aliases.<mount accessor>.id`
- `identity.entity.aliases.<mount accessor>.name`
- `identity.groups.ids.<group id>.name`
- `identity.groups.names.<group name>.id`

### Пример политики для персонального KV-пространства

```hcl
path "secret/data/{{identity.entity.id}}/*" {
  capabilities = ["create", "update", "patch", "read", "delete"]
}

path "secret/metadata/{{identity.entity.id}}/*" {
  capabilities = ["list"]
}
```

### Пример политики для группы

```hcl
path "secret/data/groups/{{identity.groups.ids.fb036ebc-2f62-4124-9503-42aa7A869741.name}}/*" {
  capabilities = ["create", "update", "patch", "read", "delete"]
}

path "secret/metadata/groups/{{identity.groups.ids.fb036ebc-2f62-4124-9503-42aa7A869741.name}}/*" {
  capabilities = ["list"]
}
```

> Предупреждение  
> При разработке шаблонных политик по возможности используйте идентификаторы, а не имена. Идентификаторы стабильны, а имена могут изменяться или использоваться повторно.

## Тонкая настройка управления доступом

В дополнение к обычным возможностям Stronghold позволяет ограничивать параметры запросов и требовать определённые условия для работы с путём.

### Ограничения параметров

Поддерживаются следующие механизмы:

- `required_parameters`
- `allowed_parameters`
- `denied_parameters`

#### required_parameters

Определяет обязательные параметры:

```hcl
path "secret/profile" {
  capabilities = ["create"]
  required_parameters = ["name", "id"]
}
```

#### allowed_parameters

Определяет допустимые параметры и допустимые значения:

```hcl
path "auth/userpass/users/*" {
  capabilities = ["update"]
  allowed_parameters = {
    "password" = []
  }
}
```

```hcl
path "transit/keys/*" {
  capabilities = ["create", "update"]
  allowed_parameters = {
    "auto_rotate_period" = ["8h", "24h", "5d"]
  }
}
```

#### denied_parameters

Определяет запрещённые параметры или значения:

```hcl
path "auth/userpass/users/*" {
  capabilities = ["update"]
  denied_parameters = {
    "token_policies" = []
    "policies" = []
  }
}
```

```hcl
path "auth/token/roles/*" {
  capabilities = ["create", "update"]
  denied_parameters = {
    "allowed_policies" = ["admin"]
  }
}
```

> Примечание  
> Поля `allowed_parameters`, `denied_parameters` и `required_parameters` не поддерживаются для политик, используемых с механизмом секретов `KV v2`.

### TTL для обёртывания ответа

Для некоторых путей можно задавать ограничения на `response wrapping`:

- `min_wrapping_ttl`
- `max_wrapping_ttl`

Пример:

```hcl
path "auth/approle/role/my-role/secret-id" {
  capabilities = ["create", "update"]
  min_wrapping_ttl = "1s"
  max_wrapping_ttl = "90s"
}
```

Это позволяет требовать обязательное обёртывание ответа или ограничивать допустимый TTL wrapping token.

## Встроенные политики

В Stronghold есть две встроенные политики:

- `default`
- `root`

### Политика default

Политика `default` существует всегда и не может быть удалена. По умолчанию она прикрепляется ко всем токенам, если это не отключено явно.

Политика `default` содержит базовую функциональность, например:

- возможность токена узнать информацию о себе;
- работу с `cubbyhole`;
- другие минимально необходимые действия.

Чтобы посмотреть содержимое политики:

```bash
d8 stronghold read sys/policy/default
```

Чтобы создать токен без политики `default`:

```bash
d8 stronghold token create -no-default-policy
```

### Root-политика

Политика `root` не может быть изменена или удалена. Любой токен с этой политикой получает полный доступ к Stronghold.

Это самый привилегированный режим доступа, и его следует использовать только для ограниченного круга административных задач.

Чтобы отозвать root-токен:

```bash
d8 stronghold token revoke "<token>"
```

## Управление политиками

Политики можно создавать, обновлять, читать и удалять через CLI или API.

### Список политик

```bash
d8 stronghold read sys/policy
```

### Создание политики

```bash
d8 stronghold policy write policy-name policy-file.hcl
```

### Обновление политики

```bash
d8 stronghold write sys/policy/my-existing-policy policy=@updated-policy.json
```

### Удаление политики

```bash
d8 stronghold delete sys/policy/policy-name
```

Удаление является идемпотентной операцией: Stronghold не вернёт ошибку, если политика уже отсутствует.

## Прикрепление политик

Stronghold может автоматически прикреплять политики к токену на основе результата аутентификации.

Пример для встроенного метода `Userpass`:

```bash
d8 stronghold write auth/userpass/users/sethvargo \
    password="s3cr3t!" \
    policies="dev-readonly,logs"
```

После этого при успешной аутентификации пользователь получит токен с указанными политиками.

## Root-защищённые endpoint'ы

Некоторые пути в API Stronghold требуют root-политики или возможности `sudo`.

Типовые примеры:

- `sys/audit`
- `sys/auth/:path`
- `sys/config/cors`
- `sys/internal/inspect/router/:tag`
- `sys/leases/revoke-prefix/:prefix`
- `sys/plugins/catalog/:type/:name`
- `sys/raw`
- `sys/remount`
- `sys/rotate`
- `sys/seal`
- `sys/step-down`
- `sys/storage/raft/snapshot-auto/config/:name`

Это означает, что даже административный пользователь не сможет выполнять часть операций без соответствующего уровня доступа.

## Политики токена и политики идентичности

У токена есть два набора политик:

- **политики токена** — назначаются при создании токена или через метод аутентификации;
- **политики идентичности** — вычисляются на основе сущности, групп и identity-модели.

Политиками идентичности нельзя управлять напрямую так же, как политиками токена. Они зависят от identity-сущностей, групп и членства в них.

Именно совокупность этих политик определяет итоговый набор разрешений токена.

## Что важно запомнить

- Политики — это основной механизм авторизации в Stronghold.
- Аутентификация определяет, кто вы, а политика — что вам разрешено.
- По умолчанию доступ запрещён, пока политика явно не разрешит его.
- Политики можно писать на HCL или JSON.
- Для некоторых путей требуются `sudo` или root-права.
- Identity-модель может дополнять права токена через сущности и группы.
