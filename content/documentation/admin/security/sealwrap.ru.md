`Seal wrap` — это механизм дополнительного шифрования чувствительных данных поверх обычного криптографического барьера хранилища. В документации Stronghold его можно рассматривать как **двойное шифрование**: данные сначала защищаются встроенным барьером Stronghold, а затем дополнительно оборачиваются через используемый `seal`-механизм.

Такой подход особенно полезен в сценариях с повышенными требованиями к защите ключевого материала, а также в инфраструктурах, где используются внешние KMS- или HSM-устройства.

## Что даёт seal wrap

Дополнительный слой шифрования может быть полезен для:

- повышения защищённости критически важных параметров;
- соответствия внутренним требованиям безопасности и регуляторным политикам;
- сценариев, где корневой материал и наиболее чувствительные значения должны дополнительно защищаться внешним `seal`-механизмом;
- интеграции Stronghold в инфраструктуры с KMS или HSM.

## Как это работает

При использовании `seal wrap` Stronghold сохраняет часть данных не только в зашифрованном виде внутри хранилища, но и с дополнительной обёрткой через поддерживаемый `seal`.

Это означает:

- root key и другие критические параметры получают дополнительный уровень защиты;
- часть значений в механизмах аутентификации и механизмах секретов может храниться в `seal wrap`-виде;
- сама логика репликации и хранения для клиента остаётся прозрачной, но операции чтения и записи таких значений могут становиться дороже по времени.

## Какие данные обычно seal-wrapped

Двойное шифрование применяется прежде всего к критичным security-параметрам внутри core Stronghold, включая:

- root key;
- keyring;
- recovery key и связанные с распечатыванием данные;
- другие внутренние критичные параметры Stronghold.

Кроме того, поддерживающие backend'ы и плагины могут использовать `seal wrap` для хранения отдельных чувствительных значений.

Типовые примеры:

- `LDAP`;
- `PKI` для ключей и issuer-данных;
- `SSH` для ключей CA;
- `Transit` для ключей и их политик.

## Связь с KMS и HSM

`Seal wrap` тесно связан с используемым `seal`-механизмом.

Если Stronghold использует:

- HSM через `seal "pkcs11"`;
- внешний KMS, например `seal "yandexcloudkms"`,

то дополнительная защита критичных данных реализуется поверх встроенного криптографического барьера Stronghold и через тот же внешний `seal`-контур.

> Примечание  
> Встроенное шифрование хранилища Stronghold и `seal wrap` решают разные задачи. Встроенный барьер защищает все данные в `storage` backend'е, а `seal wrap` добавляет дополнительную защиту только для наиболее чувствительных внутренних значений.

## Включение и отключение

Для поддерживаемых `seal`-механизмов `seal wrap` в Vault включён по умолчанию. В Stronghold отключить дополнительное шифрование для данных, кроме root key, можно параметром:

```hcl
disable_sealwrap = true
```

Важно учитывать:

- отключение `seal wrap` не отключает защиту root key через `seal`;
- изменение `disable_sealwrap` действует постепенно, по мере чтения и перезаписи данных;
- повторное включение также происходит постепенно, а не мгновенно для всего хранилища.

## Что происходит при изменении disable_sealwrap

При изменении параметра `disable_sealwrap` Stronghold не выполняет одномоментную перешифровку всего хранилища. Вместо этого переход происходит постепенно:

- при отключении новые или перезаписанные значения перестают оборачиваться дополнительным `seal`;
- при повторном включении новые или перезаписанные значения снова начинают получать `seal wrap`;
- ранее сохранённые значения будут приводиться к новому состоянию по мере операций чтения и записи.

Это важно учитывать при эксплуатации, потому что после изменения параметра состояние хранилища в течение некоторого времени может быть смешанным.

## Sealwrap rewrap

При изменении конфигурации `seal` Stronghold должен заново обернуть все `seal wrap`-значения, чтобы они соответствовали актуальной конфигурации шифрования. Этот процесс называется **seal rewrap**.

`Rewrap` особенно полезен:

- после миграции на новый `seal`;
- после изменения параметров `seal`-конфигурации;
- после ротации ключевого материала во внешнем KMS или HSM.

Для работы с процессом `rewrap` в API Stronghold доступны endpoint'ы:

| Метод | Путь |
| --- | --- |
| `GET` | `/sys/sealwrap/rewrap` |
| `POST` | `/sys/sealwrap/rewrap` |

- `GET /sys/sealwrap/rewrap` возвращает статус процесса и счётчики обработанных записей;
- `POST /sys/sealwrap/rewrap` запускает процесс `rewrap`, если он ещё не выполняется.

### Пример запуска rewrap

```bash
curl \
  --header "X-Vault-Token: ${VAULT_TOKEN}" \
  --request POST \
  "${VAULT_ADDR}/v1/sys/sealwrap/rewrap"
```

## Что учитывать во время rewrap

Во время выполнения `rewrap`:

- процесс выполняется в фоне;
- прогресс удобно отслеживать через `GET /sys/sealwrap/rewrap` и журналы сервера;
- в период выполнения не следует вносить новые изменения в `seal`-конфигурацию.

Длительность `rewrap` зависит от:

- количества `seal wrap`-записей;
- характеристик внешнего `seal` backend'а;
- производительности хранилища;
- сетевой связности с внешним KMS или HSM, если они используются.

## Ограничения и особенности эксплуатации

При использовании `seal wrap` учитывайте следующие особенности:

- `seal wrap` работает только при использовании `seal`, который поддерживает эту возможность;
- если используется внешний HSM или удалённый KMS, операции с `seal wrap`-значениями могут быть заметно медленнее обычных;
- это особенно важно для write-heavy-сценариев и для инфраструктур с нестабильной связностью до KMS или HSM;
- если внешний `seal` недоступен во время работы, это может повлиять на часть операций Stronghold;
- изменение `seal`-конфигурации требует осторожности и предварительной проверки сценария `rewrap`.

## Влияние на производительность

Дополнительный слой шифрования увеличивает стоимость некоторых операций.

На практике это может выражаться в следующем:

- операции записи и изменения чувствительных значений становятся медленнее;
- часть внутренних операций требует обращения к внешнему `seal`-механизму;
- рост задержек особенно заметен при удалённом KMS или перегруженном HSM;
- при большом количестве `seal wrap`-значений время `rewrap` может быть существенным.

> Примечание  
> В большинстве сценариев это оправданная плата за дополнительный уровень защиты. Однако перед включением `seal wrap` в production-окружении стоит оценить влияние на производительность в своей инфраструктуре.

## Seal wrap и репликация

Механизм `seal wrap` находится ниже уровня логики репликации. Это означает, что:

- репликация передаёт данные и признак необходимости `seal wrap`;
- конкретная реализация дополнительного шифрования выполняется локальным `seal`-механизмом на каждой стороне;
- разные кластеры могут использовать разные локальные ключи KMS или HSM, сохраняя общую модель защиты.

Это позволяет использовать единый подход к защите чувствительных значений даже в распределённых сценариях.

## Практические рекомендации

- Используйте `seal wrap`, если у вас есть требования к дополнительной защите ключевого материала.
- Перед включением в production-окружении учитывайте влияние на производительность.
- Для HSM-интеграций заранее проверяйте стабильность доступа к устройству.
- Для внешних KMS заранее проверяйте сетевую доступность, права и поведение при отказах.
- Изменение `disable_sealwrap` лучше выполнять контролируемо и с пониманием, что перешифровка происходит постепенно.
- После смены `seal`-механизма или ключевого материала контролируйте выполнение `rewrap`.
- Не вносите параллельные изменения в `seal`-конфигурацию, пока `rewrap` не завершён.
