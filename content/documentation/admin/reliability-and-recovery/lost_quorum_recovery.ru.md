## Восстановление после потери кворума

Кворум — минимальное количество узлов в кластере, необходимое для голосования и избрания лидера.

Лидер (`Raft leader`) — активный узел кластера, принимающий операции чтения и записи и координирующий работу остальных узлов.

При использовании интегрированного backend-хранилища поддержание кворума `Raft` является критически важным условием для работы Stronghold в режиме высокой доступности. Кластер Stronghold окончательно теряет кворум, когда нет возможности восстановить достаточное количество серверов Stronghold для достижения консенсуса и избрания лидера.

Без кворума серверов кластер Stronghold больше не может выполнять операции чтения и записи.

> Важно  
> Этот раздел относится к кластерам Stronghold, которые используют интегрированное backend-хранилище `Raft`. Если Stronghold использует внешний backend-хранилище, например `etcd` или `postgresql`, необходимо использовать процедуры восстановления соответствующего backend-хранилища.

## Как рассчитывается кворум

Кворум кластера динамически обновляется при подключении к нему новых узлов. Stronghold рассчитывает кворум по формуле `(n+1)/2`, где `n` — количество серверов в кластере.

Примеры:

- для кластера из 3 серверов требуется как минимум 2 рабочих сервера;
- для кластера из 5 серверов требуется как минимум 3 рабочих сервера.

В частности, для выполнения операций чтения и записи должно оставаться достаточное число постоянно активных серверов.

> Примечание  
> Существует исключение из этого правила, если при присоединении к кластеру используется опция `-non-voter`. Эта функция доступна только в версии Stronghold формата отдельной инсталляции.

## Как понять, что кластер потерял кворум

Потеря кворума обычно проявляется так:

- кластер не может выбрать лидера;
- операции чтения и записи перестают выполняться;
- команда `raft list-peers` не может получить сведения об active-узле;
- в журналах появляются ошибки присоединения к лидеру или ошибки bootstrap/challenge.

### Пример симптомов в CLI

```text
$ stronghold operator raft list-peers
* local node not active but active cluster node not found

$ stronghold kv get kv/apikey
* local node not active but active cluster node not found
```

### Пример симптомов в журналах

```text
окт 20 10:54:32 standalone-astra stronghold[647]: {"@level":"info","@message":"attempting to join possible raft leader node","@module":"core","@timestamp":"2025-10-20T10:54:02.578963Z","leader_addr":"https://stronghold-0.stronghold.tld:8201"}
окт 20 10:54:32 standalone-astra stronghold[647]: {"@level":"error","@message":"failed to get raft challenge","@module":"core","@timestamp":"2025-10-20T10:54:32.597558Z","error":"error during raft bootstrap init call: Put \"https://10.0.101.22:8201/v1/sys/storage/raft/bootstrap/challenge\": dial tcp 10.0.101.22:8201: i/o timeout","leader_addr":"https://stronghold-0.stronghold.tld:8201"}
```

## Когда восстанавливать кворум, а когда — восстанавливаться из снимка

После потери кворума возможны два базовых сценария.

### Сценарий 1. Восстановление кворума

Этот сценарий подходит, если:

- у вас остался хотя бы один полностью работоспособный узел;
- данные на surviving-узле считаются актуальными;
- нужно как можно быстрее вернуть кластер в рабочее состояние;
- вы готовы временно преобразовать кластер в уменьшенную конфигурацию, например из 3 узлов в 1 узел.

В этом случае используется процедура восстановления через `peers.json`.

### Сценарий 2. Восстановление из снимка

Этот сценарий подходит, если:

- surviving-узел отсутствует или его состояние нельзя считать надёжным;
- повреждены данные backend-хранилища;
- нужно восстановить кластер в заранее известное сохранённое состояние;
- проще или безопаснее развернуть новый кластер и восстановить его из резервной копии.

В этом случае используйте раздел [Восстановление из резервной копии](../Восстановление из резервной копии/).

> Важно  
> Если у вас есть рабочий surviving-узел и данные на нём считаются консистентными, восстановление кворума через `peers.json` обычно быстрее. Если же есть сомнения в целостности данных, предпочтительнее использовать восстановление из заранее проверенного снимка.

## Типовой сценарий потери кворума

Когда два сервера из трёх выходят из строя, кластер теряет кворум и перестаёт функционировать. Несмотря на наличие одного полностью работоспособного сервера, кластер не может обрабатывать запросы на чтение или запись.

Процесс восстановления работы Stronghold при потере 2 из 3 серверов обычно выполняется путём преобразования кластера в вариант из одного узла.

Для выполнения этой процедуры один сервер должен быть полностью работоспособным.

В кластере из 5 серверов или при наличии неголосующих узлов необходимо остановить остальные исправные серверы перед выполнением восстановления через `peers.json`.

## Особенности восстановления при работе Autopilot

Автопилот (`Autopilot`) — это механизм Stronghold, который автоматически отслеживает состояние узлов `Raft`-кластера и управляет их участием в кворуме.

В некоторых случаях Stronghold теряет кворум из-за Autopilot и серверов, помеченных как неработоспособные, хотя Stronghold формально продолжает работать.

В таких ситуациях перед выполнением процедуры восстановления с использованием `peers.json` необходимо остановить службы Stronghold на неработоспособных серверах.

## Восстановление кворума через peers.json

### Перед началом

Перед запуском процедуры убедитесь, что:

- у вас есть один исправный узел Stronghold;
- остальные узлы, которые могут помешать восстановлению, остановлены;
- вы знаете путь к каталогу данных `Raft`;
- у вас есть доступ к файловой системе узла;
- у вас есть ключи для последующей распечатки, если auto-unseal не используется.

> Предупреждение  
> Процедура с `peers.json` изменяет конфигурацию кластера `Raft`. Применяйте её только в аварийном сценарии, когда стандартным способом восстановить кворум уже невозможно.

### Шаг 1. Найдите каталог хранилища

На сервере с исправным узлом Stronghold найдите каталог хранилища `Raft`. Для этого просмотрите файл конфигурации Stronghold. Строка `storage` будет содержать `path` к каталогу.

Пример конфигурации:

```hcl
storage "raft" {
  path    = "/opt/stronghold/data"
  server_id = "stronghold_0"
}
listener "tcp" {
  address     = "0.0.0.0:8200"
  cluster_address     = "0.0.0.0:8201"
  tls_disable = true
}
api_addr = "http://stronghold-0.stronghold.tld:8200"
cluster_addr = "http://stronghold-0.stronghold.tld:8201"
disable_mlock = true
ui=true
```

В этом примере:

- `path` — путь к файловой системе, где Stronghold хранит данные;
- `server_id` — идентификатор сервера в кластере `Raft`.

### Шаг 2. Найдите подкаталог raft

Внутри каталога хранилища находится директория `raft`.

Пример структуры:

```text
/opt
└ stronghold
  └── data
      ├── raft
      │   ├── raft.db
      │   └── snapshots
      └── vault.db
```

### Шаг 3. Создайте файл peers.json

Чтобы единственный оставшийся сервер Stronghold мог достичь кворума и избрать себя лидером, создайте файл `raft/peers.json`, содержащий информацию о работоспособном сервере.

Формат файла — массив JSON, содержащий ID работоспособного сервера Stronghold, его `адрес:порт` и информацию о возможности голосовать.

Пример:

```bash
cat > /opt/stronghold/data/raft/peers.json << EOF
[
  {
    "id": "stronghold_0",
    "address": "stronghold-0.stronghold.tld:8201",
    "non_voter": false
  }
]
EOF
```

Параметры:

- `id` — идентификатор сервера;
- `address` — хост и порт сервера, используемые для кластерного взаимодействия;
- `non_voter` — участвует ли сервер в голосовании.

### Шаг 4. Проверьте права доступа к peers.json

Убедитесь, что пользователь `stronghold` имеет право на чтение и изменение файла `peers.json`:

```bash
chown stronghold:stronghold /opt/stronghold/data/raft/peers.json
chmod 600 /opt/stronghold/data/raft/peers.json
```

### Шаг 5. Перезапустите Stronghold

Перезапустите процесс Stronghold, чтобы он загрузил новый файл `peers.json`:

```bash
sudo systemctl restart stronghold
```

> Примечание  
> Если вы используете `systemd`, сигнал `SIGHUP` не подойдёт для этой процедуры.

### Шаг 6. Выполните распечатывание Stronghold

Если не настроено автоматическое распечатывание (`auto-unseal`), выполните его вручную, а затем проверьте статус.

Пример:

```bash
stronghold operator unseal
```

Проверьте состояние:

```bash
stronghold status
```

Пример вывода:


## Проверка успешности восстановления

Процедура восстановления прошла успешно, если Stronghold запустился и в системных журналах появились сообщения вида:

```text
[INFO]  core.cluster-listener: serving cluster requests: cluster_listen_address=[::]:8201
[INFO]  storage.raft: raft recovery initiated: recovery_file=peers.json
[INFO]  storage.raft: raft recovery found new config: config="{[{Voter stronghold_0 https://stronghold-0.stronghold.tld:8201}]}"
[INFO]  storage.raft: raft recovery deleted peers.json
```

### Проверка списка узлов

Теперь в кластере должен числиться только один сервер. Это позволит Stronghold достичь кворума и восстановить работоспособность.

Чтобы убедиться в количестве серверов, выполните:

```bash
stronghold operator raft list-peers
```

Пример вывода:

```text
server          Address                                     State     Voter
----            -------                                     -----     -----
stronghold_0    https://stronghold-0.stronghold.tld:8201    leader    true
```

Это означает, что в кластере остался один сервер и он успешно избрал себя лидером.

## Дальнейшие действия после восстановления кворума

После восстановления кворума через `peers.json` у вас есть рабочий, но уменьшенный кластер. Дальнейшие действия зависят от состояния остальных узлов.

### Если вышедшие из строя узлы поддаются восстановлению

Лучший вариант — вернуть их в сеть и подключить к кластеру с использованием тех же адресов хостов. Это позволит вернуть кластер в полностью рабочее состояние.

В этом случае в `raft/peers.json` должны быть указаны данные для каждого сервера, который нужно включить в кластер:

```json
[
  {
    "id": "stronghold_0",
    "address": "stronghold-0.stronghold.tld:8201",
    "non_voter": false
  },
  {
    "id": "stronghold_1",
    "address": "stronghold-1.stronghold.tld:8201",
    "non_voter": false
  },
  {
    "id": "stronghold_2",
    "address": "stronghold-2.stronghold.tld:8201",
    "non_voter": false
  }
]
```

### Если узлы не подлежат восстановлению

Если вернуть остальные узлы невозможно, рассмотрите:

- развёртывание нового кластера;
- восстановление из снимка;
- повторное присоединение новых узлов уже после стабилизации surviving-узла.

## Когда лучше использовать восстановление из снимка

Восстановление из снимка предпочтительнее, если:

- surviving-узел отсутствует;
- данные surviving-узла вызывают сомнения;
- потеряно слишком много узлов и восстановление конфигурации `Raft` небезопасно;
- требуется восстановить кластер в гарантированно известное состояние;
- выполняется миграция или перенос на новый контур.

В этом случае используйте раздел [Восстановление из резервной копии](../Восстановление из резервной копии/).

## Практические рекомендации

- заранее документируйте `server_id`, сетевые адреса и пути к данным;
- регулярно выполняйте резервное копирование и проверку снимков;
- храните резервные копии и ключи отдельно от рабочего кластера;
- периодически отрабатывайте аварийные процедуры на тестовом контуре;
- если используется Autopilot, учитывайте его влияние на членство узлов и кворум;
- не выполняйте восстановление через `peers.json`, если у вас нет уверенности в консистентности surviving-узла.

## Что дальше

После восстановления кворума или данных рекомендуется:

- проверить состояние кластера;
- убедиться, что операции чтения и записи снова работают;
- проверить список узлов и роль лидера;
- при необходимости перейти к разделу [Резервное копирование](../Резервное копирование/) и убедиться, что стратегия аварийного восстановления актуальна.
