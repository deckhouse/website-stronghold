Резервное копирование **Deckhouse Stronghold** строится на снимках встроенного backend-хранилища `Raft`. Снимок позволяет сохранить состояние кластера и в дальнейшем использовать его для восстановления данных после сбоя, аварийного восстановления или выполнения регламентных операций.

Stronghold поддерживает два основных сценария резервного копирования:

- **ручные снимки** — администратор явно сохраняет файл снимка и при необходимости использует его для восстановления;
- **автоматические снимки** — Stronghold по расписанию сохраняет резервные копии на локальный диск или в `S3`-совместимое объектное хранилище.

> Примечание  
> Этот раздел посвящён резервному копированию. Подробное восстановление описано в разделе [Восстановление из резервной копии](../Восстановление из резервной копии/), а настройка расписания и целевых хранилищ — в разделе [Автоматические снимки](../automated-snapshots/).

## Когда использовать снимки

Снимки полезны в следующих ситуациях:

- защита от повреждения данных или ошибочных действий;
- подготовка к обновлениям, переключению редакций и другим рискованным операциям;
- восстановление после аварии;
- вынос резервных копий за пределы кластера или площадки размещения;
- периодическая проверка восстановимости данных.

## Что важно учитывать

При организации резервного копирования Stronghold учитывайте следующие ограничения и особенности:

- снимки относятся к встроенному backend-хранилищу `Raft`;
- снимки работают только в кластерах, которые используют интегрированное `Raft`-хранилище как основной `storage` backend;
- если Stronghold использует `etcd`, `postgresql` или другой внешний backend-хранилище, необходимо использовать штатные механизмы резервного копирования самого backend-хранилища, а не команды `stronghold operator raft snapshot`;
- в резервной копии данные остаются зашифрованными;
- для восстановления доступа к данным после восстановления понадобятся корректные unseal-ключи или recovery-ключи в соответствии с конфигурацией кластера;
- автоматические снимки являются административной функцией и требуют продуманной политики хранения.

> Важно  
> Снимок не заменяет процедуру восстановления. Наличие файла резервной копии само по себе не гарантирует успешное восстановление, если не проверены ключи, доступность файла и корректность процедуры восстановления.

## Standalone и DKP

Подход к резервному копированию Stronghold одинаков на уровне самого продукта, но способ доступа к нему зависит от варианта развёртывания.

### Standalone

В Standalone-сценарии администратор, как правило, выполняет команды `stronghold` напрямую на узле или через настроенный CLI-доступ к инстансу Stronghold.

### DKP

В составе **Deckhouse Kubernetes Platform** резервное копирование Stronghold обычно выполняется через `d8 stronghold`, а доступ к сервису организуется через Ingress и/или OIDC-аутентификацию.

Пример получения адреса Stronghold в DKP:

```bash
export STRONGHOLD_ADDR=https://$(d8 k -n d8-stronghold get ing stronghold -o json | jq -r '.spec.rules[0].host')
```

> Примечание  
> Независимо от варианта развёртывания, если Stronghold использует интегрированное `Raft`-хранилище, логика создания и проверки снимков остаётся одинаковой.

## Перед началом резервного копирования

Перед созданием снимка рекомендуется убедиться, что:

- Stronghold использует встроенное backend-хранилище `Raft`;
- `Raft` используется как основной `storage` backend, а не только как `ha_storage`;
- у вас есть административный доступ к кластеру;
- подготовлено безопасное место для хранения файла снимка вне production-контура;
- вы заранее проверили, что при необходимости сможете получить доступ к unseal- или recovery-ключам.

> Предупреждение  
> Команда `stronghold operator raft snapshot save` применима только к кластерам с интегрированным `Raft`-хранилищем. Если Stronghold использует `etcd`, `postgresql` или другой внешний backend, резервную копию нужно выполнять средствами самого backend-хранилища.

## Ручное создание снимка

Ручной снимок позволяет сохранить текущее состояние встроенного хранилища Stronghold в файл и использовать его как резервную копию.

### Создание снимка через CLI

Для создания снимка используйте команду:

```bash
d8 stronghold operator raft snapshot save backup.snap
```

В результате Stronghold сохранит снимок в локальный файл `backup.snap`.

Если вы работаете не через `d8 stronghold`, а через standalone CLI, используйте эквивалентную команду вашей инсталляции.

### Создание снимка через API

Для создания снимка через API используйте endpoint `GET /sys/storage/raft/snapshot`:

```bash
curl \
  --request GET \
  --header "X-Vault-Token: ${VAULT_TOKEN}" \
  "${VAULT_ADDR}/v1/sys/storage/raft/snapshot" > backup.snap
```

> Примечание  
> Для выполнения операции требуется административный токен с достаточными правами доступа.

## Проверка снимка

После создания резервной копии рекомендуется проверить, что снимок читается и имеет корректную структуру. Для этого используйте команду `stronghold operator raft snapshot inspect`.

В отличие от `snapshot save` и `snapshot restore`, команда `inspect` работает **локально** и не требует подключения к работающему серверу Stronghold. Это делает её удобной для валидации резервных копий, первичной диагностики и анализа содержимого снимка перед восстановлением.

> Предупреждение  
> Команда `inspect` относится к снимкам интегрированного `Raft`-хранилища. Если Stronghold использует `etcd`, `postgresql` или другой внешний backend, проверку резервной копии нужно выполнять средствами соответствующего backend-хранилища.

### Базовое использование

```bash
stronghold operator raft snapshot inspect <snapshot_file>
```

Пример:

```bash
stronghold operator raft snapshot inspect raft.snap
```

Команда выводит сводную информацию о снимке и таблицу с количеством ключей и их суммарным размером по группам.

### Что показывает команда

В стандартном выводе отображаются:

- метаданные снимка, включая `ID`, `Size`, `Index`, `Term` и `Version`;
- сгруппированные ключи хранилища;
- количество ключей в каждой группе;
- суммарный размер данных в каждой группе.

Пример сокращённого вывода:

```text
ID           bolt-snapshot
Size         93290
Index        8957
Term         11
Version      1
Key Name                                          Count      Size
--------                                          -----      ----
wal/logs                                          54         9 KB
index/pages                                       14         26.1 KB
sys/policy                                        3          3.4 KB
core/cluster                                      2          236 B
```

### Основные флаги inspect

| Флаг | Описание |
| --- | --- |
| `-details` | Включает детальный анализ ключей и вывод таблицы групп. По умолчанию включён |
| `-depth` | Управляет глубиной группировки ключей по сегментам пути |
| `-filter` | Показывает только ключи с указанным префиксом |
| `-format` | Формат вывода: `table` или `json` |
| `-validate` | Выполняет дополнительную проверку консистентности снимка |

### Полезные сценарии проверки

#### Быстрая проверка после создания резервной копии

Сразу после создания снимка можно проверить, что файл читается и содержит ожидаемые данные:

```bash
stronghold operator raft snapshot save /backup/raft.snap
stronghold operator raft snapshot inspect /backup/raft.snap
```

#### Проверка консистентности снимка

Для расширенной проверки используйте флаг `-validate`:

```bash
stronghold operator raft snapshot inspect -validate raft.snap
```

Такая проверка помогает убедиться, что:

- снимок не пустой;
- содержимое `state.bin` корректно разбирается;
- в снимке присутствуют критические пути, например `core` и `sys`.

Это полезно для автоматических проверок резервных копий, но не заменяет полноценного тестового восстановления.

#### Анализ отдельных префиксов

Чтобы посмотреть только интересующую часть данных, используйте `-filter` и `-depth`:

```bash
stronghold operator raft snapshot inspect -depth 3 -filter=core raft.snap
```

Это удобно для диагностики роста хранилища или поиска крупных групп ключей.

#### Вывод в формате JSON

Для интеграции со скриптами и monitoring-сценариями можно использовать JSON:

```bash
stronghold operator raft snapshot inspect -format=json raft.snap
```

При использовании `-validate` JSON-вывод удобно обрабатывать автоматически, например через `jq`.

## Что проверка снимка не гарантирует

Команда `inspect` полезна, но имеет ограничения:

- она не восстанавливает снимок и не изменяет состояние кластера;
- проверка контрольных сумм и базовой структуры не гарантирует, что снимок точно подойдёт для восстановления в конкретный кластер;
- для полной уверенности в восстановимости нужно периодически выполнять тестовое восстановление на отдельном контуре.

## Практические рекомендации по резервному копированию

При организации резервного копирования Stronghold рекомендуется:

- сохранять файл снимка в место, доступ к которому ограничен;
- хранить резервные копии вне того же контура, который они защищают;
- после создания снимка выполнять `stronghold operator raft snapshot inspect` как базовую валидацию;
- периодически проверять не только создание снимков, но и возможность тестового восстановления;
- для production-окружений использовать внешнее объектное хранилище вместо локального диска, если это возможно;
- заранее определить, кто отвечает за доступ к резервным копиям и к ключам распечатывания;
- если кластер участвует в схемах репликации, продумать отдельную стратегию резервного копирования для каждого контура.

## Типовой сценарий резервного копирования

Ниже приведён минимальный рекомендуемый сценарий:

1. Проверить, что Stronghold использует интегрированное `Raft`-хранилище.
2. Создать снимок вручную.
3. Проверить снимок командой `inspect`.
4. Переместить снимок в защищённое хранилище.
5. Зафиксировать, где хранится снимок и какие ключи потребуются для восстановления.
6. Периодически выполнять тестовое восстановление в отдельном контуре.
