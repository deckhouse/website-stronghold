## Восстановление из резервной копии

Восстановление из резервной копии используется, когда необходимо вернуть состояние **Deckhouse Stronghold** из ранее сохранённого снимка.

> Важно  
> Этот раздел относится только к кластерам Stronghold, которые используют интегрированное backend-хранилище `Raft` как основной `storage` backend.  
> Если Stronghold работает поверх `etcd`, `postgresql` или другого внешнего backend-хранилища, необходимо использовать штатную процедуру восстановления самого backend-хранилища, а не команды `stronghold operator raft snapshot restore`.

## Когда использовать восстановление

Восстановление из резервной копии обычно требуется в следующих случаях:

- повреждение данных хранилища;
- критическая ошибка после обновления или изменения конфигурации;
- аварийное восстановление после сбоя инфраструктуры;
- перенос кластера на новый контур;
- возврат к сохранённому состоянию после неудачной операции.

## Что важно понимать перед восстановлением

Перед запуском процедуры учитывайте следующее:

- восстановление изменяет состояние хранилища Stronghold;
- снимок должен соответствовать кластеру, который вы восстанавливаете, или сценарию переноса;
- после восстановления понадобятся **исходные** unseal-ключи или recovery-ключи;
- новые ключи, выданные при промежуточной инициализации, не заменяют исходные ключи для доступа к данным из снимка;
- даже если файл снимка успешно читается, это не гарантирует автоматического успешного восстановления без корректной процедуры распечатывания и доступа к ключам.

## Перед началом

Перед восстановлением выполните следующие действия:

- верните кластер в рабочее техническое состояние после инцидента;
- подготовьте новое хранилище или новую инсталляцию, если это требуется по сценарию;
- получите временный root-токен после инициализации, если это часть вашей процедуры восстановления;
- убедитесь, что восстанавливаемый кластер использует интегрированное `Raft`-хранилище как основной `storage` backend;
- скопируйте файл снимка на узел, с которого будет выполняться восстановление;
- убедитесь, что у вас есть исходные unseal-ключи или recovery-ключи;
- при возможности заранее проверьте снимок с помощью команды `stronghold operator raft snapshot inspect`.

> Предупреждение  
> Если у вас нет доступа к исходным ключам распечатывания или recovery-ключам, восстановление данных может оказаться бесполезным даже при наличии корректного снимка.

## Восстановление через CLI

Для восстановления снимка используйте команду с флагом `-force`:

```bash
d8 stronghold operator raft snapshot restore -force /tmp/snapshots/backup.snap
```

Флаг `-force` необходим, потому что текущее состояние кластера и данные снимка относятся к разным состояниям хранилища.

Если вы работаете не через `d8 stronghold`, а через standalone CLI, используйте эквивалентную команду вашей инсталляции Stronghold.

## Восстановление через API

Для восстановления через API используйте endpoint `POST /sys/storage/raft/snapshot-force`:

```bash
curl \
  --request POST \
  --header "X-Vault-Token: ${VAULT_TOKEN}" \
  --data-binary @/tmp/snapshots/backup.snap \
  "${VAULT_ADDR}/v1/sys/storage/raft/snapshot-force"
```

> Примечание  
> Для выполнения восстановления через API требуется токен с достаточными привилегиями.

## Действия после восстановления

После загрузки снимка необходимо выполнить процедуру распечатывания Stronghold с использованием исходных ключей:

```bash
d8 stronghold operator unseal
```

Если используется стандартная схема с несколькими частями ключа, выполните команду несколько раз, вводя необходимое количество частей ключа.

После распечатывания рекомендуется:

- проверить статус Stronghold;
- убедиться, что хранилище инициализировано и не запечатано;
- проверить доступность API;
- выполнить вход с административной учётной записью;
- проверить наличие ожидаемых данных и конфигурации;
- убедиться, что механизмы секретов, методы аутентификации и политики находятся в ожидаемом состоянии.

## Что проверить после восстановления

Минимальный набор проверок после восстановления:

1. Статус Stronghold:

   ```bash
   d8 stronghold status
   ```

2. Базовая доступность API:

   ```bash
   curl -kL ${STRONGHOLD_ADDR}/v1/sys/health
   ```

3. Возможность входа с административным доступом.

4. Наличие ожидаемых путей, политик, методов аутентификации и секретов.

5. При необходимости — работоспособность веб-интерфейса.

> Примечание  
> Если Stronghold используется в составе DKP, после восстановления дополнительно проверьте состояние подов, Ingress и доступность Stronghold через платформенный контур доступа.

## Практические рекомендации

- проверяйте процесс восстановления на тестовом контуре, а не только создание снимка;
- документируйте, где хранятся резервные копии и кто отвечает за доступ к ключам;
- если используется несколько контуров или схем репликации, продумайте порядок восстановления для каждого из них отдельно;
- не ограничивайтесь только проверкой файла снимка — подтверждайте возможность полноценного восстановления;
- выполняйте восстановление по заранее согласованной и протестированной процедуре.

## Типовой сценарий восстановления

Ниже приведён рекомендуемый общий порядок действий:

1. Подготовить целевой кластер или узел для восстановления.
2. Убедиться, что используется интегрированное `Raft`-хранилище.
3. Скопировать файл снимка в доступное место.
4. Выполнить восстановление из снимка.
5. Распечатать Stronghold исходными ключами.
6. Проверить доступность сервиса, API и критичных данных.
7. Зафиксировать результат восстановления и дальнейшие действия.

## Ограничения

При восстановлении из резервной копии учитывайте следующие ограничения:

- процедура применима только к снимкам интегрированного `Raft`-хранилища;
- снимок не заменяет требования к наличию ключей распечатывания;
- успешная загрузка снимка не гарантирует, что восстановлены все внешние зависимости;
- если Stronghold интегрирован с внешними identity-провайдерами, KMS, HSM или другими системами, после восстановления может потребоваться отдельная проверка этих интеграций.
