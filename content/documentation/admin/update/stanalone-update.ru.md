## Обновление Standalone

В Standalone-развёртывании **Deckhouse Stronghold** работает как отдельный сервис в Linux и обычно управляется через `systemd`. Обновление в этом сценарии выполняется путём замены бинарного файла Stronghold и последующего перезапуска сервиса.

> Важно  
> Перед обновлением ознакомьтесь с [Историей версий](/products/stronghold/documentation/О продукте/versions/) и убедитесь, что в новой версии нет изменений, влияющих на обратную совместимость.

## Когда выполнять обновление

Рекомендуется выполнять обновление:

- в окно обслуживания;
- после создания резервной копии;
- после проверки доступности unseal-ключей или recovery-ключей;
- после проверки совместимости конфигурации с целевой версией.

Даже если Stronghold развёрнут в конфигурации с высокой доступностью, обновление лучше выполнять в периоды низкой нагрузки на API.

## Перед началом

Перед обновлением убедитесь, что у вас есть:

- актуальная резервная копия данных;
- доступ к unseal-ключам, если auto-unseal не настроен;
- доступ к recovery-ключам, если они используются в вашей конфигурации;
- новый бинарный файл `stronghold`, полученный из официального источника или внутреннего доверенного зеркала;
- план отката на случай неуспешного обновления.

## Рекомендуемая подготовка

Перед обновлением рекомендуется выполнить следующие действия:

1. Проверьте текущую версию Stronghold:

   ```bash
   stronghold version
   ```

2. Создайте резервную копию данных. Если используется встроенное `Raft`-хранилище, сохраните снимок:

   ```bash
   stronghold operator raft snapshot save stronghold-$(date +%F_%H-%M).snap
   ```

3. При необходимости проверьте снимок:

   ```bash
   stronghold operator raft snapshot inspect stronghold-$(date +%F_%H-%M).snap
   ```

4. Убедитесь, что у вас есть доступ к unseal-ключам.

5. Подготовьте новый бинарный файл Stronghold.

6. Сделайте резервную копию текущего исполняемого файла:

   ```bash
   cp /usr/local/bin/stronghold /usr/local/bin/stronghold.bak
   ```

> Предупреждение  
> Если ваша инсталляция использует другой путь к бинарному файлу, замените `/usr/local/bin/stronghold` на фактический путь.

## Обновление одного узла

Этот сценарий подходит для single-node-инсталляции или для тестового контура.

### Шаг 1. Остановите сервис

```bash
sudo systemctl stop stronghold
```

### Шаг 2. Замените бинарный файл

Скопируйте новый бинарный файл поверх текущего и установите права на выполнение:

```bash
sudo cp ./stronghold /usr/local/bin/stronghold
sudo chmod +x /usr/local/bin/stronghold
```

При необходимости восстановите владельца:

```bash
sudo chown root:root /usr/local/bin/stronghold
```

> Примечание  
> Если в вашей инсталляции бинарный файл запускается не из `/usr/local/bin`, используйте фактический путь.

### Шаг 3. Запустите сервис

```bash
sudo systemctl start stronghold
```

### Шаг 4. Проверьте статус

```bash
sudo systemctl status stronghold
stronghold version
```

Убедитесь, что:

- сервис находится в состоянии `active (running)`;
- версия изменилась на ожидаемую;
- в журналах нет критических ошибок запуска.

### Шаг 5. При необходимости выполните распечатывание

Если auto-unseal не настроен, выполните распечатывание Stronghold:

```bash
stronghold operator unseal
```

Повторите команду необходимое количество раз в соответствии с вашей схемой `Shamir` или recovery-процедурой.

## Обновление кластера с высокой доступностью

Если Stronghold развёрнут в конфигурации `HA`, рекомендуется выполнять обновление поочерёдно, чтобы минимизировать риск простоя.

> Важно  
> Перед обновлением кластера убедитесь, что кластер находится в работоспособном состоянии, а кворум не нарушен.

### Общий порядок

Рекомендуемый порядок действий:

1. определить active-узел;
2. поочерёдно обновить standby-узлы;
3. выполнить `step-down` active-узла;
4. обновить бывший active-узел;
5. проверить состояние кластера.

### Шаг 1. Определите active-узел

```bash
stronghold status
```

Определите текущий active-узел по полям `HA Mode` и `Active Node Address`.

### Шаг 2. Обновите standby-узлы

На каждом standby-узле:

1. Остановите сервис:

   ```bash
   sudo systemctl stop stronghold
   ```

2. Сделайте резервную копию текущего бинарного файла:

   ```bash
   cp /usr/local/bin/stronghold /usr/local/bin/stronghold.bak
   ```

3. Замените бинарный файл:

   ```bash
   sudo cp ./stronghold /usr/local/bin/stronghold
   sudo chmod +x /usr/local/bin/stronghold
   ```

4. Запустите сервис:

   ```bash
   sudo systemctl start stronghold
   ```

5. Проверьте состояние:

   ```bash
   sudo systemctl status stronghold
   stronghold version
   stronghold status
   ```

6. Если auto-unseal не настроен, выполните распечатывание:

   ```bash
   stronghold operator unseal
   ```

Переходите к следующему standby-узлу только после успешной проверки текущего.

### Шаг 3. Выполните step-down active-узла

После обновления standby-узлов переведите active-узел в standby:

```bash
stronghold operator step-down
```

Затем проверьте, что роль active перешла к другому узлу:

```bash
stronghold status
```

### Шаг 4. Обновите бывший active-узел

На бывшем active-узле выполните те же действия:

```bash
sudo systemctl stop stronghold
cp /usr/local/bin/stronghold /usr/local/bin/stronghold.bak
sudo cp ./stronghold /usr/local/bin/stronghold
sudo chmod +x /usr/local/bin/stronghold
sudo systemctl start stronghold
sudo systemctl status stronghold
stronghold version
stronghold status
```

Если auto-unseal не настроен, выполните распечатывание:

```bash
stronghold operator unseal
```

### Шаг 5. Проверьте кластер после обновления

После завершения обновления всех узлов проверьте:

```bash
stronghold status
stronghold operator raft list-peers
```

Убедитесь, что:

- кластер доступен;
- есть active-узел;
- кворум сохранён;
- все узлы присутствуют в списке peers;
- версия Stronghold соответствует ожидаемой на всех узлах.

## Что проверять после обновления

После обновления рекомендуется выполнить минимальный набор проверок:

1. Проверить версию:

   ```bash
   stronghold version
   ```

2. Проверить статус:

   ```bash
   stronghold status
   ```

3. Проверить доступность API:

   ```bash
   curl -k https://127.0.0.1:8200/v1/sys/health
   ```

4. При необходимости проверить веб-интерфейс.

5. Выполнить тестовую операцию чтения или записи секрета, если это допустимо в вашем контуре.

## Что делать при проблемах

Если после обновления Stronghold не запускается или работает некорректно:

1. Проверьте журналы:

   ```bash
   journalctl -u stronghold -f
   ```

2. Проверьте, что новый бинарный файл корректно заменён и имеет права на выполнение.

3. Проверьте совместимость конфигурации с новой версией.

4. Если auto-unseal не настроен, убедитесь, что выполнено распечатывание.

5. При необходимости верните предыдущий бинарный файл:

   ```bash
   sudo cp /usr/local/bin/stronghold.bak /usr/local/bin/stronghold
   sudo systemctl restart stronghold
   ```

> Предупреждение  
> Возврат бинарного файла не всегда гарантирует безопасный откат, если новая версия уже изменила внутреннее состояние или формат данных. Перед откатом ориентируйтесь на release notes и проверяйте сценарий в тестовом контуре.

## Практические рекомендации

- Всегда создавайте резервную копию перед обновлением.
- Для production-окружений используйте окно обслуживания.
- В `HA`-кластере обновляйте узлы поочерёдно.
- Не обновляйте все узлы одновременно без крайней необходимости.
- Проверяйте release notes перед каждым обновлением.
- Заранее проверяйте доступность unseal- или recovery-ключей.
- Если используется внешний `seal`-механизм, убедитесь, что внешний KMS или HSM доступен после перезапуска.
