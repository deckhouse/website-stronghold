В составе **Deckhouse Kubernetes Platform** Stronghold работает как управляемый платформенный компонент. В этом сценарии его версия привязана к версии **Deckhouse Kubernetes Platform**, поэтому обновление Stronghold происходит вместе с обновлением платформы или при подтверждении нового релиза.

> Важно  
> Перед обновлением ознакомьтесь с [Историей версий](../../about/release-notes/) и убедитесь, что в новой версии нет изменений, влияющих на обратную совместимость.

## Как обновляется Stronghold в DKP

В DKP Stronghold не обновляется заменой бинарного файла вручную. Вместо этого обновление происходит через жизненный цикл платформы:

- при автоматическом применении новых релизов DKP Stronghold обновляется автоматически;
- при ручном режиме обновления администратор подтверждает релиз DKP, после чего платформа применяет изменения;
- после обновления платформа пересоздаёт поды Stronghold с новыми образами.

Таким образом, Stronghold обновляется как часть платформы, а не как отдельный сервис.

## Перед началом

Перед обновлением рекомендуется:

- убедиться, что кластер DKP работает штатно;
- проверить наличие резервной копии данных Stronghold;
- убедиться, что у вас есть доступ к unseal-ключам или настроен auto-unseal;
- проверить текущий канал обновлений;
- ознакомиться с release notes платформы и Stronghold;
- по возможности выполнять обновление в окно обслуживания.

> Примечание  
> Даже при наличии высокой доступности рекомендуется проводить обновление в периоды низкой нагрузки на API.

## Рекомендуемая подготовка

Перед запуском обновления выполните следующие проверки.

### Проверьте текущий канал обновлений

```bash
d8 k get mc deckhouse -o jsonpath='{.spec.settings.releaseChannel}'
```

Эта команда покажет текущий канал обновлений DKP, от которого зависит доступность новых релизов.

### Проверьте состояние кластера

Убедитесь, что кластер и системные компоненты находятся в работоспособном состоянии:

```bash
d8 k get nodes
d8 k -n d8-system get pods
d8 k -n d8-stronghold get pods
```

Проверьте, что:

- узлы находятся в состоянии `Ready`;
- системные поды DKP работают без критических ошибок;
- поды Stronghold находятся в состоянии `Running` или `Ready`;
- отсутствуют состояния `CrashLoopBackOff`, `ImagePullBackOff`, `ErrImagePull`.

### Создайте резервную копию Stronghold

Перед обновлением рекомендуется создать резервную копию данных Stronghold.

Если Stronghold использует встроенное `Raft`-хранилище, создайте снимок:

```bash
export STRONGHOLD_ADDR=https://$(d8 k -n d8-stronghold get ing stronghold -o json | jq -r '.spec.rules[0].host')
d8 stronghold login -method=oidc -path=oidc_deckhouse
# Либо через токен:
# d8 stronghold login -method=token

d8 stronghold operator raft snapshot save stronghold-$(date +%F_%H-%M).snap
```

При необходимости проверьте снимок:

```bash
stronghold operator raft snapshot inspect stronghold-$(date +%F_%H-%M).snap
```

## Способ A. Автоматическое обновление

Если в кластере настроено автоматическое применение обновлений, Stronghold обновится автоматически при переходе платформы на новый релиз.

В этом сценарии администратор в основном контролирует:

- выбранный канал обновлений;
- окно обновления, если оно настроено в DKP;
- состояние кластера до и после обновления.

После того как DKP перейдёт на новый релиз, платформа автоматически обновит и компоненты Stronghold.

## Способ B. Ручное подтверждение релиза

Если автоматическое применение обновлений отключено, обновление происходит через подтверждение ресурса `DeckhouseRelease`.

### Шаг 1. Посмотрите доступные релизы

```bash
d8 k get deckhousereleases
```

Эта команда покажет список доступных релизов и их состояние.

### Шаг 2. Подтвердите нужный релиз

Подтвердите целевой релиз в соответствии с принятой в вашей инсталляции процедурой:

- через редактирование YAML-ресурса;
- через административный интерфейс Deckhouse, если такой сценарий используется в вашей организации.

После подтверждения релиза платформа начнёт применять обновление.

### Шаг 3. Дождитесь обновления компонентов Stronghold

Во время обновления платформа пересоздаст поды Stronghold с новыми образами.

Проверяйте состояние подов:

```bash
d8 k -n d8-stronghold get pods -w
```

## Что важно учитывать при обновлении

### Высокая доступность

Если Stronghold развернут в нескольких репликах, обновление обычно проходит без полного простоя за счёт платформенного механизма пересоздания подов.

Тем не менее фактическое поведение зависит от:

- количества реплик;
- состояния кластера;
- доступности backend-хранилища;
- схемы распечатывания;
- общей стабильности платформы в момент обновления.

### Распечатывание после обновления

После перезапуска подов Stronghold может оказаться в состоянии `Sealed`.

Поэтому перед обновлением необходимо убедиться, что:

- у вас есть доступ к unseal-ключам;
- либо настроен auto-unseal через внешний KMS или HSM;
- либо используется подходящая для вашей инсталляции автоматическая схема распечатывания.

> Предупреждение  
> Если после обновления Stronghold остаётся в состоянии `Sealed`, сервис формально будет запущен, но не сможет обслуживать пользовательские запросы.

## Что проверять после обновления

После обновления рекомендуется выполнить минимальный набор проверок.

### Проверьте состояние подов Stronghold

```bash
d8 k -n d8-stronghold get pods
```

Убедитесь, что:

- все поды Stronghold находятся в состоянии `Running`;
- контейнеры готовы;
- отсутствуют ошибки перезапуска и загрузки образов.

### Проверьте доступность Stronghold

Если Stronghold опубликован через Ingress, проверьте health-эндпоинт:

```bash
curl -kL https://stronghold.<ваш-домен>/v1/sys/health
```

### Проверьте вход и базовые операции

Выполните вход:

```bash
export STRONGHOLD_ADDR=https://stronghold.<ваш-домен>
d8 stronghold login -method=oidc -path=oidc_deckhouse
```

Либо используйте вход по токену, если это допустимо в вашей инсталляции.

После входа выполните базовую проверку, например:

```bash
d8 stronghold status
```

При необходимости выполните тестовую операцию чтения или записи секрета.

### Проверьте состояние Stronghold в кластере

Дополнительно проверьте:

- доступность namespace `d8-stronghold`;
- состояние Ingress;
- корректность публикации Stronghold через DNS и Ingress;
- журналы Stronghold, если есть признаки ошибки.

Примеры команд:

```bash
d8 k get ns d8-stronghold
d8 k -n d8-stronghold get ing,svc
d8 k -n d8-stronghold logs <имя-пода>
```

## Что делать при проблемах

Если после обновления Stronghold работает некорректно:

1. Проверьте состояние системных компонентов DKP:

   ```bash
   d8 k -n d8-system get pods
   ```

2. Проверьте состояние подов Stronghold:

   ```bash
   d8 k -n d8-stronghold get pods
   ```

3. Проверьте журналы:

   ```bash
   d8 k -n d8-stronghold logs <имя-пода>
   ```

4. Проверьте, не остался ли Stronghold в состоянии `Sealed`.

5. Убедитесь, что Ingress, DNS и TLS продолжают работать корректно.

6. При необходимости используйте резервную копию и процедуру восстановления.

> Примечание  
> Если проблема связана не с самим Stronghold, а с общей платформенной логикой обновления, используйте документацию DKP по обновлению платформы.

## Практические рекомендации

- Перед обновлением всегда создавайте резервную копию.
- Проверяйте release notes Stronghold и DKP.
- Выполняйте обновление в окно обслуживания.
- Перед обновлением проверяйте готовность механизма распечатывания.
- После обновления всегда выполняйте проверку API, входа и базовых операций.
- Не ограничивайтесь проверкой статуса подов — подтверждайте реальную работоспособность Stronghold.
