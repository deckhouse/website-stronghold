Журналы помогают диагностировать проблемы в работе **Deckhouse Stronghold**, проверять состояние сервиса и анализировать ошибки запуска, аутентификации, репликации, интеграций и сетевого взаимодействия.

При работе с журналами важно различать три источника информации:

- **журналы сервера Stronghold** — основной источник технической диагностики;
- **аудит-логи** — журналируют запросы и ответы API и используются в первую очередь для контроля доступа, расследований и соответствия требованиям безопасности;
- **журналы платформенного окружения DKP** — используются при диагностике Stronghold в составе **Deckhouse Kubernetes Platform**.

> Примечание  
> Аудит-логи не заменяют обычные журналы Stronghold и наоборот. Для диагностики инцидентов часто требуется анализировать оба источника.

## Что обычно ищут в журналах

Журналы Stronghold особенно полезны в следующих сценариях:

- сервис не запускается;
- Stronghold остаётся в состоянии `Sealed`;
- возникают ошибки аутентификации;
- нарушена работа `Raft` или потерян кворум;
- есть проблемы с `TLS`, KMS, HSM или внешними identity-провайдерами;
- не работают плагины;
- после обновления сервис ведёт себя не так, как ожидалось.

## Журналы в Standalone

В Standalone-развёртывании Stronghold чаще всего запускается как сервис `systemd`, поэтому основным источником журналов является `journalctl`, если не настроен вывод в отдельный файл.

### Просмотр журналов сервиса

Показать последние записи журнала:

```bash
journalctl -u stronghold
```

Следить за журналом в реальном времени:

```bash
journalctl -u stronghold -f
```

Показать только последние 100 строк:

```bash
journalctl -u stronghold -n 100
```

Эти команды удобно использовать при:

- запуске Stronghold после изменения конфигурации;
- диагностике проблем с `seal` и `unseal`;
- анализе ошибок кластера `Raft`;
- поиске ошибок при работе плагинов.

## Журналы в DKP

Если Stronghold развернут в составе **Deckhouse Kubernetes Platform**, журналы обычно просматриваются через Kubernetes.

### Просмотр списка подов

Сначала определите поды Stronghold:

```bash
d8 k -n d8-stronghold get po
```

### Просмотр журналов пода

Показать журнал конкретного пода:

```bash
d8 k -n d8-stronghold logs <имя-пода>
```

Следить за журналом в реальном времени:

```bash
d8 k -n d8-stronghold logs -f <имя-пода>
```

Если в поде несколько контейнеров, может потребоваться явно указать имя контейнера.

Журналы в DKP особенно полезны при диагностике:

- проблем запуска после обновления;
- ошибок публикации через Ingress;
- проблем с доставкой конфигурации или плагинов;
- ошибок интеграции с платформенными сервисами.

## Настройка журналирования

Stronghold поддерживает базовые параметры настройки журналов через конфигурационный файл.

Наиболее важные параметры:

- `log_level` — уровень логирования;
- `log_format` — формат вывода;
- `log_file` — путь к файлу журнала;
- `log_rotate_duration` — период ротации;
- `log_rotate_bytes` — ротация по размеру;
- `log_rotate_max_files` — число сохраняемых файлов.

### Уровни логирования

Поддерживаются значения:

- `trace`
- `debug`
- `info`
- `warn`
- `error`

Для обычной эксплуатации рекомендуется использовать:

```hcl
log_level = "info"
```

Для диагностики проблем можно временно повысить уровень:

```hcl
log_level = "debug"
```

> Предупреждение  
> Не оставляйте `debug` или `trace` включёнными в production-окружении без необходимости. Это может увеличивать объём журналов и усложнять анализ.

### Формат журналов

Поддерживаются форматы:

- `standard`
- `json`

Пример:

```hcl
log_format = "json"
```

Формат `json` удобен для централизованной обработки журналов и внешних систем анализа.

## Журналы и аудит-логи

Журналы Stronghold и аудит-логи решают разные задачи.

### Журналы Stronghold

Используются для диагностики работы самого сервиса:

- запуск и остановка;
- ошибки конфигурации;
- кластерные события;
- взаимодействие с `Raft`;
- ошибки интеграций;
- сообщения о `seal`, `unseal`, `rewrap` и др.

### Аудит-логи

Используются для фиксации действий через API:

- запросов;
- ответов;
- ошибок выполнения операций;
- identity-контекста и политик.

> Примечание  
> Если нужно понять, почему пользователь получил отказ в доступе или какой токен выполнял запрос, чаще полезнее аудит-лог. Если нужно понять, почему Stronghold не стартует или не может присоединиться к кластеру, нужно смотреть обычные журналы Stronghold.

## Полезные сценарии диагностики

### Проверка запуска после изменения конфигурации

После изменения `config.hcl`:

```bash
systemctl restart stronghold
journalctl -u stronghold -n 100
```

### Проверка проблем с Raft

Если есть признаки потери кворума или проблем с кластером, в журналах часто можно увидеть ошибки подключения к active-узлу, bootstrap challenge и другие сообщения `Raft`.

### Проверка ошибок аутентификации

Если не работает OIDC, LDAP или другой метод аутентификации, журналы Stronghold помогут понять:

- доходит ли запрос до сервиса;
- есть ли ошибка на стороне провайдера;
- корректно ли настроены пути и конфигурация.

### Проверка поведения после обновления в DKP

После обновления полезно проверить журналы подов Stronghold:

```bash
d8 k -n d8-stronghold logs <имя-пода>
```

Это помогает быстро выявить проблемы с новым образом, конфигурацией или распечатыванием.

## Практические рекомендации

- Используйте `info` как базовый уровень логирования для эксплуатации.
- Включайте `debug` только на время диагностики.
- Не путайте журналы Stronghold с аудит-логами.
- В DKP при диагностике Stronghold проверяйте не только журналы Stronghold, но и состояние подов, Ingress и системных компонентов платформы.
- Если проблема связана с кластером `Raft`, сопоставляйте журналы Stronghold с фактическим состоянием узлов и командой `stronghold status`.
