## Проверка состояния

Этот раздел описывает базовую проверку текущего состояния **Deckhouse Stronghold** в процессе эксплуатации.

В отличие от раздела **«Проверка работоспособности»** в руководстве по развёртыванию, здесь речь идёт не о первичной валидации после установки, а о регулярной операционной проверке: доступен ли сервис, корректно ли работает кластер, есть ли active-узел, не нарушен ли кворум, доступен ли API и опубликован ли сервис во внешнем контуре.

## Что проверять регулярно

При эксплуатации Stronghold рекомендуется регулярно проверять:

- доступность сервиса;
- состояние `Initialized` и `Sealed`;
- наличие active-узла;
- состояние `Raft` и кворума, если используется встроенное `Raft`-хранилище;
- доступность API;
- доступность веб-интерфейса, если он используется;
- состояние подов и Ingress в DKP;
- отсутствие признаков деградации после обновлений, перезапусков и изменений конфигурации.

## Общие проверки

Независимо от варианта развёртывания, полезно начать с базовых проверок состояния самого Stronghold.

### Проверка статуса Stronghold

```bash
stronghold status
```

Эта команда позволяет быстро оценить:

- инициализирован ли сервис;
- распечатан ли он;
- какой используется `storage` backend;
- включён ли HA;
- есть ли active-узел;
- в каком состоянии находится кластер.

На что обратить внимание:

- `Initialized` должно быть `true`;
- `Sealed` должно быть `false`;
- `HA Enabled` должно соответствовать вашей схеме развёртывания;
- при использовании `Raft` должны отображаться поля состояния кластера.

### Проверка доступности API

Проверьте health-эндпоинт:

```bash
curl -k https://127.0.0.1:8200/v1/sys/health
```

Если обращение выполняется к удалённому узлу, используйте его фактический адрес или DNS-имя.

Ожидаемый результат:

- сервер отвечает;
- возвращается JSON;
- отсутствуют ошибки соединения и TLS.

### Проверка кластера Raft

Если Stronghold использует встроенное `Raft`-хранилище, проверьте список узлов:

```bash
stronghold operator raft list-peers
```

Эта команда помогает определить:

- сколько узлов участвуют в кластере;
- какой узел является active;
- есть ли признаки потери кворума;
- все ли ожидаемые peers присутствуют.

> Примечание  
> Если команда не возвращает корректный список peers или сообщает об отсутствии active-узла, это может указывать на проблему с кластером `Raft` или потерю кворума.

## Проверка состояния в Standalone

В Standalone-развёртывании Stronghold обычно работает как сервис `systemd`, поэтому дополнительно рекомендуется проверять состояние процесса и доступность локального окружения.

### Проверка сервиса

```bash
systemctl status stronghold
```

Убедитесь, что сервис находится в состоянии:

```text
active (running)
```

Если сервис не активен, переходите к разделу [Журналы](../Журналы/) и [Типовые проблемы](../Типовые проблемы/).

### Проверка процесса и порта

При необходимости можно убедиться, что процесс запущен и порт прослушивается средствами ОС, принятыми в вашей инфраструктуре.

### Проверка веб-интерфейса

Если веб-интерфейс включён, проверьте, что он открывается по ожидаемому адресу:

```text
https://<stronghold-address>:8200/ui/
```

Проверьте:

- страница доступна;
- TLS-сертификат соответствует ожидаемому сценарию;
- вход выполняется без ошибок.

## Проверка состояния в DKP

Если Stronghold развернут в составе **Deckhouse Kubernetes Platform**, состояние сервиса дополнительно зависит от состояния Kubernetes-объектов и платформенной публикации.

### Проверка namespace и подов

Убедитесь, что namespace существует:

```bash
d8 k get ns d8-stronghold
```

Проверьте поды Stronghold:

```bash
d8 k -n d8-stronghold get po
```

Убедитесь, что:

- поды существуют;
- находятся в состоянии `Running`;
- готовы к работе;
- отсутствуют состояния `CrashLoopBackOff`, `ImagePullBackOff`, `ErrImagePull`.

### Проверка Ingress и сервиса

Если Stronghold публикуется через Ingress, проверьте связанные ресурсы:

```bash
d8 k -n d8-stronghold get ing,svc
```

Проверьте, что:

- Ingress создан;
- сервис существует;
- у публикации есть ожидаемые адреса и маршруты.

### Проверка Ingress-контроллера

Если Stronghold недоступен извне, проверьте состояние Ingress-контроллера:

```bash
d8 k -n d8-ingress-nginx get po
```

### Проверка DNS и health-эндпоинта

Проверьте доступность Stronghold через внешний адрес:

```bash
curl -kL https://stronghold.<ваш-домен>/v1/sys/health
```

Если используется OIDC через Dex, дополнительно можно проверить:

```bash
curl -kL https://dex.<ваш-домен>/healthz
```

## На что обращать внимание в выводе status

Команда `stronghold status` остаётся основной точкой проверки текущего состояния сервиса.

Ниже приведены ключевые поля, на которые рекомендуется смотреть в первую очередь.

### Initialized

Показывает, инициализирован ли Stronghold.

Ожидаемое значение в рабочей инсталляции:

```text
Initialized: true
```

### Sealed

Показывает, распечатан ли Stronghold.

Ожидаемое значение в рабочем состоянии:

```text
Sealed: false
```

Если значение `true`, Stronghold не сможет обслуживать пользовательские запросы.

### Storage Type

Показывает используемый `storage` backend.

Это особенно важно при диагностике сценариев с `Raft`, резервным копированием и кворумом.

### HA Enabled и HA Mode

Показывают, включён ли режим высокой доступности и какую роль выполняет текущий узел.

При кластерном развёртывании важно убедиться, что:

- HA включён;
- один из узлов является active;
- остальные узлы работают ожидаемо.

### Raft Committed Index / Raft Applied Index

Для `Raft` эти поля помогают оценить, продвигается ли состояние кластера и нет ли явных признаков рассинхронизации.

## Минимальный операционный чек-лист

Ниже приведён минимальный набор проверок, который удобно выполнять регулярно.

### Для Standalone

1. Проверить сервис:

   ```bash
   systemctl status stronghold
   ```

2. Проверить статус Stronghold:

   ```bash
   stronghold status
   ```

3. Проверить API:

   ```bash
   curl -k https://127.0.0.1:8200/v1/sys/health
   ```

4. Если используется `Raft`, проверить peers:

   ```bash
   stronghold operator raft list-peers
   ```

### Для DKP

1. Проверить поды:

   ```bash
   d8 k -n d8-stronghold get po
   ```

2. Проверить status Stronghold:

   ```bash
   d8 stronghold status
   ```

3. Проверить Ingress и сервис:

   ```bash
   d8 k -n d8-stronghold get ing,svc
   ```

4. Проверить внешний health-эндпоинт:

   ```bash
   curl -kL https://stronghold.<ваш-домен>/v1/sys/health
   ```

## Когда проверку состояния стоит выполнять обязательно

Рекомендуется выполнять проверку состояния:

- после обновления;
- после перезапуска Stronghold;
- после изменений конфигурации;
- после переключения редакции;
- после операций с `seal`, `rewrap`, KMS или HSM;
- после восстановления из резервной копии;
- при подозрении на деградацию кластера;
- перед плановыми работами.

## Если состояние отличается от ожидаемого

Если при проверке состояния обнаружены отклонения:

- сервис не запущен;
- Stronghold остаётся `Sealed`;
- отсутствует active-узел;
- нарушен кворум;
- недоступен внешний endpoint;
- поды в DKP не переходят в `Running`;

используйте следующие разделы:
