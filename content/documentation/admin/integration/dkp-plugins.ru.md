## Плагины в DKP

В **Deckhouse Kubernetes Platform** порядок загрузки плагинов отличается от Standalone-развёртывания: администратор не копирует бинарные файлы на сервер вручную, а описывает список плагинов в `ModuleConfig`.

После этого платформа:

- скачивает плагины по указанным URL;
- проверяет их контрольные суммы;
- помещает плагины в контейнер Stronghold;
- перезапускает Stronghold при изменении списка плагинов.

После загрузки плагин всё равно нужно зарегистрировать в Stronghold и затем включить по нужному пути.

## Как устроен сценарий в DKP

Сценарий подключения внешнего плагина в DKP обычно состоит из следующих этапов:

1. описать плагин в `ModuleConfig`;
2. дождаться, пока платформа скачает и подготовит плагин;
3. зарегистрировать плагин в Stronghold;
4. включить плагин как `secret`, `auth` или `database` engine;
5. проверить работоспособность.

Такой подход соответствует платформенной модели DKP и позволяет централизованно управлять источниками и версиями плагинов.

## Настройка списка плагинов

Список загружаемых плагинов задаётся в `ModuleConfig`.

Пример:

spec:
  enabled: true
  version: 1
  settings:

## Параметры загрузки

Для каждого плагина можно задать следующие параметры:

- `name` — имя бинарного файла плагина;
- `url` — URL, откуда нужно скачать плагин;
- `sha256` — `SHA256`-контрольная сумма плагина;
- `ignoreFailure` — разрешает продолжить запуск Stronghold, даже если данный плагин не удалось скачать;
- `insecureSkipVerify` — отключает проверку TLS-сертификата удалённого сервера;
- `ca` — дополнительный CA-сертификат для проверки TLS-соединения.

> Важно  
> Для production-окружений всегда рекомендуется указывать `sha256` и использовать только доверенные источники плагинов.

## Что происходит при обновлении списка плагинов

Добавление или удаление плагинов вызывает перезапуск Stronghold.

Если плагин не удаётся скачать или провалидировать:

- перезапуск Stronghold блокируется;
- исключение составляют плагины с `ignoreFailure: true`;
- если не проходит проверка контрольной суммы, плагин считается не загруженным и удаляется.

Это означает, что изменение списка плагинов — это эксплуатационно значимое действие, которое следует выполнять осознанно и по возможности в контролируемое окно изменений.

## Закрытые контуры

В закрытых контурах, где у Stronghold нет доступа во внешний интернет, плагин можно разместить внутри самого кластера DKP.

Один из практических вариантов:

1. подготовить контейнер с `nginx`, в который помещён бинарный файл плагина;
2. запустить этот контейнер в Kubernetes;
3. создать внутренний `Service`, который публикует `nginx` внутри кластера;
4. указать в `ModuleConfig` URL вида `http://<service>.<namespace>.svc.cluster.local/...`, чтобы платформа скачивала плагин через внутренний Kubernetes-сервис.

Пример фрагмента `ModuleConfig`:

```yaml
spec:
  settings:
    plugins:
      - name: "vault-plugin-auth-any"
        url: "http://plugin-repo.plugins.svc.cluster.local/vault-plugin-auth-any"
        sha256: "c943b505b39b53e1f4cb07f2a3455b59eac523ebf600cb04813b9ad28a848b21"
```

Такой подход позволяет:

- не открывать исходящий доступ в интернет;
- хранить плагины внутри внутреннего контура;
- централизованно управлять версиями плагинов через внутренний сервис.

## Регистрация плагина

После доставки плагина в контейнер его необходимо зарегистрировать через CLI.

Общий пример:

```bash
PLUGIN_SHA=$(sha256sum <plugin_binary> | awk '{print $1;}')

d8 stronghold plugin register \
  -command <command_to_run_plugin_binary> \
  -sha256 "${PLUGIN_SHA}" \
  -version "<semantic_version>" \
  <plugin_type> \
  <plugin_name>
```

Пример регистрации `secret`-плагина `mykv`:

```bash
d8 stronghold plugin register \
  -command mykvplugin \
  -sha256 "${PLUGIN_SHA}" \
  -version "v1.0.1" \
  secret \
  mykv
```

> Примечание  
> Даже если плагин уже доставлен платформой, Stronghold не начнёт использовать его автоматически до явной регистрации.

## Включение плагина

После регистрации плагин можно включить как `secret` или `auth` engine.

Общий пример:

```bash
d8 stronghold <secrets|auth> enable \
  -path <mount_path> \
  <plugin_name>
```

Пояснение:

- `secrets` — для плагинов типа `secret`;
- `auth` — для плагинов аутентификации;
- `-path` — путь монтирования;
- `plugin_name` — имя, под которым плагин зарегистрирован.

Пример:

```bash
d8 stronghold secrets enable -path test-kv mykv
```

После этого плагин становится доступен через указанный `mount path`.

## Отключение и удаление плагина

Чтобы удалить внешний плагин из эксплуатации:

1. отключите все `secret` и `auth` методы, использующие плагин;
2. снимите плагин с регистрации;
3. удалите плагин из конфигурации `ModuleConfig`.

Пример снятия с регистрации:

```bash
d8 stronghold plugin deregister secret my-custom-plugin
```

После удаления плагина из `ModuleConfig` DKP больше не будет доставлять его в контейнер Stronghold.

## Что проверять после подключения

После загрузки, регистрации и включения плагина рекомендуется проверить:

- что конфигурация `ModuleConfig` применена без ошибок;
- что Stronghold успешно перезапустился, если это требовалось;
- что плагин зарегистрирован в Stronghold;
- что `mount path` создан корректно;
- что операции чтения, записи или аутентификации работают ожидаемо;
- что в журналах отсутствуют ошибки скачивания, валидации, RPC-запуска или регистрации.

При необходимости проверьте:

- состояние подов Stronghold;
- журналы Stronghold;
- очередь задач DKP;
- сетевую доступность источника, из которого загружается плагин.

## Практические рекомендации

- Для production-окружений желательно использовать внутренний репозиторий плагинов или доверенное хранилище артефактов.
- Всегда задавайте `sha256` и проверяйте, что он соответствует реальному бинарному файлу.
- Используйте `ignoreFailure` только для необязательных плагинов.
- Помните, что изменение списка плагинов приводит к перезапуску Stronghold.
- В закрытых контурах заранее продумайте внутренний способ доставки бинарных файлов.
- Перед обновлением версии плагина проверяйте совместимость с текущей версией Stronghold.
- Выполняйте тестирование новых плагинов на отдельном контуре до использования в production-окружении.
