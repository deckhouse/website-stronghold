## Плагины в Standalone

В Standalone-развёртывании внешние плагины администратор доставляет на сервер самостоятельно. После этого плагин необходимо зарегистрировать в каталоге плагинов и только затем включить как `secret`, `auth` или `database` engine.

## Как устроены внешние плагины

Внешние плагины представляют собой отдельные приложения, которые Stronghold запускает и с которыми взаимодействует по RPC.

Это означает:

- процесс плагина не разделяет память с основным процессом Stronghold;
- плагин получает только явно переданные интерфейсы и аргументы;
- сбой плагина не должен приводить к падению всего Stronghold.

Такой подход упрощает изоляцию и снижает риск того, что ошибка в одном расширении нарушит работу всего сервера.

## Предварительные требования

Для работы внешних плагинов в Standalone-конфигурации необходимо:

- указать `api_addr` в конфигурации Stronghold;
- настроить `plugin_directory`;
- поместить бинарные файлы плагинов в этот каталог;
- зарегистрировать плагин в каталоге плагинов.

> Важно  
> Если `plugin_directory` не задан, внешние плагины использовать нельзя.

## Настройка plugin_directory

Stronghold использует параметр `plugin_directory` в конфигурационном файле, чтобы определить каталог, из которого разрешено запускать внешние плагины.

Пример:

```hcl
api_addr = "https://stronghold.example.com:8200"
plugin_directory = "/opt/stronghold/plugins"
```

Важно учитывать:

- `plugin_directory` не должен быть символической ссылкой;
- бинарный файл плагина должен физически находиться в этом каталоге;
- каталог должен быть доступен процессу Stronghold на чтение и выполнение.

После изменения конфигурации Stronghold примените её стандартным способом, например через перезапуск сервиса.

## Подготовка бинарного файла плагина

Перед регистрацией убедитесь, что:

- бинарный файл помещён в `plugin_directory`;
- имя файла соответствует ожидаемому имени плагина;
- файл имеет корректные права доступа;
- файл получен из доверенного источника;
- рассчитана и проверена контрольная сумма `SHA256`.

Пример вычисления контрольной суммы:

```bash
sha256sum /opt/stronghold/plugins/passthrough-plugin
```

## Регистрация внешнего плагина

Перед включением внешний плагин нужно зарегистрировать в каталоге плагинов, чтобы Stronghold мог проверить его подлинность и целостность.

Пример регистрации:

```bash
PLUGIN_SHA=$(sha256sum /opt/stronghold/plugins/passthrough-plugin | awk '{print $1}')

stronghold plugin register \
  -sha256="${PLUGIN_SHA}" \
  secret \
  passthrough-plugin
```

Если плагин использует versioning, можно явно указать версию:

```bash
stronghold plugin register \
  -sha256="${PLUGIN_SHA}" \
  -version="v1.0.0" \
  secret \
  passthrough-plugin
```

Stronghold поддерживает регистрацию нескольких версий одного и того же плагина.

## Включение внешнего плагина

После регистрации плагин можно включить по нужному `mount path`.

Пример для `secret`-плагина:

```bash
stronghold secrets enable -path=my-secrets passthrough-plugin
```

Пример ожидаемого результата:

```text
Success! Enabled the passthrough-plugin secrets engine at: my-secrets/
```

После этого плагин будет отображаться в списке подключённых механизмов.

> Примечание  
> Тип команды зависит от типа плагина. Для плагинов аутентификации используется контур `auth`, для плагинов механизмов секретов — `secrets`, для database plugins — соответствующий сценарий подключения к database engine.

## Отключение плагина

Отключение внешнего плагина выполняется так же, как и для встроенных механизмов.

Пример:

```bash
stronghold secrets disable my-secrets
```

Если необходимо полностью убрать внешний плагин из каталога плагинов, после отключения его можно снять с регистрации:

```bash
stronghold plugin deregister secret passthrough-plugin
```

После этого при необходимости можно удалить сам бинарный файл из `plugin_directory`.

## Версии плагинов

Плагин может сам сообщать semantic version. Если это происходит, Stronghold может автоматически использовать эту информацию при регистрации.

Если версия задаётся вручную:

- она должна соответствовать semantic version;
- рекомендуется использовать формат с префиксом `v`, например `v1.0.0`;
- один и тот же тип и имя плагина могут существовать в каталоге в нескольких версиях.

Если версия не указана при монтировании, Stronghold выбирает плагин по следующему приоритету:

1. плагин, зарегистрированный без версии;
2. плагин с наиболее новой semantic version;
3. встроенный плагин.

> Важно  
> Перед обновлением или сменой версии плагина убедитесь, что новая версия совместима с вашей версией Stronghold и не меняет ожидаемое поведение mount'ов.

## Права на файлы и mlock

Если Stronghold использует `mlock`, бинарным файлам внешних плагинов также может понадобиться право на использование `mlock`.

В таких случаях может потребоваться:

```bash
sudo setcap cap_ipc_lock=+ep /opt/stronghold/plugins/<plugin-binary>
```

Дополнительно Stronghold может проверять владельца и права каталога и файлов плагинов при включённой проверке file permissions.

Рекомендуется убедиться, что:

- каталог плагинов принадлежит ожидаемому пользователю;
- бинарные файлы не доступны на запись посторонним пользователям;
- права доступа соответствуют вашей политике безопасности.

## Что проверять после подключения

После регистрации и включения внешнего плагина рекомендуется проверить:

- что Stronghold видит зарегистрированный плагин;
- что mount успешно создан;
- что команды чтения или настройки плагина выполняются без ошибок;
- что в журналах отсутствуют ошибки запуска RPC-процесса плагина;
- что плагин работает ожидаемо в целевом сценарии.

При необходимости проверьте журналы Stronghold и системные журналы сервера.

## Практические рекомендации

- Храните внешние плагины в отдельном каталоге с ограниченным доступом.
- Проверяйте `SHA256` перед регистрацией.
- Не используйте символические ссылки для `plugin_directory`.
- Для production-окружений заранее проверяйте совместимость версии плагина с версией Stronghold.
- Перед обновлением плагина имейте план отката.
- Не регистрируйте плагины из недоверенных источников.
- Тестируйте новые плагины на отдельном контуре до внедрения в production-окружение.
