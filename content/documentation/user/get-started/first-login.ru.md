После того как доступ к **Deckhouse Stronghold** настроен, следующий шаг — выполнить первый вход и убедиться, что аутентификация работает корректно.

В типовом пользовательском сценарии первый вход выполняется через **OIDC** с помощью CLI `d8 stronghold login`. Если в вашей инсталляции доступен веб-интерфейс Stronghold, войти можно и через него.

## Что должно быть готово до входа

Перед первым входом убедитесь, что:

- у вас есть адрес Stronghold;
- администратор сообщил способ аутентификации;
- вашей учётной записи выданы необходимые права;
- доступ к Stronghold уже настроен, как описано в разделе [Доступ к проекту](../Доступ к проекту/).

Если используется CLI-сценарий, переменная окружения `STRONGHOLD_ADDR` уже должна быть установлена.

Пример:

```bash
export STRONGHOLD_ADDR=https://stronghold.domain.my
```

## Что происходит при входе

Перед началом работы со Stronghold пользователь или сервис должен пройти аутентификацию.

После успешной аутентификации Stronghold:

- проверяет предоставленные данные;
- создаёт токен доступа;
- привязывает к нему политики;
- использует этот токен для последующих запросов.

Именно токен определяет, какие действия пользователь сможет выполнять в Stronghold.

## Первый вход через CLI

Для аутентификации с помощью CLI используется команда `d8 stronghold login`.

В типовом сценарии через OIDC выполните:

```bash
d8 stronghold login -path=oidc_deckhouse -method=oidc -no-print
```

В зависимости от конфигурации вашей инсталляции путь аутентификации может отличаться. Если администратор сообщил другой путь, используйте его.

### Что происходит дальше

После запуска команды CLI инициирует вход через настроенного OIDC-провайдера. Как правило:

- откроется браузер для завершения входа;
- либо CLI покажет URL, который нужно открыть вручную;
- после успешной аутентификации Stronghold выдаст токен доступа;
- CLI начнёт использовать его для последующих команд.

Если браузер не открывается автоматически, URL можно скопировать и открыть вручную.

> Примечание  
> Для CLI-аутентификации через OIDC обычно используется redirect URI на `localhost`, например `http://localhost:8250/oidc/callback`. Эти параметры определяются конфигурацией роли OIDC и настройками провайдера.

## Первый вход через веб-интерфейс

Если в вашей инсталляции включён веб-интерфейс Stronghold, войти можно через него.

Типовой сценарий:

1. Откройте адрес веб-интерфейса Stronghold.
2. Выберите способ входа, например **OIDC**.
3. При необходимости укажите имя роли.
4. Нажмите кнопку входа через провайдера аутентификации.
5. Завершите аутентификацию у внешнего провайдера.

После успешного входа Stronghold откроет пользовательский интерфейс с правами, соответствующими вашей учётной записи и назначенным политикам.

## Как понять, что вход выполнен успешно

После входа через CLI выполните простую команду, которая не изменяет данные:

```bash
d8 stronghold status
```

Если команда выполняется успешно, это означает, что:

- Stronghold доступен;
- аутентификация выполнена;
- CLI использует действующий токен;
- базовый доступ к сервису работает.

При входе через веб-интерфейс признаком успешного входа является открытие интерфейса Stronghold без ошибки аутентификации.

## Что важно знать о токене после входа

После аутентификации Stronghold создаёт токен, который используется для последующих запросов.

Важно учитывать:

- токен создаётся даже при входе через OIDC;
- токен имеет срок действия;
- набор доступных операций определяется прикреплёнными политиками;
- в CLI токен автоматически используется для следующих команд.

> Предупреждение  
> Не пытайтесь опираться на внутреннюю структуру токена в автоматизации. Токены Stronghold являются непрозрачными значениями, и их формат не предназначен для анализа со стороны клиента.

## Если вход не работает

Если первый вход не выполняется, проверьте:

- правильно ли указан `STRONGHOLD_ADDR`;
- доступен ли Stronghold из вашей сети;
- корректен ли путь аутентификации;
- доступен ли внешний OIDC-провайдер;
- не блокирует ли браузер открытие redirect URI;
- выданы ли вашей учётной записи нужные права;
- не истёк ли ранее выданный токен.

Если используется OIDC, дополнительно убедитесь, что:

- redirect URI настроен корректно;
- адреса в Stronghold и у провайдера совпадают;
- используются правильные `http` / `https`, хост и порт;
- администратор настроил соответствующую роль аутентификации.

## Практические рекомендации

- Для первого входа используйте тот способ аутентификации, который рекомендован администратором.
- После успешного входа сразу проверьте базовую команду, например `d8 stronghold status`.
- Если вы работаете через CLI, не сохраняйте токены в открытом виде в скриптах и заметках.
- Если вы впервые входите через OIDC, убедитесь, что ваш браузер может открыть страницу провайдера и вернуться на локальный callback URI.
