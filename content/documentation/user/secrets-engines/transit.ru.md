Механизм секретов `Transit` выполняет криптографические операции с данными в процессе передачи. Он не хранит данные, которые получает на вход. Поэтому `Transit` часто используют как «криптографию как услугу» или «шифрование как услугу».

С помощью `Transit` можно:

- шифровать и расшифровывать данные;
- подписывать и проверять данные;
- генерировать хеши и HMAC;
- получать случайные байты;
- выпускать ключи данных для прикладных сценариев.

Основной сценарий использования — приложение отправляет данные в `Transit`, Stronghold шифрует их, а приложение сохраняет уже зашифрованное значение в своей базе данных или другом хранилище. Это снимает с команды разработки ответственность за реализацию шифрования и дешифрования в приложении.

## Когда использовать

Используйте `Transit`, если нужно:

- шифровать данные в приложениях без хранения ключей в коде;
- централизовать криптографические операции;
- разделить хранение данных и управление ключами;
- ротировать ключи без ручного перешифрования на стороне приложения;
- использовать производные ключи для разных контекстов;
- применять конвергентное шифрование там, где одинаковые входные данные должны давать одинаковый шифротекст.

## Как это работает

Базовый сценарий выглядит так:

1. Администратор включает механизм секретов `Transit`.
2. Администратор создаёт именованный ключ.
3. Приложение передаёт в Stronghold данные в Base64.
4. Stronghold возвращает шифротекст.
5. Приложение само хранит этот шифротекст и при необходимости отправляет его обратно в Stronghold для расшифровки.

Важно: Stronghold не хранит переданный открытый текст и не хранит возвращённый шифротекст. Их хранение остаётся на стороне вызывающей системы.

## Настройка

### Шаг 1. Включите механизм секретов Transit

```shell-session
$ stronghold secrets enable transit
Success! Enabled the transit secrets engine at: transit/
```

По умолчанию механизм секретов монтируется по имени `transit/`. Если нужен другой путь, используйте аргумент `-path`.

### Шаг 2. Создайте именованный ключ

```shell-session
$ stronghold write -f transit/keys/my-key
Success! Data written to: transit/keys/my-key
```

Обычно для каждого приложения создают отдельный ключ шифрования.

## Использование

После настройки механизма секретов и получения токена Stronghold с нужными правами можно выполнять криптографические операции.

### Шифрование данных

Для шифрования используйте путь `/encrypt` и имя ключа:

```shell-session
$ stronghold write transit/encrypt/my-key plaintext=$(echo "my secret data" | base64)

Key           Value
---           -----
ciphertext    vault:v1:8SDd3WHDOjf7mq69CyCqYjBXAiQQAVZRkFM13ok481zoCmHnSeDX9vyf7w==
```

Все данные для шифрования нужно передавать в Base64. Это важно, потому что `Transit` может принимать не только текст, но и бинарные данные, например PDF или изображения.

В ответ Stronghold возвращает шифротекст. Префикс и версия в значении помогают Stronghold использовать нужную версию ключа при расшифровке.

> Важно  
> Stronghold не хранит этот шифротекст. Если вы хотите расшифровать данные позже, сохраните результат на стороне приложения или в другом хранилище.
>
> Важно  
> Stronghold HTTP API ограничивает размер запроса до 32 МБ, чтобы снизить риск DoS-атак. Этот лимит можно изменить в блоке `listener` конфигурации Stronghold.

### Расшифровка данных

Для расшифровки передайте шифротекст обратно в Stronghold:

```shell-session
$ stronghold write transit/decrypt/my-key ciphertext=vault:v1:8SDd3WHDOjf7mq69CyCqYjBXAiQQAVZRkFM13ok481zoCmHnSeDX9vyf7w==

Key          Value
---          -----
plaintext    bXkgc2VjcmV0IGRhdGEK
```

Результат возвращается в Base64. Чтобы получить исходные данные, декодируйте строку:

```shell-session
$ base64 --decode <<< "bXkgc2VjcmV0IGRhdGEK"
my secret data
```

Или выполните это в одну команду:

```shell-session
$ stronghold write -field=plaintext transit/decrypt/my-key ciphertext=... | base64 --decode
my secret data
```

Через ACL можно разделить права доступа. Например, администраторам можно разрешить управлять ключами, а приложениям — только шифровать и расшифровывать данные нужным ключом.

## Ротация ключей

`Transit` поддерживает версионирование ключей. Это позволяет ротировать ключи без потери доступа к уже зашифрованным данным.

### Как устроено версионирование

У каждого ключа может быть несколько версий. Версии старше значения `min_decryption_version` архивируются, а остальные входят в рабочий набор.

Это важно по двум причинам:

- **производительность** — Stronghold быстрее загружает актуальные версии ключей;
- **безопасность** — можно запретить расшифровку старыми версиями ключей.

При необходимости `min_decryption_version` можно временно сдвинуть назад, чтобы разрешить легитимную расшифровку старых данных в аварийной ситуации.

### Когда ротировать ключи

Периодическая ротация рекомендуется даже без признаков компрометации ключа. Для ключей AES-GCM ориентир — около `2^32` операций шифрования на одну версию ключа, согласно рекомендациям NIST 800-38D.

Например, если ключ используется около 40 миллионов раз в день, ротации раз в три месяца будет достаточно.

### Поверните ключ

Чтобы создать новую версию ключа, выполните:

```shell-session
$ stronghold write -f transit/keys/my-key/rotate
Success! Data written to: transit/keys/my-key/rotate
```

После этого новые операции шифрования будут использовать новую версию ключа. Старые данные по-прежнему можно расшифровать, потому что Stronghold хранит связку версий этого именованного ключа.

## Повторное обёртывание данных без раскрытия открытого текста

Если данные уже зашифрованы старой версией ключа, их можно обновить на новую версию через `rewrap`.

Stronghold расшифрует значение внутренне, а затем снова зашифрует его новейшей версией ключа. При этом открытый текст не будет раскрыт вызывающей стороне.

Пример:

```shell-session
$ stronghold write transit/rewrap/my-key ciphertext=vault:v1:8SDd3WHDOjf7mq69CyCqYjBXAiQQAVZRkFM13ok481zoCmHnSeDX9vyf7w==

Key           Value
---           -----
ciphertext    vault:v2:...
```

Это позволяет выдавать право на `rewrap` даже недоверенным процессам, не предоставляя им доступ к самим данным.

## Рабочий набор и архив ключей

Архив старых версий ключа сейчас хранится в одной записи хранилища. В некоторых бэкендах, особенно при использовании Raft или Paxos для высокой доступности, частая ротация может привести к слишком большому размеру этой записи.

Если ключ приходится ротировать очень часто, хорошей альтернативой может быть использование нескольких именованных ключей, привязанных к временным интервалам. Например, можно создавать отдельный ключ на каждые 5 минут и детерминированно выбирать его по времени.

## Типы ключей

Механизм секретов `Transit` поддерживает несколько типов ключей.

### Ключи для шифрования и дешифрования

- `aes128-gcm96` — AES-GCM с 128-битным ключом AES и 96-битным nonce; поддерживает шифрование, дешифрование, производные ключи и конвергентное шифрование;
- `aes256-gcm96` — AES-GCM с 256-битным ключом AES и 96-битным nonce; поддерживает шифрование, дешифрование, производные ключи и конвергентное шифрование; это тип по умолчанию;
- `chacha20-poly1305` — ChaCha20-Poly1305 с 256-битным ключом; поддерживает шифрование, дешифрование, производные ключи и конвергентное шифрование.

### Ключи для подписи

- `ed25519` — поддерживает подпись, проверку подписи и производные ключи;
- `ecdsa-p256` — поддерживает подпись и проверку подписи;
- `ecdsa-p384` — поддерживает подпись и проверку подписи;
- `ecdsa-p521` — поддерживает подпись и проверку подписи.

### RSA-ключи

- `rsa-2048` — поддерживает шифрование, дешифрование, подпись и проверку подписи;
- `rsa-3072` — поддерживает шифрование, дешифрование, подпись и проверку подписи;
- `rsa-4096` — поддерживает шифрование, дешифрование, подпись и проверку подписи.

Для RSA используются такие методы:

- OAEP — для шифрования и дешифрования, с SHA-256 и MGF;
- PSS — для подписи и проверки подписи, с настраиваемой хеш-функцией;
- PKCS#1 v1.5 — для подписи и проверки подписи, с настраиваемой хеш-функцией.

### HMAC

- `hmac` — поддерживает только генерацию и проверку HMAC.

Все типы ключей также поддерживают операции HMAC через отдельный случайно сгенерированный ключ, который создаётся при инициализации или ротации ключа. Для типа `hmac` по умолчанию используется 256-битный ключ.

## Производные ключи и конвергентное шифрование

`Transit` поддерживает производные ключи. Это позволяет использовать один базовый ключ для разных целей, создавая новый ключ на основе контекста, который передаёт клиент.

В этом режиме можно включить конвергентное шифрование. Тогда одинаковые входные данные и одинаковый контекст будут всегда давать одинаковый шифротекст.

### Когда это полезно

Один из типовых сценариев — хранение шифротекста в базе данных, где нужно искать записи с одинаковыми значениями конкретного поля. В этом случае одинаковые значения будут давать одинаковый результат после шифрования.

### Что важно учитывать

У конвергентного шифрования были разные версии реализации:

- **Version 1** — клиент сам передавал nonce; этот вариант гибкий, но опасен при ошибках реализации, и такие ключи нельзя обновить;
- **Version 2** — использовала алгоритмический подход, но была уязвима к офлайн-атакам подтверждения открытого текста для небольших значений;
- **Version 3** — использует другой алгоритм, который лучше защищён от таких атак и генерирует nonce на основе псевдослучайной функции.

Если у вас есть ключи с версией 2, их можно обновить: выполнить `rotate`, а затем повторно обернуть существующие значения. После этого они будут использовать алгоритм версии 3.

## Генерация ключей данных

`Transit` может выдавать высокоэнтропийные ключи данных заданной длины. Обычно такой ключ возвращается в двух формах:

- в зашифрованном виде — под указанным ключом `Transit`;
- в открытом виде — для немедленного использования.

При необходимости возврат открытого значения можно отключить, если этого требуют правила аудита.

## Импорт собственного ключа (BYOK)

`Transit` поддерживает сценарий **Bring your own key** — импорт собственного ключа. Это может понадобиться, если ключ уже существует в HSM или другой системе и его нужно перенести в Stronghold.

> Важно  
> Импорт нужен только в тех случаях, когда ключ уже существует вне Stronghold. С точки зрения безопасности предпочтительнее, когда Stronghold сам генерирует ключ и сам им управляет.

### Шаг 1. Получите wrapping key

Сначала прочитайте wrapping key из `Transit`:

```shell-session
stronghold read transit/wrapping_key
```

Stronghold вернёт 4096-битный публичный RSA-ключ. Его нужно использовать для подготовки значения, которое затем будет передано в `import`.

### Импорт из HSM

Если ключ импортируется из HSM с поддержкой PKCS#11, возможны два сценария:

- если HSM поддерживает механизм `CKM_RSA_AES_KEY_WRAP`, используйте его для обёртывания целевого ключа wrapping key;
- если этот механизм недоступен, сначала сгенерируйте 256-битный AES-ключ, затем оберните целевой ключ через `CKM_AES_KEY_WRAP_KWP`, а сам AES-ключ — через `CKM_RSA_PKCS_OAEP` с `MGF1` и одной из функций SHA-1, SHA-224, SHA-256, SHA-384 или SHA-512.

После этого шифротекст формируется как конкатенация обёрнутого целевого ключа и обёрнутого AES-ключа. Получившиеся байты нужно закодировать в Base64.

### Импорт вручную

Если целевой ключ не хранится в HSM или KMS, используйте такой порядок:

1. Сгенерируйте эфемерный 256-битный AES-ключ.
2. Оберните целевой ключ этим AES-ключом с помощью AES-KWP.
3. Оберните AES-ключ в wrapping key Stronghold через RSAES-OAEP с MGF1 и одной из хеш-функций SHA-1, SHA-224, SHA-256, SHA-384 или SHA-512.
4. Удалите эфемерный AES-ключ.
5. Добавьте обёрнутый целевой ключ к обёрнутому AES-ключу.
6. Закодируйте результат в Base64.

> Важно  
> Если вы импортируете симметричный ключ, например AES или ChaCha20, нужно оборачивать сырые байты ключа без Base64 и других кодировок.  
> Если вы импортируете асимметричный ключ, например RSA или ECDSA, нужно оборачивать ключ в формате **PKCS8** в сыром DER/бинарном виде. PEM и Base64 перед этим использовать не нужно.

## Практические рекомендации

Чтобы механизм секретов `Transit` было проще использовать и сопровождать, придерживайтесь таких правил:

- создавайте отдельный именованный ключ для каждого приложения или сценария;
- передавайте данные в `Transit` только в Base64;
- храните шифротекст в прикладном хранилище, потому что Stronghold его не сохраняет;
- регулярно ротируйте ключи;
- используйте `rewrap`, если нужно обновить шифротекст на новую версию ключа без раскрытия данных;
- заранее оценивайте нагрузку на ключ, чтобы подобрать частоту ротации;
- при очень частой ротации рассмотрите схему с несколькими именованными ключами, привязанными ко времени;
- импортируйте внешние ключи только там, где это действительно нужно, а в остальных случаях создавайте их внутри Stronghold.
