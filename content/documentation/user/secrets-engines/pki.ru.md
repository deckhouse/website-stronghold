Когда механизм секретов `PKI` включён, экземпляр **Deckhouse Stronghold** может работать как сервер центра сертификации (`CA`), генерируя динамические корневые и промежуточные сертификаты, а также подписывая сертификаты по запросу.

Механизм секретов инфраструктуры открытых ключей (`Public Key Infrastructure`, `PKI`) генерирует динамические сертификаты `X.509`. С его помощью сервисы могут получать сертификаты без ручного процесса создания приватного ключа и CSR, отправки запроса в CA и ожидания завершения проверки и подписи.

Это особенно полезно, если нужно:

- выпускать короткоживущие сертификаты для приложений и сервисов;
- автоматизировать работу с внутренним CA;
- отказаться от ручной выдачи и ротации сертификатов;
- использовать эфемерные сертификаты, которые не сохраняются на диск.

## Почему PKI полезен

При относительно коротком TTL сертификатов необходимость их отзыва снижается, а список отзыва сертификатов (`CRL`) остаётся компактным. Это помогает выдерживать высокую нагрузку и позволяет выдавать отдельный сертификат каждому экземпляру приложения, не заставляя разные экземпляры делить один и тот же сертификат.

В некоторых сценариях сертификаты можно получать и держать только в памяти процесса, никогда не записывая их на диск.

## Настройка

Большинство механизмов секретов должно быть заранее настроено, прежде чем ими смогут пользоваться клиенты. Обычно эти шаги выполняет администратор или средство управления конфигурацией.

### Шаг 1. Включите механизм секретов PKI

```bash
d8 stronghold secrets enable pki
```

Пример ожидаемого результата:

```text
Success! Enabled the pki secrets engine at: pki/
```

По умолчанию механизм будет смонтирован по имени `pki/`. Если нужен другой `mount path`, используйте параметр `-path`.

### Шаг 2. Увеличьте TTL механизма

Значение TTL по умолчанию может быть слишком коротким для вашей схемы CA, поэтому его часто увеличивают заранее:

```bash
d8 stronghold secrets tune -max-lease-ttl=8760h pki
```

Пример ожидаемого результата:

```text
Success! Tuned the secrets engine at: pki/
```

Это значение задаёт верхнюю границу TTL для данного mount'а. Отдельные роли могут ограничивать его более коротким сроком.

### Шаг 3. Настройте сертификат CA и приватный ключ

Stronghold может:

- использовать уже существующую пару ключей;
- либо сгенерировать собственный самоподписанный корневой сертификат.

В общем случае рекомендуется хранить корневой CA вне Stronghold и загружать в Stronghold промежуточный CA. Однако для тестовых и упрощённых сценариев можно сгенерировать внутренний корневой сертификат.

Пример генерации:

```bash
d8 stronghold write pki/root/generate/internal \
     common_name=my-website.ru \
     ttl=8760h
```

Пример результата:

```text
Key              Value
---              -----
certificate      -----BEGIN CERTIFICATE-----...
expiration       1756317679
issuing_ca       -----BEGIN CERTIFICATE-----...
serial_number    fc:f1:fb:2c:6d:4d:99:1e:82:1b:08:0a:81:ed:61:3e:1d:fa:f5:29
```

Возвращаемый сертификат полезен как информационный вывод. Приватный ключ при этом безопасно хранится внутри Stronghold.

### Шаг 4. Настройте URL выпускающего сертификата и CRL

```bash
d8 stronghold write pki/config/urls \
     issuing_certificates="http://127.0.0.1:8200/v1/pki/ca" \
     crl_distribution_points="http://127.0.0.1:8200/v1/pki/crl"
```

Пример ожидаемого результата:

```text
Success! Data written to: pki/config/urls
```

Эти значения можно обновить позднее, если схема публикации сертификатов или CRL изменится.

### Шаг 5. Создайте роль

Роль определяет, какие сертификаты можно выпускать через данный `PKI` mount.

Пример:

```bash
d8 stronghold write pki/roles/example-dot-ru \
     allowed_domains=my-website.ru \
     allow_subdomains=true \
     max_ttl=72h
```

Пример ожидаемого результата:

```text
Success! Data written to: pki/roles/example-dot-ru
```

В этом примере роль разрешает выпуск сертификатов для домена `my-website.ru` и его поддоменов с максимальным сроком действия `72h`.

## Использование

После того как механизм настроен и у пользователя или приложения есть токен Stronghold с нужными правами, можно выпускать сертификаты.

### Выпуск сертификата

Сгенерируйте сертификат, записав запрос в путь `/issue/<role>`:

```bash
d8 stronghold write pki/issue/example-dot-ru \
     common_name=www.my-website.ru
```

Пример результата:

```text
Key                 Value
---                 -----
certificate         -----BEGIN CERTIFICATE-----...
issuing_ca          -----BEGIN CERTIFICATE-----...
private_key         -----BEGIN RSA PRIVATE KEY-----...
private_key_type    rsa
serial_number       1d:2e:c6:06:45:18:60:0e:23:d6:c5:17:43:c0:fe:46:ed:d1:50:be
```

В ответ Stronghold возвращает:

- сертификат;
- выпускающий CA;
- приватный ключ;
- тип приватного ключа;
- серийный номер сертификата.

## Что важно учитывать

- `PKI` — это не просто хранилище сертификатов, а полноценный механизм выпуска сертификатов;
- реальный TTL сертификата определяется сочетанием настроек mount'а и роли;
- для production-сценариев чаще используют не внутренний root CA, а внешнюю корневую цепочку и промежуточный CA;
- доступ к путям `PKI` должен быть ограничен политиками;
- чем короче TTL сертификатов, тем проще ротация и тем меньше необходимость в отзыве.

## Практические рекомендации

- Для production-окружений заранее определите, будет ли Stronghold хранить root CA или только промежуточный CA.
- Не выдавайте избыточно длинные TTL без необходимости.
- Создавайте отдельные роли для разных доменов, сред и сценариев.
- Проверяйте, что `issuing_certificates` и `crl_distribution_points` указывают на корректные и доступные адреса.
- Ограничивайте права пользователей и приложений только нужными ролями и путями.
