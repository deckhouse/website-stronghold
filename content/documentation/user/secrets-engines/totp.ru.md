Механизм секретов `TOTP` в Deckhouse Stronghold генерирует одноразовые пароли по стандарту TOTP. Его можно использовать в двух режимах:

- как **генератор** TOTP-кодов;
- как **провайдер** TOTP-кодов.

В режиме генератора механизм секретов `TOTP` может заменить обычные приложения для генерации кодов, например Google Authenticator. В этом случае возможность получать коды защищается политиками, а все операции проходят через аудит.

В режиме провайдера Stronghold сам создаёт секретный ключ и затем проверяет коды, которые пользователи получают во внешнем TOTP-приложении.

## Когда использовать

Используйте механизм секретов `TOTP`, если нужно:

- централизованно работать с одноразовыми кодами;
- генерировать TOTP-коды через Stronghold;
- проверять TOTP-коды, созданные на стороне пользователя;
- ограничить доступ к операциям через политики;
- получить аудит операций с TOTP-кодами.

## Как это работает

Механизм секретов `TOTP` поддерживает два сценария:

- **режим генератора** — Stronghold хранит настроенный ключ и по запросу возвращает новый TOTP-код;
- **режим провайдера** — Stronghold создаёт новый ключ, пользователь добавляет его в TOTP-приложение, а затем Stronghold проверяет введённые коды.

Выбор режима зависит от того, где именно должен создаваться одноразовый код и кто должен его проверять.

## Режим генератора

В этом режиме Stronghold сам генерирует TOTP-коды на основе заранее заданного ключа.

### Когда подходит этот режим

Используйте режим генератора, если хотите:

- заменить отдельные TOTP-генераторы централизованным сервисом;
- выдавать коды через API или CLI;
- контролировать доступ к генерации кодов через политики Stronghold;
- аудировать каждое обращение к TOTP-коду.

### Настройка

#### Шаг 1. Включите механизм секретов TOTP

```shell-session
$ stronghold secrets enable totp
Success! Enabled the totp secrets engine at: totp/
```

По умолчанию механизм секретов монтируется по имени `totp/`. Если нужен другой путь, используйте аргумент `-path`.

#### Шаг 2. Создайте именованный ключ

```shell-session
$ stronghold write totp/keys/my-key \
    url="otpauth://totp/Stronghold:test@test.com?secret=Y64VEVMBTSXCYIWRSHRNDZW62MPGVU2G&issuer=Stronghold"
Success! Data written to: totp/keys/my-key
```

Параметр `url` должен содержать секретный ключ или значение из штрих-кода, который вы получили от стороннего сервиса.

Имя ключа, в этом примере `my-key`, Stronghold использует для идентификации настроенного TOTP-ключа.

### Использование

После настройки механизма секретов и получения токена Stronghold с нужными правами можно запрашивать новые OTP-коды.

#### Сгенерируйте TOTP-код

```shell-session
$ stronghold read totp/code/my-key
Key     Value
---     -----
code    260610
```

Stronghold вернёт текущий TOTP-код для указанного ключа.

### Что важно учитывать

Через ACL можно ограничить доступ так, чтобы доверенные операторы управляли определениями ключей, а пользователи и приложения получали доступ только к нужным учётным данным.

## Режим провайдера

В этом режиме Stronghold сам создаёт секретный ключ и затем проверяет одноразовые пароли, которые пользователь получает в TOTP-приложении.

### Когда подходит этот режим

Используйте режим провайдера, если нужно:

- выпустить новый TOTP-ключ для пользователя;
- передать пользователю URL или штрих-код для подключения TOTP-приложения;
- проверять коды через Stronghold.

### Настройка

#### Шаг 1. Включите механизм секретов TOTP

```shell-session
$ stronghold secrets enable totp
Success! Enabled the totp secrets engine at: totp/
```

По умолчанию механизм секретов монтируется по имени `totp/`. Если нужен другой путь, используйте аргумент `-path`.

#### Шаг 2. Создайте именованный ключ в режиме провайдера

```shell-session
$ stronghold write totp/keys/my-user \
    generate=true \
    issuer=Stronghold \
    account_name=user@test.com

Key        Value
---        -----
barcode    iVBORw0KGgoAAAANSUhEUgAAAMgAAADIEAAAAADYoy0BA...
url        otpauth://totp/Stronghold:user@test.com?algorithm=SHA1&digits=6&issuer=Stronghold&period=30&secret=V7MBSK324I7KF6KVW34NDFH2GYHIF6JY
```

Параметр `generate=true` включает режим провайдера. В ответ Stronghold возвращает:

- `barcode` — Base64-представление штрих-кода;
- `url` — OTP-ссылку.

Оба варианта эквивалентны. Передайте один из них пользователю, который будет проходить аутентификацию через TOTP.

### Использование

После того как пользователь добавит ключ в TOTP-приложение, Stronghold сможет проверять коды.

#### Проверьте TOTP-код

```shell-session
$ stronghold write totp/code/my-user code=886531
Key      Value
---      -----
valid    true
```

Если код корректен, Stronghold вернёт `valid    true`.

## Практические рекомендации

Чтобы с механизмом секретов `TOTP` было проще работать, придерживайтесь таких правил:

- используйте отдельные именованные ключи для разных пользователей и сценариев;
- выбирайте режим генератора, если Stronghold должен сам возвращать TOTP-коды;
- выбирайте режим провайдера, если Stronghold должен выпускать ключи и проверять коды пользователей;
- ограничивайте доступ к ключам и кодам через ACL;
- учитывайте, что операции с TOTP лучше централизовать, если вам важен аудит действий.
