Deckhouse Stronghold может выпускать OIDC-совместимые ID-токены на основе данных механизма идентичности. Эти токены представляют собой подписанные JWT и соответствуют структуре OIDC ID token.

Такой сценарий полезен, если нужно:

- передавать другим приложениям данные об идентичности из Stronghold;
- выпускать подписанные JWT с OIDC-совместимыми claims;
- использовать стандартные OIDC discovery и JWKS endpoint для проверки токенов;
- проверять токены через introspection endpoint.

## Как это работает

Аутентифицированный пользователь или приложение может запросить токен, который Stronghold сформирует для связанной сущности. Нельзя выпустить токен для произвольной сущности — токен всегда создаётся для того, кто делает запрос.

Параметры токена задаются через роль. Роль определяет:

- TTL токена;
- `client_id`, который попадёт в claim `aud`;
- ключ, которым будет подписан токен;
- шаблон дополнительных claims, если он нужен.

Stronghold также публикует открытые ключи на стандартных OIDC discovery и JWKS endpoint. Это позволяет клиентским библиотекам проверять подпись токена без отдельной авторизации в Stronghold.

## Что нужно подготовить

Для выпуска OIDC identity tokens нужно настроить:

- именованный ключ для подписи токенов;
- роль, которая будет определять состав и срок жизни токена.

### Именованные ключи

Именованный ключ — это пара открытый/закрытый ключ, которую Stronghold использует для подписи и проверки токенов. Закрытый ключ Stronghold использует для подписи, а открытый публикует для проверки.

Ключи регулярно ротируются. При ротации Stronghold создаёт новую пару ключей, а предыдущий открытый ключ некоторое время сохраняет для проверки уже выпущенных токенов.

В конфигурации ключа задаются:

- период ротации;
- `verification_ttl` — сколько хранить старый открытый ключ для проверки;
- алгоритм подписи;
- список разрешённых `client_id`.

По умолчанию ключи ротируются каждые 24 часа и ещё 24 часа остаются доступными для проверки после ротации.

### Роли

Роль определяет, каким будет токен. Она связывается с именованным ключом и задаёт требования к токену.

У каждой роли есть связанный `client_id`. Это значение попадает в claim `aud`. Если оператор не задал `client_id` явно, Stronghold сгенерирует его автоматически.

Невозможно выпустить неподписанный ID-токен. Для роли всегда нужен ключ подписи.

## Что содержит токен

OIDC identity token всегда содержит базовые OIDC claims:

- `iss` — URL эмитента;
- `sub` — идентификатор сущности;
- `aud` — `client_id` роли;
- `iat` — время выпуска;
- `exp` — время истечения.

Кроме этого, в токен можно добавить дополнительные claims через шаблон роли.

## Шаблоны токенов

Шаблон роли позволяет включать в токен данные об идентичности. Шаблон задаётся как JSON с подстановкой значений из механизма идентичности Stronghold.

Пример шаблона:

```json
{
  "color": {{identity.entity.metadata.color}},
  "userinfo": {
     "username": {{identity.entity.aliases.usermap_123.metadata.username}},
     "groups": {{identity.entity.groups.names}}
  },
  "nbf": {{time.now}}
}
```

Если запросить токен для сущности, итоговый шаблон может выглядеть так:

```json
{
  "color": "green",
  "userinfo": {
     "username": "bob",
     "groups": ["web", "engr", "default"]
  },
  "nbf": 1561411915
}
```

После этого Stronghold объединит эти данные с базовыми OIDC claims и сформирует итоговый токен.

Пример итогового содержимого:

```json
{
  "iss": "https://10.1.1.45:8200/v1/identity/oidc",
  "sub": "a2cd63d3-5364-406f-980e-8d71bb0692f5",
  "aud": "SxSouteCYPBoaTFy94hFghmekos",
  "iat": 1561411915,
  "exp": 1561412215,
  "color": "green",
  "userinfo": {
    "username": "bob",
    "groups": ["web", "engr", "default"]
  },
  "nbf": 1561411915
}
```

### Что важно учитывать

Шаблон сливается с базовыми claims на верхнем уровне. Поэтому шаблон не должен переопределять стандартные OIDC claims.

Если в шаблоне используются отсутствующие значения, например несуществующие metadata или alias metadata, Stronghold подставит пустую строку или пустой объект — в зависимости от типа данных.

Шаблон можно задавать в обычном виде или в Base64.

## Доступные параметры шаблона

В шаблонах можно использовать такие параметры:

| Имя | Описание |
| --- | --- |
| `identity.entity.id` | Идентификатор сущности |
| `identity.entity.name` | Имя сущности |
| `identity.entity.groups.ids` | Идентификаторы групп сущности |
| `identity.entity.groups.names` | Имена групп сущности |
| `identity.entity.metadata` | Метаданные сущности |
| `identity.entity.metadata.<metadata key>` | Значение конкретного metadata-ключа сущности |
| `identity.entity.aliases.<mount accessor>.id` | ID alias для указанного mount accessor |
| `identity.entity.aliases.<mount accessor>.name` | Имя alias для указанного mount accessor |
| `identity.entity.aliases.<mount accessor>.metadata` | Метаданные alias |
| `identity.entity.aliases.<mount accessor>.metadata.<metadata key>` | Значение конкретного metadata-ключа alias |
| `identity.entity.aliases.<mount accessor>.custom_metadata` | Пользовательские метаданные alias |
| `identity.entity.aliases.<mount accessor>.custom_metadata.<custom_metadata key>` | Значение конкретного custom metadata-ключа alias |
| `time.now` | Текущее время в секундах от Unix epoch |
| `time.now.plus.<duration>` | Текущее время плюс указанная длительность |
| `time.now.minus.<duration>` | Текущее время минус указанная длительность |

## Генерация токенов

Аутентифицированный клиент может запросить токен через endpoint генерации токена. Stronghold создаст его в соответствии с ролью для сущности, связанной с этим клиентом.

Нельзя выпустить токен от имени другой сущности.

## Проверка токенов

Проверять OIDC identity tokens можно двумя способами:

- на стороне клиента через открытые ключи, опубликованные Stronghold;
- через introspection endpoint Stronghold.

### Проверка через OIDC discovery и JWKS

Stronghold публикует стандартные `.well-known` endpoint. Клиентские JWT- и OIDC-библиотеки могут автоматически получать открытые ключи и проверять:

- подпись токена;
- issuer;
- audience;
- срок действия.

Плюс этого способа в том, что библиотеке нужен только сетевой доступ к Stronghold. Дополнительная авторизация не требуется, потому что `.well-known` endpoint доступны без аутентификации.

### Проверка через introspection endpoint

Другой вариант — отправить токен в Stronghold на introspection endpoint.

В ответ Stronghold сообщит:

- активен токен или нет;
- возникли ли ошибки проверки.

Преимущество этого способа в том, что Stronghold дополнительно проверяет, активна ли сама сущность. Этого нельзя узнать только по содержимому JWT.

Для обращения к introspection endpoint уже нужен действующий токен Stronghold и достаточные права доступа.

## Настройка issuer

Параметр `iss` особенно важен для корректной проверки токена. Потребители токена используют issuer URL, чтобы получить открытые ключи Stronghold и проверить подпись.

По умолчанию Stronghold использует в качестве issuer значение `api_addr` текущего экземпляра Stronghold. Это значит, что токены, выпущенные в кластере, нужно проверять в пределах этого же кластера.

При необходимости issuer можно задать явно. В этом случае он должен указывать на путь `identity/oidc` конкретного экземпляра Stronghold и быть доступным по сети для всех клиентов, которые будут проверять токены.

Пример формата:

```text
https://stronghold.example.com:8200/v1/identity/oidc
```

## Практические рекомендации

Чтобы клиенту было проще использовать OIDC identity tokens, придерживайтесь такого порядка:

- сначала определите, какие данные об идентичности должны попадать в токен;
- затем создайте именованный ключ и настройте его ротацию;
- после этого создайте роль с нужным TTL, `client_id` и шаблоном claims;
- для проверки токенов по возможности используйте стандартные OIDC discovery и JWKS endpoint;
- используйте introspection endpoint, если важно дополнительно проверять активность сущности в Stronghold.
