Stronghold Agent нужен там, где приложение должно получать секреты из Deckhouse Stronghold, но не умеет работать с ним напрямую. Чаще всего это относится к приложениям на виртуальных машинах, bare-metal-серверах, legacy-системам и self-hosted-сервисам, которым нужны файлы конфигурации, переменные окружения, ключи или сертификаты.

Эта страница показывает, в каких сценариях Stronghold Agent особенно полезен и как его обычно используют на практике.

## Когда Stronghold Agent подходит лучше всего

Stronghold Agent особенно удобен, если нужно:

- передавать секреты в приложение без изменения его кода;
- работать не в Kubernetes, а на VM или на bare metal;
- автоматически обновлять секреты и токены;
- доставлять секреты в файлы конфигурации, ключи, сертификаты или переменные окружения;
- интегрировать приложения, для которых нет SDK для работы с Deckhouse Stronghold.

## Развёртывание на VM и bare metal

Основной сценарий использования Stronghold Agent — развёртывание на виртуальных машинах и bare-metal-серверах.

В Kubernetes обычно используют нативные механизмы работы с секретами, например CSI-драйверы и сайдкар-контейнеры. На VM и bare metal таких механизмов нет, поэтому нужен отдельный компонент, который будет забирать секреты из Deckhouse Stronghold и доставлять их в приложение.

Что это даёт:

- централизованное управление секретами для всего парка серверов;
- простую интеграцию через `systemd`-сервис;
- автоматическое обновление секретов без простоя;
- поддержку legacy-систем без необходимости менять код приложений.

## Доставка секретов в legacy-приложения

Legacy-приложения часто ожидают конфигурацию в одном из стандартных форматов:

- файлы конфигурации, например `config.ini`, `application.properties`, `.env`;
- переменные окружения;
- файлы ключей и сертификатов.

Stronghold Agent позволяет встроить Deckhouse Stronghold в такие приложения без прямой интеграции с API Stronghold. Для этого Agent может:

- рендерить шаблоны с секретами в файлы;
- запускать приложения с подставленными переменными окружения;
- автоматически перезапускать приложение при обновлении секретов.

### Пример: Spring Boot и переменные окружения

Stronghold Agent может подготовить переменные окружения с секретами:

```bash
DB_USERNAME=v-approle-myapp-abc123
DB_PASSWORD=A1b2C3d4E5f6
DB_HOST=postgres.example.com
```

После этого приложение может использовать их в `application.properties`:

```java
spring.datasource.url=jdbc:postgresql://${DB_HOST}:5432/production
spring.datasource.username=${DB_USERNAME}
spring.datasource.password=${DB_PASSWORD}
```

В этом сценарии приложение не требует изменений. Spring Boot сам подставит значения из переменных окружения.

## Интеграция с приложениями без SDK

Stronghold Agent полезен не только для legacy-приложений, но и для любых систем, у которых нет готового SDK для работы с Deckhouse Stronghold.

Это может быть:

- legacy-приложение на C или C++;
- специализированная система, например SCADA или промышленное ПО;
- бинарное приложение без исходного кода.

В таких сценариях Stronghold Agent выступает как промежуточный слой и передаёт секреты через стандартные механизмы операционной системы — файлы и переменные окружения.

## Автоматическое обновление credentials без перезапуска

Если секреты и учётные данные нужно обновлять регулярно, Stronghold Agent помогает автоматизировать этот процесс.

В зависимости от сценария Agent может:

- обновлять файлы с новыми значениями;
- отправлять приложению сигналы, например `SIGHUP`, чтобы оно перечитало конфигурацию;
- выполнять команды для reload или hot-reload;
- перезапускать процессы при критичных изменениях.

### Пример: обновление сертификатов для nginx

```hcl
template {
  source      = "/etc/nginx/ssl/cert.ctmpl"
  destination = "/etc/nginx/ssl/cert.pem"
  command     = "nginx -s reload"
}
```

В этом примере Stronghold Agent обновляет сертификат и затем выполняет `nginx -s reload`, чтобы nginx перечитал конфигурацию без простоя.

## Получение динамических секретов

Stronghold Agent хорошо подходит для сценариев, где секреты выдаются не как постоянные значения, а динамически и на ограниченное время.

Например:

- временные учётные данные для баз данных;
- TLS-сертификаты с ограниченным сроком действия;
- другие short-lived credentials.

### Временные учётные данные для баз данных

Deckhouse Stronghold поддерживает динамическую генерацию учётных данных для разных СУБД. В черновике перечислены такие примеры:

- PostgreSQL;
- MySQL;
- MongoDB;
- Oracle.

В этом сценарии Stronghold Agent может получать временные логины и пароли, записывать их в файл или передавать через переменные окружения, а затем автоматически обновлять их.

### PKI-сертификаты

Stronghold Agent можно использовать и для работы с сертификатами:

- автоматически получать TLS-сертификаты;
- обновлять их до окончания срока действия;
- подставлять новые сертификаты в файлы приложения.

Это особенно полезно для веб-серверов и внутренних сервисов, которым нужны короткоживущие сертификаты.

## Использование в CI/CD-пайплайнах

Stronghold Agent можно использовать на self-hosted CI/CD runner'ах, например в Jenkins или GitLab Runner.

В этом сценарии Agent работает на сервере runner'а и предоставляет секреты локально — через файлы или другие локальные механизмы.

Типовой порядок работы:

1. Stronghold Agent устанавливается на сервер CI/CD runner.
2. Agent проходит аутентификацию в Deckhouse Stronghold, например через AppRole.
3. Agent рендерит секреты в файлы или предоставляет к ним доступ локально.
4. Пайплайн использует эти секреты в своей работе.

### Типовые сценарии в CI/CD

Stronghold Agent можно использовать для:

- SSH-ключей и `kubeconfig` для развёртывания;
- credentials для доступа к Docker registry;
- credentials для облачных провайдеров.

### Пример: GitLab Runner и SSH-ключ

```yaml
deploy:
  script:
    - ssh -i /var/run/stronghold-agent/deploy_key deploy@server.example.com "cd /app && git pull && systemctl restart app"
```

Роль Stronghold Agent в этом сценарии:

- Agent получает SSH-ключ из Deckhouse Stronghold;
- рендерит его в файл `/var/run/stronghold-agent/deploy_key` с правами `0600`;
- автоматически обновляет файл при ротации ключа;
- GitLab Runner использует этот файл как обычный SSH-ключ.

## Практические рекомендации

Чтобы подобрать подходящий сценарий использования Stronghold Agent, ориентируйтесь на такие правила:

- если приложение работает на VM или bare metal, Stronghold Agent часто будет самым простым способом интеграции;
- если приложение читает конфигурацию из файлов, используйте рендеринг шаблонов;
- если приложение читает настройки из переменных окружения, используйте сценарий с переменными окружения;
- если секреты часто меняются, заранее продумайте, как приложение будет применять обновления — через reload, сигнал или перезапуск;
- если у приложения нет SDK или исходного кода, Stronghold Agent позволяет обойти это ограничение за счёт стандартных механизмов ОС.
