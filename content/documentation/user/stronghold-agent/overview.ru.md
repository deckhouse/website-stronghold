Stronghold Agent — это клиентский демон, который упрощает интеграцию приложений с Deckhouse Stronghold. Он помогает автоматически проходить аутентификацию, управлять токенами и доставлять секреты в приложение без изменения его кода.

Stronghold Agent особенно полезен для приложений, у которых нет встроенной поддержки систем управления секретами. Это часто касается legacy-приложений, сервисов на виртуальных машинах и bare-metal-серверах, а также бинарных приложений без SDK для работы со Stronghold.

Stronghold Agent работает как посредник между приложением и сервером Deckhouse Stronghold:

- автоматически проходит аутентификацию;
- получает и обновляет токены;
- запрашивает секреты;
- записывает секреты в файлы или передаёт их через переменные окружения;
- может обновлять секреты без изменения кода приложения.

![Схема работы Stronghold Agent](/images/stronghold/agent.png)

На схеме показан типовой сценарий: Stronghold Agent получает секреты из кластера Deckhouse Stronghold по HTTPS, читает локальную конфигурацию и затем передаёт секреты приложениям — через файлы конфигурации, рендеринг шаблонов или переменные окружения.

## Когда использовать Stronghold Agent

Используйте Stronghold Agent, если нужно:

- подключить к Deckhouse Stronghold приложение без нативной интеграции;
- доставлять секреты в файлы конфигурации, сертификаты или переменные окружения;
- автоматически обновлять секреты и токены;
- запускать приложения на VM или на bare metal;
- упростить интеграцию legacy-приложений с системой управления секретами.

Основной сценарий использования Stronghold Agent — развёртывание на виртуальных машинах и bare-metal-серверах. В таких окружениях обычно нет нативных механизмов, которые есть в Kubernetes, поэтому нужен отдельный компонент для доставки секретов в приложение.

## Что умеет Stronghold Agent

Stronghold Agent решает несколько практических задач.

### Автоматическая аутентификация

Agent может сам проходить аутентификацию в Deckhouse Stronghold и получать токен для дальнейшей работы. Это избавляет приложение от необходимости работать с аутентификацией напрямую.

### Доставка секретов

Agent может доставлять секреты несколькими способами:

- рендерить секреты в файлы;
- передавать секреты через переменные окружения;
- подготавливать конфигурационные файлы для приложений;
- обновлять ключи и сертификаты.

### Автоматическое обновление

Agent может отслеживать изменения секретов и обновлять их без ручного вмешательства. Если приложение умеет перечитывать конфигурацию или корректно обрабатывать перезапуск, секреты можно обновлять без изменения кода приложения.

### Упрощение интеграции

Если приложение не умеет работать с Deckhouse Stronghold напрямую, Stronghold Agent берёт эту задачу на себя. Это особенно полезно для legacy-приложений, специализированного ПО и сервисов без SDK.

### Дополнительные возможности

В зависимости от сценария Stronghold Agent можно использовать для:

- рендеринга шаблонов через `template`;
- передачи секретов через `env_template` и `exec`;
- автоматической аутентификации через Auto-Auth;
- работы через локальный API Proxy.

Подробно эти возможности описаны в дочерних страницах раздела.

## Типовые сценарии использования

Stronghold Agent подходит для таких сценариев:

- доставка секретов в legacy-приложения;
- работа с файлами конфигурации, `.env`-файлами, ключами и сертификатами;
- получение динамических учётных данных для баз данных;
- обновление TLS-сертификатов;
- запуск self-hosted CI/CD runner'ов;
- интеграция приложений, у которых нет SDK для работы со Stronghold.

Например, Stronghold Agent можно использовать, чтобы:

- записывать секреты в `application.properties`, `config.ini` или `.env`;
- подставлять переменные окружения перед запуском приложения;
- перезагружать сервис после обновления конфигурации;
- автоматически обновлять временные учётные данные для баз данных;
- обновлять TLS-сертификаты перед окончанием срока действия.

## С чего начать

Если вы только начинаете работать с Stronghold Agent, рекомендуем такой порядок:

1. Откройте [Сценарии использования](./use-cases/), чтобы понять, подходит ли Stronghold Agent под ваш сценарий.
2. Перейдите в [Основные возможности](./capabilities/), чтобы выбрать подходящий способ доставки секретов.
3. Затем изучите [Основные настройки](./settings/) и подготовьте конфигурационный файл.
4. После этого используйте [Запуск и управление](./launch_and_control/) для проверки конфигурации и запуска Agent.

## Практические рекомендации

Чтобы начать работу со Stronghold Agent быстрее и без лишних изменений в приложении:

- сначала определите, как приложение получает конфигурацию — из файлов или через переменные окружения;
- для VM и bare-metal-сценариев в первую очередь рассматривайте Stronghold Agent как основной способ интеграции;
- если приложение не поддерживает SDK, используйте templating или `env_template`;
- заранее продумайте, как приложение будет реагировать на обновление секретов: перечитывать файлы, получать сигнал или перезапускаться;
- храните конфигурацию Agent и чувствительные файлы с минимально необходимыми правами доступа.
