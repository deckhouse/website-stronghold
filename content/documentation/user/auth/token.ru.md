## Token

Метод аутентификации по токену (`Token auth`) является встроенным и автоматически доступен в **Deckhouse Stronghold** по пути `/auth/token`.

Этот метод позволяет:

- проходить аутентификацию с помощью уже существующего токена;
- использовать токены для дальнейшей работы с Stronghold;
- создавать новые токены и отзывать существующие, если у клиента есть соответствующие права.

## Когда использовать Token auth

Метод `Token` обычно используется в следующих случаях:

- у пользователя или приложения уже есть действующий токен Stronghold;
- нужно выполнить вход без повторной аутентификации через другой backend;
- используется автоматизация или интеграция с API;
- требуется работать с токеном напрямую как с основным способом доступа.

> Примечание  
> Когда любой другой метод аутентификации успешно завершает вход, Stronghold всё равно создаёт токен. Поэтому `Token auth` можно рассматривать как базовый встроенный механизм доступа, который используется и самостоятельно, и как результат других методов аутентификации.

## Как работает метод Token

Хранилище токенов может использоваться в обход любого другого метода аутентификации: вы можете создавать токены напрямую, а также выполнять различные операции с токенами, например продление и отзыв.

Это означает, что для некоторых сценариев достаточно иметь уже выданный токен, чтобы выполнять дальнейшие действия в Stronghold без повторного прохождения внешней аутентификации.

## Аутентификация через CLI

Для входа с использованием токена выполните:

```bash
d8 stronghold login token=<token>
```

После этого CLI будет использовать переданный токен для последующих запросов.

## Аутентификация через API

При работе с HTTP API токен передаётся в заголовке запроса. Поддерживаются стандартные варианты передачи токена, например:

- `X-Vault-Token: <token>`
- `Authorization: Bearer <token>`

Типовой пример работы через API:

```bash
curl \
   --request POST \
   --data '{"password": "foo"}' \
   http://127.0.0.1:8200/v1/auth/userpass/login/mitchellh
```

В ответе Stronghold возвращает токен в поле `auth.client_token`.

Пример ответа:

```json
{
   "lease_id": "",
   "renewable": false,
   "lease_duration": 0,
   "data": null,
   "auth": {
      "client_token": "c4f280f6-fdb2-18eb-89d3-589e2e834cdb",
      "policies": ["admins"],
      "metadata": {
         "username": "mitchellh"
      },
      "lease_duration": 0,
      "renewable": false
   }
}
```

## Что важно учитывать

- `Token auth` не требует отдельного включения, так как встроен в Stronghold;
- токен даёт доступ в пределах назначенных ему политик;
- если токен истёк или был отозван, аутентификация с его помощью не сработает;
- в пользовательских сценариях токен часто является результатом входа через другой метод, например `OIDC`, `LDAP` или `Userpass`.

## Практические рекомендации

- Используйте `Token auth`, если у вас уже есть действующий токен Stronghold.
- Не передавайте токены через небезопасные каналы.
- Не храните токены в открытом виде в скриптах, заметках или системах без контроля доступа.
- Если токен используется в автоматизации, учитывайте его TTL, возможность продления и порядок отзыва.
