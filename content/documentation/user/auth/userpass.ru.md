## Userpass

Метод аутентификации `userpass` позволяет пользователям проходить аутентификацию в **Deckhouse Stronghold** с помощью комбинации имени пользователя и пароля [10].

Этот метод подходит для сценариев, где:

- требуется простой локальный способ входа;
- не используется внешний identity-провайдер;
- нужно быстро выдать пользователю доступ к Stronghold;
- используется тестовый, внутренний или ограниченный по масштабу контур [10].

## Как работает Userpass

Метод `Userpass` хранит комбинации имени пользователя и пароля непосредственно в Stronghold. Он не читает пользователей и пароли из внешнего источника [10].

Это означает, что:

- учётные записи создаются и поддерживаются внутри Stronghold;
- права доступа определяются через политики, назначенные пользователю;
- метод не зависит от внешнего каталога, такого как `LDAP` или `OIDC`-провайдер.

Все вводимые имена пользователей сохраняются в нижнем регистре. Например, `Mary` и `mary` будут считаться одной и той же учётной записью [10].

## Когда использовать Userpass

`Userpass` полезен, если:

- нужен простой вход по логину и паролю;
- нет необходимости подключать внешнюю систему аутентификации;
- администратор хочет вручную управлять списком пользователей;
- требуется базовый способ входа для локального контура, пилотного стенда или демонстрационной среды.

Если в вашей организации уже используется корпоративная система идентификации, обычно предпочтительнее использовать `OIDC`, `LDAP` или `SAML`.

## Включение метода

Перед тем как пользователи смогут входить через `Userpass`, метод необходимо включить [10].

### Включение по стандартному пути

```bash
d8 stronghold auth enable userpass
```

После этого метод будет доступен по пути:

```text
auth/userpass
```

### Включение по произвольному пути

Если необходимо смонтировать метод по другому пути, используйте параметр `-path`:

```bash
d8 stronghold auth enable -path=<path> userpass
```

Это может быть полезно, если в одной инсталляции нужно использовать несколько независимых точек входа `Userpass` с разными настройками [10].

## Создание пользователя

После включения метода необходимо создать пользователя, которому будет разрешён вход.

Пример:

```bash
d8 stronghold write auth/<userpass:path>/users/mitchellh \
   password=foo \
   policies=admins
```

В результате создаётся пользователь `mitchellh` с паролем `foo`, которому назначается политика `admins` [10].

> Примечание  
> Путь `<userpass:path>` должен соответствовать фактическому пути монтирования метода. Если метод включён по умолчанию, используйте `auth/userpass`.

## Что получает пользователь после входа

После успешной аутентификации Stronghold выдаёт токен, к которому прикрепляются политики пользователя.

Именно этот токен далее используется для всех последующих запросов к Stronghold, а прикреплённые политики определяют доступные действия.

## Блокировка пользователя

Для метода `Userpass` поддерживается механизм блокировки пользователя (`user_lockout`), который защищает от подбора пароля [10].

Если пользователь несколько раз подряд введёт неверные учётные данные, Stronghold на некоторое время прекратит попытки проверки и сразу начнёт возвращать отказ в доступе [10].

### Основные параметры блокировки

- `lockout_threshold` — количество неудачных попыток входа, после которых пользователь блокируется;
- `lockout_duration` — время, на которое пользователь блокируется;
- `lockout_counter_reset` — время, после которого счётчик неудачных попыток сбрасывается при отсутствии новых попыток [10].

### Значения по умолчанию

По умолчанию используются следующие значения:

- `lockout_threshold` — 5 попыток;
- `lockout_duration` — 15 минут;
- `lockout_counter_reset` — 15 минут [10].

### Отключение блокировки

Функцию блокировки пользователя можно отключить через `auth tune`, передав параметр `disable_lockout=true` [10].

> Предупреждение  
> Механизм `user_lockout` поддерживается только методами `userpass`, `ldap` и `approle auth` [10].

## Практические рекомендации

- Используйте `Userpass`, если вам нужен простой локальный вход без внешнего identity-провайдера.
- Не используйте слабые пароли для пользователей Stronghold.
- Назначайте пользователям только необходимые политики.
- Если метод используется в production-окружении, учитывайте риски локального хранения учётных записей и необходимость регулярного управления паролями.
- При большом числе пользователей или корпоративных требованиях к централизованной аутентификации рассмотрите переход на `OIDC`, `LDAP` или `SAML`.
