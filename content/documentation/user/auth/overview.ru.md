## Обзор методов аутентификации

**Deckhouse Stronghold** поддерживает несколько методов аутентификации для разных сценариев: вход пользователей через браузер или CLI, аутентификация приложений и сервисов, интеграция с корпоративными каталогами и использование дополнительных факторов защиты.

После успешной аутентификации Stronghold выдаёт токен, с которым клиент выполняет дальнейшие запросы. Поэтому при выборе метода аутентификации важно учитывать не только способ входа, но и то, кто именно получает доступ к Stronghold:

- пользователь;
- приложение;
- сервис;
- workload в Kubernetes;
- внешний инструмент или интеграция.

> Примечание  
> Конкретный метод аутентификации должен быть заранее включён и настроен администратором. Если нужный метод недоступен в вашей инсталляции, обратитесь к администратору Stronghold.

## Как выбрать подходящий метод

В общем случае методы аутентификации можно условно разделить на три группы:

- **пользовательские методы** — для входа человека через браузер, CLI или веб-интерфейс;
- **машинные методы** — для приложений, сервисов и автоматизации;
- **дополнительные факторы защиты** — для усиления уже существующей схемы входа.

Если вы не уверены, какой вариант использовать, ориентируйтесь на следующий подход:

- если вы входите как пользователь через браузер или корпоративную учётную запись — обычно подходит `OIDC`, `SAML`, `LDAP`, `Userpass` или `WebAuthn`;
- если доступ нужен приложению или сервису — чаще используют `AppRole`, `Kubernetes`, `JWT` или `Token`;
- если для входа требуется дополнительное подтверждение — используйте `MFA`.

## Методы для пользователей

### OIDC

[`OIDC`](../OIDC/) — один из основных методов входа для пользователей. Он позволяет аутентифицироваться через внешний OIDC-провайдер с помощью браузера и поддерживает сценарии для UI и CLI.

Этот метод обычно выбирают, если:

- в организации уже есть OIDC-провайдер;
- нужен единый вход через корпоративную учётную запись;
- требуется удобный пользовательский сценарий для браузера и CLI.

Если вам нужно понять, как Stronghold работает с конкретными OIDC-провайдерами, используйте раздел [Провайдеры OIDC](../OIDC/Провайдеры OIDC/Обзор/).

### SAML

[`SAML`](../SAML/) позволяет аутентифицировать пользователей через внешний `SAML 2.0` identity-провайдер по профилю `Web SSO`.

Этот метод подходит, если:

- в организации уже используется SAML-инфраструктура;
- нужен браузерный вход через корпоративный identity-провайдер;
- требуется интеграция с системами, ориентированными на SAML.

### LDAP

[`LDAP`](../LDAP/) используется для аутентификации через существующий LDAP-каталог или Active Directory.

Этот метод подходит, если:

- учётные записи пользователей уже ведутся в LDAP;
- нужно связать пользователей и группы каталога с политиками Stronghold;
- требуется интеграция с корпоративной directory-службой.

### Userpass

[`Userpass`](../Userpass/) — встроенный метод аутентификации по имени пользователя и паролю.

Он подходит, если:

- требуется простой локальный способ входа;
- не используется внешний identity-провайдер;
- нужен базовый сценарий аутентификации для тестирования, внутренних стендов или ограниченного числа пользователей.

### WebAuthn

[`WebAuthn`](../WebAuthn/) позволяет аутентифицироваться с помощью `FIDO2`-совместимых аутентификаторов и `passkeys`.

Этот метод особенно полезен, если:

- нужен беспарольный вход;
- требуется современный браузерный пользовательский сценарий;
- нужно усилить защиту учётных записей за счёт аппаратного или платформенного аутентификатора.

### Token

[`Token`](../Token/) — встроенный метод аутентификации по токену. Он автоматически доступен в Stronghold и может использоваться как пользователями, так и автоматизацией.

Этот метод обычно применяется, если:

- у пользователя уже есть действующий токен;
- требуется выполнить вход без повторной аутентификации через внешний backend;
- используется сценарий автоматизации или прямой работы с API.

## Методы для приложений и сервисов

### AppRole

[`AppRole`](../AppRole/) предназначен для аутентификации приложений и сервисов. Он особенно хорошо подходит для машинных сценариев, когда нужно выдать приложению доступ на основе роли и дополнительных ограничений.

Этот метод обычно выбирают, если:

- доступ нужен приложению, а не человеку;
- требуется гибкий и управляемый сценарий аутентификации сервисов;
- необходимо разделить доставку `role_id` и `secret_id`;
- используется автоматизация или service-to-service-взаимодействие.

### JWT

[`JWT`](../JWT/) используется, когда клиент уже может предоставить готовый JWT, который Stronghold должен проверить.

Этот метод подходит, если:

- приложение уже получает JWT от доверенного issuer;
- требуется аутентификация без браузерного OIDC-сценария;
- нужно использовать проверки claims и привязки по утверждениям токена.

### Kubernetes

[`Kubernetes`](../Kubernetes/) предназначен для аутентификации workload'ов с помощью токена `ServiceAccount` Kubernetes.

Этот метод особенно полезен, если:

- приложение работает внутри Kubernetes;
- Stronghold используется для доставки секретов в поды;
- нужно выдавать токены Stronghold на основе identity Kubernetes.

## Дополнительный уровень защиты

### MFA

[`MFA`](../MFA/) — это не отдельный базовый способ входа, а дополнительный фактор проверки, который усиливает уже настроенный метод аутентификации.

Stronghold поддерживает разные сценарии MFA, в том числе:

- [`MULTIFACTOR Ldap Adapter`](../MFA/MULTIFACTOR Ldap Adapter/) — для двухфакторной защиты пользователей в LDAP-сценариях;
- [`TOTP`](../MFA/TOTP/) — для проверки одноразовых кодов.

MFA стоит использовать, если:

- требуется усилить защиту пользовательского входа;
- организация предъявляет повышенные требования к безопасности;
- необходимо добавить второй фактор для методов вроде `LDAP`, `Userpass` или других пользовательских сценариев.

## Что выбрать в типовых сценариях

Ниже приведены упрощённые рекомендации:

- **Вход пользователя через корпоративную учётную запись** → `OIDC` или `SAML`
- **Вход пользователя через LDAP / Active Directory** → `LDAP`
- **Простой локальный вход по логину и паролю** → `Userpass`
- **Беспарольный вход** → `WebAuthn`
- **Аутентификация приложения или сервиса** → `AppRole`
- **Аутентификация workload'а в Kubernetes** → `Kubernetes`
- **Аутентификация по уже выданному JWT** → `JWT`
- **Работа с уже существующим токеном Stronghold** → `Token`
- **Усиление существующей схемы входа** → `MFA`
