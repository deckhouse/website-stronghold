Метод `jwt auth` может использоваться для аутентификации в **Deckhouse Stronghold** двумя способами:

- через **OIDC**;
- путём прямого предоставления **JWT**.

В этом разделе рассматривается именно сценарий **OIDC**, при котором пользователь проходит аутентификацию через внешний OIDC-провайдер с помощью веб-браузера. Этот метод может быть инициирован из веб-интерфейса Stronghold или из CLI. Выбор конкретного сценария определяется настройкой роли.

> Примечание  
> Если вам нужен сценарий, в котором клиент напрямую предоставляет JWT без браузерного входа, используйте раздел [JWT](../jwt/). В `JWT`-сценарии Stronghold только проверяет предоставленный токен, тогда как `OIDC` предполагает полноценный браузерный flow через OIDC-провайдера.

## Как работает OIDC в Stronghold

Сценарий OIDC-аутентификации в Stronghold основан на стандартном browser-based flow. Stronghold перенаправляет пользователя к настроенному OIDC-провайдеру, после чего получает результат аутентификации и выдаёт собственный токен Stronghold.

Для `OIDC` Stronghold использует сценарий `Authorization Code` с расширением `PKCE`.

Stronghold включает два встроенных сценария OIDC-аутентификации:

- вход через **Deckhouse Stronghold UI**;
- вход через **CLI** с использованием `d8 stronghold login`.

## Redirect URI

Одна из самых важных частей настройки OIDC — корректная настройка redirect URI.

Redirect URI должен быть:

- задан в Stronghold в параметре роли `allowed_redirect_uris`;
- настроен у внешнего OIDC-провайдера;
- полностью совпадать в обеих системах.

При проверке совпадения учитывайте:

- `http` или `https`;
- `127.0.0.1` или `localhost`;
- номер порта;
- наличие или отсутствие завершающего слеша.

## OIDC для CLI

Если планируется вход через CLI с помощью команды:

```bash
d8 stronghold login -method=oidc
```

необходимо разрешить redirect URI на `localhost`. Типовой пример:

```text
http://localhost:8250/oidc/callback
```

Если при входе через CLI используется другой хост или порт, этот URI также должен совпадать с одним из значений в `allowed_redirect_uris` и быть разрешён у OIDC-провайдера.

### Вход через CLI

По умолчанию для CLI используется путь `/oidc_deckhouse`. Если метод аутентификации включён по другому пути, его нужно указать явно:

```bash
d8 stronghold login -method=oidc -path=oidc_deckhouse role=test
```

Пример поведения CLI:

```text
Complete the login via your OIDC provider. Launching browser to:
https://myco.auth0.com/authorize?redirect_uri=http%3A%2F%2Flocalhost%3A8400%2Foidc%2Fcallback&client_id=r3qXc2bix9eF...
```

После этого браузер откроет страницу OIDC-провайдера для завершения входа. Если браузер не может быть открыт автоматически, URL можно открыть вручную.

### Полезные параметры CLI

Для сценария входа через CLI могут использоваться следующие параметры обратного вызова:

- `mount` — по умолчанию `oidc`;
- `listenaddress` — по умолчанию `localhost`;
- `port` — по умолчанию `8250`;
- `callbackhost` — по умолчанию `localhost`;
- `callbackmethod` — по умолчанию `http`;
- `callbackport` — по умолчанию совпадает со значением `port`.

Обычно изменять эти параметры не требуется.

## OIDC для веб-интерфейса

Stronghold UI поддерживает вход через OIDC напрямую и не требует отдельной настройки самого интерфейса: нужный сценарий подготавливается автоматически при включении Stronghold.

### Вход через UI

Типовой пользовательский сценарий:

1. Выберите метод входа **OIDC**.
2. При необходимости введите имя роли.
3. Нажмите кнопку входа через OIDC-провайдера.
4. Завершите аутентификацию у настроенного провайдера.

## Что настраивается в роли OIDC

Роль OIDC определяет, как именно Stronghold будет обрабатывать результат аутентификации.

Из черновика особенно важно учитывать следующее:

- `allowed_redirect_uris` задаёт допустимые redirect URI;
- `user_claim` — обязательный и ключевой параметр для роли;
- при необходимости могут использоваться дополнительные привязки claims, например `bound_claims`;
- `bound_audiences` для OIDC-ролей обычно не требуется, так как аудитория определяется через `client_id`.

Если параметр роли требует карту значений, её следует задавать целиком как JSON-объект, а не пытаться конфигурировать по частям.

Пример:

```bash
d8 stronghold write auth/oidc/role/demo -<<EOF
{
  "user_claim": "sub",
  "bound_audiences": "abc123",
  "role_type": "oidc",
  "policies": "demo",
  "ttl": "1h",
  "bound_claims": { "groups": ["mygroup/mysubgroup"] }
}
EOF
```

## Практические рекомендации по настройке

При настройке OIDC полезно придерживаться следующего порядка:

- сначала добиться успешного базового входа;
- затем добавлять ограничения по claims и дополнительные проверки;
- отдельно сверять redirect URI у Stronghold и у провайдера;
- запрашивать у провайдера только действительно нужные scopes.

Часто полезно запросить scopes:

```text
profile,groups
```

если Stronghold должен получать профиль пользователя и его группы.

## Отладка OIDC

Если аутентификация через OIDC не работает, в первую очередь рекомендуется:

- проверить журналы Stronghold;
- убедиться, что redirect URI совпадают в Stronghold и у провайдера;
- проверить, что `user_claim` настроен корректно;
- временно не усложнять роль лишними `bound_claims`;
- уточнить у провайдера структуру claims, которые реально приходят в токене.

Если нужно вручную посмотреть содержимое JWT, можно использовать декодирование payload токена. В черновике приводится такой пример:

```bash
cat jwt.json | jq -r .access_token | cut -d. -f2 | base64 -D
```

Это полезно для диагностики claims и сопоставления их с ожидаемой конфигурацией роли.

## verbose_oidc_logging

В Stronghold доступна ролевая опция `verbose_oidc_logging`. Если она включена и журналирование работает на уровне `debug`, Stronghold будет записывать полученный OIDC-токен в журналы сервера.

Это может быть полезно для диагностики, но:

> Предупреждение  
> `verbose_oidc_logging` не следует использовать в production-окружении, так как claims могут содержать чувствительные данные.
