## AppRole

Метод аутентификации `AppRole` предназначен для аутентификации машин и приложений в **Deckhouse Stronghold**. Он особенно полезен в автоматизированных сценариях, где доступ к Stronghold нужен сервису, а не человеку.

`AppRole` — это набор политик и ограничений аутентификации, которые необходимо выполнить для получения токена Stronghold. Область применения может быть как узкой, так и широкой: роль можно настроить для одной машины, для конкретного сервиса или для сервиса, работающего на нескольких машинах.

> Примечание  
> Для `AppRole` рекомендуется использовать `batch` токены, если вашему сценарию не требуется создание дочерних токенов или иные расширенные свойства `service`-токенов.

## Как работает AppRole

В типовом сценарии для входа через `AppRole` используются два значения:

- `role_id`;
- `secret_id`.

Клиент передаёт их в Stronghold, после чего Stronghold:

1. проверяет, что роль существует;
2. проверяет ограничения роли;
3. при успешной проверке выдаёт токен Stronghold с политиками, привязанными к роли.

## Когда использовать AppRole

`AppRole` обычно выбирают, если:

- нужно аутентифицировать приложение или сервис;
- доступ к Stronghold должен выдаваться не пользователю, а workload'у;
- требуется гибко управлять сроком жизни токена, числом использований и другими ограничениями;
- нужно разделить доставку идентификатора роли и секрета роли между разными каналами.

## Включение метода

Методы аутентификации могут быть включены и отключены через UI, CLI или API. Для `AppRole` базовый способ включения через CLI выглядит так:

```bash
d8 stronghold auth enable approle
```

По умолчанию метод будет смонтирован по пути:

```text
auth/approle
```

При необходимости можно включить его по другому пути:

```bash
d8 stronghold auth enable -path=my-login approle
```

Это полезно, если в одной инсталляции нужно использовать несколько независимых экземпляров `AppRole` с разными настройками.

## Аутентификация через CLI

Путь по умолчанию — `/approle`. Если этот метод аутентификации включён по другому пути, укажите нужный путь вместо пути по умолчанию.

Пример аутентификации:

```bash
d8 stronghold write auth/approle/login \
  role_id=db02de05-fa49-4055-059b-67221c5c2f63 \
  secret_id=6a174c20-f6de-a63c-74d2-6018fcceff64
```

Пример результата:

```text
Key                Value
---                -----
token              75b74ffd-842c-fd43-1386-f7d7006e520a
token_accessor     4c29bc22-5c72-11a6-f778-2bc8f48cea0e
token_duration     20m0s
token_renewable    true
token_policies     [default]
```

После этого клиент получает токен Stronghold и может использовать его для дальнейших запросов.

## Аутентификация через API

По умолчанию используется путь:

```text
auth/approle/login
```

Если метод включён по другому пути, используйте этот путь вместо стандартного.

Пример запроса:

```bash
curl \
  --header "X-Vault-Token: ${STRONGHOLD_TOKEN}" \
  --request POST \
  --data '{"role_id":"988a9df-...","secret_id":"37b74931..."}' \
  ${STRONGHOLD_ADDR}/v1/auth/approle/login
```

Пример ответа:

```json
{
  "auth": {
    "renewable": true,
    "lease_duration": 2764800,
    "metadata": {},
    "policies": ["default", "dev-policy", "test-policy"],
    "accessor": "5d7fb475-07cb-4060-c2de-1ca3fcbf0c56",
    "client_token": "98a4c7ab-b1fe-361b-ba0b-e307aacfd587"
  }
}
```

Токен Stronghold возвращается в поле `auth.client_token`.

## Настройка AppRole

Перед тем как приложение сможет пройти аутентификацию, метод `AppRole` должен быть предварительно настроен. Обычно это делает администратор или средство управления конфигурацией.

### Шаг 1. Включите метод

```bash
d8 stronghold auth enable approle
```

### Шаг 2. Создайте роль

Пример создания роли:

```bash
d8 stronghold write auth/approle/role/my-role \
 token_type=batch \
 secret_id_ttl=10m \
 token_num_uses=10 \
 token_ttl=20m \
 token_max_ttl=30m \
 secret_id_num_uses=40
```

> Важно  
> Если токен, выдаваемый вашей ролью `AppRole`, должен уметь создавать дочерние токены, параметр `token_num_uses` необходимо установить в `0`.

### Шаг 3. Получите RoleID

```bash
d8 stronghold read auth/approle/role/my-role/role-id
```

Пример результата:

```text
role_id     db02de05-fa49-4055-059b-67221c5c2f63
```

### Шаг 4. Получите SecretID

```bash
d8 stronghold write -f auth/approle/role/my-role/secret-id
```

Пример результата:

```text
secret_id               6a174c20-f6de-a63c-74d2-6018fcceff64
secret_id_accessor      c454f7e5-996e-7230-6074-6ef26b7bcf86
secret_id_ttl           10m
secret_id_num_uses      40
```

## Настройка через API

### Включение метода

```bash
curl \
 --header "X-Vault-Token: ${STRONGHOLD_TOKEN}" \
 --request POST \
 --data '{"type": "approle"}' \
 ${STRONGHOLD_ADDR}/v1/sys/auth/approle
```

### Создание роли

```bash
curl \
 --header "X-Vault-Token: ${STRONGHOLD_TOKEN}" \
 --data '{"policies": "dev-policy,test-policy", "token_type": "batch"}' \
 ${STRONGHOLD_ADDR}/v1/auth/approle/role/my-role
```

### Получение RoleID

```bash
curl \
 --header "X-Vault-Token: ${STRONGHOLD_TOKEN}" \
 ${STRONGHOLD_ADDR}/v1/auth/approle/role/my-role/role-id
```

Пример ответа:

```json
{
  "data": {
    "role_id": "888a9dfd-ea69-4a53-6cb6-9d6b86474bba"
  }
}
```

### Создание SecretID

```bash
curl \
 --header "X-Vault-Token: ${STRONGHOLD_TOKEN}" \
 --request POST \
 ${STRONGHOLD_ADDR}/v1/auth/approle/role/my-role/secret-id
```

Пример ответа:

```json
{
  "data": {
    "secret_id_accessor": "65946873-1d96-a9d4-678c-9229f74386a5",
    "secret_id": "37b24931-c4cd-d49a-9246-ccc62d682a25",
    "secret_id_ttl": 600,
    "secret_id_num_uses": 40
  }
}
```

## Учетные данные и ограничения

### RoleID

`RoleID` — это идентификатор, который назначается роли `AppRole` и используется как обязательный параметр при входе. По умолчанию это UUID, но при необходимости он может быть установлен в пользовательское значение, если это требуется сценарием.

### SecretID

`SecretID` — это секретные учётные данные, которые по умолчанию также требуются для входа. Они должны оставаться секретными.

У `SecretID` могут быть собственные параметры:

- TTL;
- число использований;
- срок действия;
- ограничения по сценарию выдачи.

## Режимы Pull и Push

Для `SecretID` Stronghold поддерживает два подхода.

### Pull

В режиме `Pull` `SecretID` извлекается из `AppRole` самим клиентом или доверенной стороной. Это основной и предпочтительный сценарий.

### Push

В режиме `Push` клиенту задаётся пользовательский `SecretID` вручную. Этот режим может использоваться для совместимости или специфических сценариев, но в большинстве случаев считается менее предпочтительным.

Причина в том, что при `Push` полный набор учётных данных нужно передать внешней системе, тогда как при `Pull` можно безопаснее разделить распространение `RoleID` и `SecretID`.

> Примечание  
> Для доставки `SecretID` в режиме `Pull` часто удобно использовать [обёртывание ответа](/products/stronghold/documentation/Руководство пользователя/Основные понятия/Обёртывание ответа/), чтобы не передавать секрет в открытом виде.

## Дополнительные ограничения

Помимо `role_id` и `secret_id`, роль `AppRole` может иметь дополнительные ограничения. Например, можно ограничить доступ по CIDR-диапазонам или другим параметрам, если это требуется конкретному сценарию.

Это позволяет использовать `AppRole` не просто как пару идентификаторов, а как управляемую модель машинной аутентификации с дополнительными правилами допуска.

## Практические рекомендации

- Используйте `AppRole` для приложений и сервисов, а не для интерактивного входа пользователей.
- По возможности предпочитайте режим `Pull` для `SecretID`.
- Разделяйте каналы доставки `RoleID` и `SecretID`.
- Используйте `batch` токены, если вашему сценарию не нужны расширенные свойства `service`-токенов.
- Заранее планируйте TTL, число использований и возможные ограничения роли.
- Если `SecretID` нужно передавать через недоверенный контур, используйте обёртывание ответа.
