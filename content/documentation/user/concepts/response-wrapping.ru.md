Во многих сценариях развёртывания **Deckhouse Stronghold** клиенты напрямую обращаются к Stronghold и используют возвращаемые секреты. Однако в некоторых ситуациях может быть целесообразнее разделить привилегии таким образом, чтобы одна доверенная сторона отвечала за взаимодействие с большей частью API Stronghold и передавала секреты конечному потребителю.

Чем больше посредников задействовано в передаче секрета, тем выше риск случайного раскрытия, особенно если секрет передаётся в открытом виде. Например, если необходимо доставить закрытый TLS-ключ на машину, где по политике безопасности нельзя хранить ключ дешифрования в постоянном хранилище. В этом случае невозможно безопасно зашифровать ключ перед передачей без дополнительного механизма.

Для решения подобных задач в Stronghold предусмотрена функция _обёртывания ответа_ (`response wrapping`). Вместо того чтобы сразу вернуть ответ HTTP-клиенту, Stronghold сохраняет его в [`cubbyhole`](/products/stronghold/documentation/Руководство пользователя/Механизмы секретов/Cubbyhole/). Доступ к содержимому есть только у одноразового токена, который Stronghold возвращает клиенту.

С логической точки зрения ответ оборачивается токеном, который нужно распаковать, чтобы получить доступ к данным. С функциональной точки зрения токен авторизует доступ к «ключнице» Stronghold и позволяет получить сохранённый ответ.

## Зачем нужно обёртывание ответа

Обёртывание ответа решает три ключевые задачи.

### Скрытие секретной информации

Передаваемое по сети значение не является самим секретом, а лишь токеном доступа к нему. Даже если данные были перехвачены или попали в журналы, они не содержат конфиденциальной информации.

### Обнаружение злоупотреблений

Токен можно распаковать только одной стороне. Если клиент получил токен, который не распаковывается, это повод инициировать инцидент безопасности. До распаковки клиент может проверить происхождение токена через Stronghold.

### Ограничение срока действия

У токена есть собственное время жизни (`TTL`), отдельное от TTL секрета и часто значительно более короткое. Если клиент не распакует токен вовремя, срок его действия истечёт.

## Токены обёртывания ответа

При обёртывании исходный ответ Stronghold на API-запрос не возвращается напрямую. Вместо этого клиент получает набор данных о токене обёртывания:

- `TTL` — срок жизни токена;
- токен — значение токена;
- время создания — момент генерации токена;
- путь создания — API-endpoint, вызвавший генерацию ответа;
- идентификатор обёрнутого токена — указывает на обёрнутый токен, например при аутентификации или создании нового токена.

Последний параметр особенно полезен для систем оркестрации и автоматизации, поскольку позволяет управлять сроком действия токена без раскрытия самого секрета.

## Почему токены обёртывания не подписываются

В текущей модели Stronghold не использует подпись токенов обёртывания, поскольку это не даёт существенного дополнительного уровня защиты.

Причина в том, что:

- токен обёртывания не содержит сам секрет;
- токен лишь даёт право обратиться к Stronghold и получить сохранённый ответ;
- без проверки токена на стороне самого Stronghold получить данные невозможно.

Даже если злоумышленник попытается перенаправить клиента на поддельный сервер, надёжная проверка всё равно сводится к взаимодействию с корректным сервером Stronghold, а не к локальной проверке подписи токена.

Таким образом, Stronghold опирается на то, что токен не несёт чувствительных данных сам по себе и не требует отдельной подписи.

## Операции с токенами обёртывания

Через путь `sys/wrapping/` доступны следующие операции с токенами обёртывания.

### Поиск

Путь `sys/wrapping/lookup` возвращает:

- время создания токена;
- путь создания;
- TTL.

Этот путь не требует полноценной аутентификации владельца секрета: держатель токена всегда может запросить свойства токена по этому пути.

### Распаковка

Путь `sys/wrapping/unwrap` распаковывает токен и возвращает хранящийся внутри ответ в исходном API-формате.

### Повторное обёртывание

Путь `sys/wrapping/rewrap` позволяет заново обернуть уже обёрнутые данные в новый токен.

Это полезно для данных с длительным сроком жизни, когда требуется:

- продлить срок безопасной доставки;
- выполнить ротацию wrapping token;
- сохранить данные в нераскрытом виде, но при этом не потерять возможность их использовать позже.

Такой сценарий особенно полезен для чувствительных данных, которые организация должна хранить так, чтобы они не раскрывались напрямую, но при этом оставались доступными для последующего использования.

### Явное обёртывание

Путь `sys/wrapping/wrap` возвращает данные в виде токена обёртывания.

> Примечание  
> Даже при ограничении доступа к этому пути обёртывание данных может выполняться и через другие API-методы Stronghold, если клиент запрашивает wrapping для конкретного ответа.

## Как создаётся токен обёртывания

Обёртывание ответа выполняется для каждого запроса отдельно и включается путём указания желаемого TTL.

TTL задаётся клиентом:

- через заголовок `X-Vault-Wrap-TTL`;
- в CLI — через параметр `-wrap-ttl`.

TTL можно указать:

- целым числом в секундах;
- строкой, например `15s`, `20m`, `25h`.

### Процесс обёртывания

1. Исходный HTTP-ответ сериализуется.
2. Генерируется одноразовый токен с TTL, заданным клиентом.
3. Ответ сохраняется в `cubbyhole`, связанный с токеном.
4. Генерируется новый ответ с дополнительным полем, содержащим информацию о токене.
5. Новый ответ возвращается клиенту.

> Примечание  
> Минимальный и максимальный TTL токенов обёртывания регулируются [политиками](../policy/).

## Проверка токена обёртывания

Чтобы снизить риски злоупотреблений, рекомендуется проверять токены обёртывания по следующей схеме.

### 1. Убедитесь, что токен вообще был получен

Если клиент ожидал токен, но он не поступил, это может быть признаком того, что токен был перехвачен или обработан не той стороной. В таком случае следует начать расследование.

### 2. Выполните lookup токена

Если токен истёк или был отозван, это не всегда означает компрометацию. Например, клиент мог слишком долго запускаться. Тем не менее ситуацию нужно расследовать.

Используя журналы аудита Stronghold, проверьте, не был ли токен уже распакован кем-то ещё.

### 3. Сверьте путь создания

Путь создания токена должен соответствовать ожидаемому сценарию.

Например, если вы ожидаете получить TLS-ключ или сертификат, путь обычно должен быть вида:

```text
pki/issue/...
```

Несоответствие пути может означать подмену токена обёртывания. Это особенно важно для механизма секретов `KV`: если ожидается секрет из `secret/foo`, а фактически передан токен с путём `secret/bar`, простой проверки общего префикса `secret/` недостаточно.

### 4. Выполните unwrap

После проверки токен нужно распаковать. Если распаковка невозможна, это повод начать расследование инцидента.

## Что даёт такой подход

Если следовать описанной процедуре, можно быть уверенным, что:

- данные в токене обёртывания видел только целевой клиент;
- секрет не передавался в открытом виде;
- попытка подмены или перехвата токена не останется незамеченной;
- срок жизни токена ограничивает окно атаки.

## Практический смысл для пользователя

Обёртывание ответа особенно полезно в сценариях, где:

- одна система получает секрет от Stronghold и передаёт его другой;
- необходимо исключить передачу чувствительных данных в открытом виде;
- важно обнаружить факт несанкционированного доступа;
- требуется ограничить время, в течение которого данные доступны к получению.
