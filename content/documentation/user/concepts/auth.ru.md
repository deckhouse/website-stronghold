## Аутентификация

Аутентификация в **Deckhouse Stronghold** — это процесс, в ходе которого информация, предоставленная пользователем или машиной, проверяется на соответствие внутренней или внешней системой. Stronghold поддерживает множество методов аутентификации, включая `Userpass`, `LDAP`, `AppRole` и другие. У каждого метода аутентификации есть свой сценарий использования.

Перед взаимодействием со Stronghold каждый клиент должен пройти аутентификацию, используя один из поддерживаемых методов. После успешной аутентификации Stronghold генерирует токен, который по своей сути похож на идентификатор сессии на веб-сайтах. К токену могут быть прикреплены политики, которые определяют, какие действия разрешены после входа.

## Как связаны аутентификация и токен

В ходе аутентификации выполняется проверка подлинности идентификатора, после чего Stronghold создаёт токен, связанный с этим идентификатором.

Например, даже если вы проходите аутентификацию через **Dex** или другой внешний identity-провайдер, Stronghold всё равно создаёт собственный токен доступа, который используется для последующих запросов.

В CLI этот токен автоматически добавляется к запросам. При работе через API его необходимо передавать явно.

Токен, выдаваемый при аутентификации через любой backend, можно также использовать с полным набором команд для работы с токенами, например:

- создавать вложенные токены;
- продлевать токены;
- отзывать токены.

Подробнее см. раздел [Токены](../Токены/).

## Методы аутентификации

Stronghold поддерживает несколько методов аутентификации. Часть из них ориентирована на пользователей, часть — на приложения и сервисы. Большинство методов аутентификации необходимо включить перед использованием.

Чтобы включить метод аутентификации, используйте следующую команду:

```bash
d8 stronghold write sys/auth/my-auth type=userpass
```

Эта команда включает метод аутентификации `userpass` на пути `my-auth`. Использовать такую аутентификацию можно именно по пути `my-auth`. Часто путь аутентификации совпадает с её именем, однако это не является обязательным требованием.

Чтобы получить дополнительные сведения об этой аутентификации, используйте встроенную команду `path-help`:

```bash
d8 stronghold path-help auth/my-auth
```

Stronghold поддерживает одновременно несколько методов аутентификации. Один и тот же тип метода аутентификации можно включать на разных путях. Для получения доступа к Stronghold требуется единовременная аутентификация, хотя некоторые backend'ы поддерживают сценарии с многофакторной аутентификацией.

> Примечание  
> Подробное описание конкретных методов приведено в разделе [Методы аутентификации](/products/stronghold/documentation/Руководство пользователя/Методы аутентификации/).

## Аутентификация через CLI

Для аутентификации с помощью CLI используется команда `d8 stronghold login`.

Поддерживается несколько методов аутентификации. Например, чтобы пройти аутентификацию с помощью OIDC, выполните следующую команду:

```bash
d8 stronghold login -method=oidc
```

После аутентификации вы войдёте в систему. Команда CLI также может вывести токен, если это допускается выбранным способом входа и параметрами команды. Этот токен используется для продления и отзыва токена в дальнейшей работе.

Чтобы определить, какие параметры требуются для конкретного метода аутентификации, используйте флаг `-method` без дополнительных аргументов и изучите вывод справки.

Если используется метод, который не поддерживает полноценный интерактивный вход через CLI, потребуется использовать API или иной поддерживаемый сценарий.

## Аутентификация через API

Аутентификация через API обычно используется для автоматизированных сценариев. Каждый метод аутентификации реализует свой endpoint для входа.

Чтобы найти нужный путь, используйте:

```bash
d8 stronghold path-help
```

При работе через API после успешной аутентификации Stronghold возвращает токен в ответе, и его необходимо использовать при последующих запросах.

## Аутентификация и аренда

Как и у секретов, у результатов аутентификации есть срок действия аренды. Это означает, что доступ к Stronghold не обязательно является бессрочным: через заданный интервал может потребоваться повторная аутентификация или продление токена.

Чтобы установить срок действия аренды для конкретного идентификатора, необходимо учитывать настройки выбранного метода аутентификации. Каждый backend реализует свою схему TTL и продления.

Как и в случае с секретами, срок действия аутентификации можно продлить без повторного входа, если это разрешено конфигурацией. Для этого используется команда:

```bash
d8 stronghold token renew <token>
```

Подробнее о сроках действия и продлении см. раздел [Аренда](../Аренда/).

## Что важно запомнить

- Перед любой работой с Stronghold необходимо пройти аутентификацию.
- После успешной аутентификации Stronghold создаёт собственный токен.
- Именно токен и прикреплённые к нему политики определяют доступные действия.
- Один и тот же Stronghold может одновременно поддерживать несколько методов аутентификации.
- Для пользователей и сервисов обычно применяются разные методы аутентификации.
- У результата аутентификации есть срок действия, который нужно учитывать в работе и автоматизации.
