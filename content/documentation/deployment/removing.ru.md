В этом разделе описано удаление **Deckhouse Stronghold** для двух вариантов развёртывания:

- **Standalone**;
- **в DKP**.

> Предупреждение  
> Перед удалением Deckhouse Stronghold рекомендуется создать резервную копию данных, unseal-ключей, root-токена и всех связанных конфигураций. После удаления доступ к данным может быть утрачен без возможности восстановления.

## Удаление Standalone

Этот сценарий подходит для инсталляций, где Deckhouse Stronghold был установлен как отдельный сервис в Linux и запускается через `systemd`.

### Перед удалением

Перед началом работ рекомендуется:

- проверить, что у вас есть актуальная резервная копия данных;
- сохранить конфигурационный файл;
- сохранить TLS-сертификаты и ключи, если они могут понадобиться повторно;
- убедиться, что удаление действительно допустимо для вашей среды.

Если используется встроенное backend-хранилище `Raft`, до удаления рекомендуется дополнительно сохранить снимок хранилища.

### Шаг 1. Остановите сервис

Остановите сервис Stronghold:

```bash
sudo systemctl stop stronghold
```

Проверьте, что сервис остановлен:

```bash
sudo systemctl status stronghold
```

### Шаг 2. Отключите автозапуск

Отключите запуск сервиса при старте системы:

```bash
sudo systemctl disable stronghold
```

### Шаг 3. Удалите unit-файл systemd

Удалите unit-файл сервиса:

```bash
sudo rm -f /etc/systemd/system/stronghold.service
sudo systemctl daemon-reload
```

При необходимости выполните:

```bash
sudo systemctl reset-failed
```

### Шаг 4. Удалите бинарный файл и конфигурацию

Удалите бинарный файл Stronghold, конфигурацию и служебные каталоги, если они больше не нужны.

Пример:

```bash
sudo rm -f /opt/stronghold/stronghold
sudo rm -f /opt/stronghold/config.hcl
```

Если в вашей инсталляции используются другие пути, удалите файлы по фактическим расположениям.

### Шаг 5. При необходимости удалите TLS-материалы

Если TLS-сертификаты и ключи больше не нужны, удалите их:

```bash
sudo rm -rf /opt/stronghold/tls
```

> Предупреждение  
> Удаляйте TLS-материалы только в том случае, если они не используются другими сервисами и не потребуются для повторного развёртывания.

### Шаг 6. При необходимости удалите данные Stronghold

Если необходимо полностью удалить данные Stronghold, удалите каталог данных:

```bash
sudo rm -rf /opt/stronghold/data
```

> Предупреждение  
> Это разрушительная операция. После удаления каталога данных восстановление будет возможно только из заранее созданной резервной копии или снимка.

### Шаг 7. Проверьте результат

Убедитесь, что сервис больше не зарегистрирован в `systemd`:

```bash
systemctl status stronghold
```

Также проверьте, что бинарный файл, конфигурация и данные удалены, если это входило в ваш сценарий.

## Удаление в DKP

В этом сценарии удаление Stronghold выполняется как отключение платформенного модуля `stronghold`.

> Важно  
> При отключении модуля удаляются контейнеры Stronghold из namespace `d8-stronghold`, а также секрет `stronghold-keys` с root-токеном и unseal-ключами. При этом данные сервиса на узлах не удаляются автоматически.

### Перед удалением

Перед отключением модуля рекомендуется:

- сохранить секрет `stronghold-keys`;
- создать резервную копию или снимок данных Stronghold;
- убедиться, что данные больше не используются приложениями и пользователями;
- проверить, используется ли [`secrets-store-integration`](/modules/secrets-store-integration/stable/) для доставки секретов в приложения.

Сохраните секрет `stronghold-keys`:

```bash
d8 k -n d8-stronghold get secret stronghold-keys -o yaml > stronghold-keys.yaml
chmod 600 stronghold-keys.yaml
```

При необходимости создайте снимок Stronghold:

```bash
export STRONGHOLD_ADDR=https://$(d8 k -n d8-stronghold get ing stronghold -o json | jq -r '.spec.rules[0].host')
d8 stronghold login -method=oidc -path=oidc_deckhouse
# Либо через root-токен:
# d8 stronghold login -method=token
d8 stronghold operator raft snapshot save stronghold-$(date +%F_%H-%M).snap
```

### Шаг 1. Отключите модуль Stronghold

Отключите модуль через CLI:

```bash
d8 system module disable stronghold
```

При необходимости модуль можно также отключить через веб-интерфейс DKP, выключив переключатель **Модуль включен** в настройках модуля.

Если управление конфигурацией выполняется декларативно, можно отключить модуль через `ModuleConfig`:

```yaml
apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: stronghold
spec:
  enabled: false
```

Примените конфигурацию:

```bash
d8 k apply -f stronghold-moduleconfig.yaml
```

### Шаг 2. Дождитесь применения изменений

Проверьте очередь задач DKP:

```bash
d8 system queue main
```

Дождитесь завершения обработки изменений.

### Шаг 3. Проверьте, что компоненты Stronghold удалены

Проверьте, что поды Stronghold больше не работают:

```bash
d8 k -n d8-stronghold get po
```

Проверьте, что секрет `stronghold-keys` удалён:

```bash
d8 k -n d8-stronghold get secret stronghold-keys
```

Если секрет был удалён, команда завершится ошибкой `NotFound`, что в данном случае является ожидаемым результатом.

### Шаг 4. При необходимости сохраните возможность восстановления

Если планируется повторное включение Stronghold с восстановлением доступа к старым данным, сохраните:

- файл `stronghold-keys.yaml`;
- резервную копию или снимок Stronghold;
- сведения о лицензии и конфигурации модуля.

После повторного включения модуля сохранённый секрет `stronghold-keys` можно будет вернуть в namespace `d8-stronghold`, чтобы восстановить доступ к данным.

### Шаг 5. При необходимости полностью удалите данные с узлов

Если старые данные Stronghold больше не нужны, удалите каталог:

```bash
sudo rm -rf /var/lib/deckhouse/stronghold
```

Удаление необходимо выполнить на всех узлах, где размещались данные Stronghold.

> Предупреждение  
> Удаление каталога `/var/lib/deckhouse/stronghold` приводит к безвозвратному удалению данных Stronghold с узлов. Выполняйте этот шаг только при полной уверенности, что данные больше не потребуются.

## Проверка результата

После удаления проверьте:

- сервис Stronghold больше не запущен;
- модуль `stronghold` отключён, если использовался DKP;
- поды Stronghold отсутствуют;
- доступ к веб-интерфейсу Stronghold более недоступен;
- при полном удалении данные и конфигурация удалены из целевых каталогов.
