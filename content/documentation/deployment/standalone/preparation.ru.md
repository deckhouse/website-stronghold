В этом разделе описана подготовка окружения для **Standalone-развёртывания Deckhouse Stronghold**.

Подготовка включает:
- проверку совместимости операционной системы;
- выбор топологии развёртывания;
- подготовку хоста и каталогов;
- подготовку backend-хранилища;
- настройку сети;
- подготовку TLS-сертификатов;
- подготовку запуска через `systemd`.

> Примечание  
> Этот раздел посвящён подготовительным действиям. Пошаговая установка и запуск Deckhouse Stronghold описаны в разделе [Установка](../installation/).

## Выбор варианта развёртывания

Перед началом работ определите, как будет развернут Deckhouse Stronghold:

- **на одном сервере** — подходит для разработки, тестирования и пилотных внедрений;
- **в кластере с высокой доступностью (HA)** — подходит для production-окружений;
- **с интегрированным backend-хранилищем Raft** — типовой вариант для отказоустойчивой конфигурации.

Если планируется эксплуатация в режиме высокой доступности с использованием `Raft`, рекомендуется подготовить как минимум **три сервера Stronghold**. Это необходимо для достижения кворума.

## Проверка системных требований

Перед установкой убедитесь, что целевые серверы соответствуют требованиям продукта:

- используется поддерживаемая операционная система;
- доступны необходимые вычислительные ресурсы;
- дисковая подсистема обеспечивает требуемые IOPS и пропускную способность;
- отключены или учтены факторы, которые могут негативно повлиять на работу backend-хранилища.

Подробные требования приведены в разделе [Требования](../../about/system-requirements/).

## Поддерживаемая операционная система

Убедитесь, что на сервере установлена поддерживаемая ОС и версия. Для Standalone-развёртывания должны использоваться только поддерживаемые Linux-дистрибутивы.

> Важно  
> Если вы развёртываете кластер из нескольких узлов, рекомендуется использовать одинаковую операционную систему и единообразные системные настройки на всех серверах.

## Подготовка серверов

На каждом сервере, где будет запущен Deckhouse Stronghold, рекомендуется заранее выполнить следующие действия:

- создать отдельного системного пользователя для запуска сервиса, например `stronghold`;
- подготовить директории для бинарного файла, данных, конфигурации и TLS-материалов;
- проверить права доступа к этим директориям;
- убедиться, что серверы синхронизируют системное время;
- при необходимости отключить swap или убедиться, что его использование соответствует вашей политике безопасности.

Типовой набор директорий может выглядеть так:

- `/opt/stronghold` — рабочая директория продукта;
- `/opt/stronghold/data` — данные backend-хранилища;
- `/opt/stronghold/tls` — TLS-сертификаты и ключи;
- `/etc/systemd/system/stronghold.service` — unit-файл `systemd`.

## Подготовка backend-хранилища

Deckhouse Stronghold требует backend-хранилище для сохранения конфигурации, служебного состояния, политик, токенов, метаданных и секретов.

Для Standalone-развёртывания чаще всего используются следующие варианты:

- **`filesystem`** — подходит для локальной разработки и сценариев без высокой доступности;
- **`raft`** — рекомендуется для production-окружений и конфигураций с высокой доступностью.

Если планируется использование `Raft`, заранее определите:

- список узлов кластера;
- уникальный `node_id` для каждого узла;
- адреса, которые будут использоваться как `api_addr` и `cluster_addr`;
- параметры сетевого взаимодействия между узлами;
- расположение каталога данных.

> Важно  
> Для production-окружений рекомендуется заранее проверить производительность и надёжность выбранного backend-хранилища, а также определить порядок резервного копирования и восстановления.

## Подготовка сети

Перед установкой необходимо проверить сетевую связность между узлами и клиентами.

Обычно требуется:

- доступ к API Deckhouse Stronghold по TCP-порту `8200`;
- доступ между узлами кластера по TCP-порту `8201`, если используется `Raft`;
- корректное разрешение DNS-имён или наличие стабильных IP-адресов;
- отсутствие сетевых ограничений, мешающих сервер-серверному взаимодействию.

Если на серверах используется файрвол, заранее откройте необходимые порты.

> Примечание  
> Если в вашей конфигурации используются нестандартные порты, это должно быть отражено в конфигурационном файле Stronghold и в правилах сетевого доступа.

## Подготовка TLS-сертификатов

Для безопасной эксплуатации Deckhouse Stronghold рекомендуется использовать TLS на всех внешних и межузловых соединениях.

Заранее подготовьте:

- сертификат корневого центра сертификации (CA), если используется собственная PKI;
- TLS-сертификат для каждого узла;
- закрытый ключ для каждого сертификата;
- корректные значения `subjectAltName` (`SAN`) для всех DNS-имён и IP-адресов, которые будут использоваться при подключении к узлам.

Если развёртывается кластер `Raft`, каждый узел должен иметь собственный сертификат, соответствующий его сетевой идентичности.

Типовой набор файлов на узле:

- сертификат узла;
- закрытый ключ узла;
- сертификат CA.

> Предупреждение  
> Самоподписанные сертификаты допустимы только для тестовых сценариев. Для production-окружений рекомендуется использовать сертификаты, подписанные доверенным центром сертификации.

## Подготовка запуска через systemd

Для эксплуатации в Linux рекомендуется запускать Deckhouse Stronghold как сервис `systemd`.

Перед установкой подготовьте:

- unit-файл `systemd`;
- системного пользователя и группу, от имени которых будет работать сервис;
- права доступа к рабочим каталогам;
- параметры рестарта сервиса;
- ограничения и capability, необходимые для безопасного запуска.

Рекомендуется заранее определить:

- путь к бинарному файлу `stronghold`;
- путь к конфигурационному файлу;
- политику автозапуска сервиса;
- требования к журналированию и перезапуску при сбоях.

## Подготовка конфигурации

Deckhouse Stronghold настраивается с помощью конфигурационного файла в формате HCL или JSON.

На этапе подготовки рекомендуется определить:

- будет ли включён веб-интерфейс (`ui`);
- какой backend-хранилище будет использоваться;
- какие адреса будут заданы в `api_addr` и `cluster_addr`;
- какой `listener` будет принимать подключения;
- где будут расположены TLS-сертификаты и ключи;
- какие параметры логирования и телеметрии потребуются;
- нужно ли использовать каталог плагинов;
- требуется ли настройка блокировки пользователей (`user_lockout`).

> Примечание  
> Подробное описание параметров конфигурации и их значений лучше использовать на этапе администрирования и тонкой настройки. Для установки достаточно заранее определить минимально необходимую конфигурацию запуска.

## Рекомендуемый чек-лист перед установкой

Перед переходом к установке убедитесь, что выполнены следующие условия:

- выбрана целевая топология развёртывания;
- подтверждена совместимость ОС;
- подготовлены серверы и директории;
- выбран и подготовлен backend-хранилище;
- проверена сетевая связность;
- открыты необходимые TCP-порты;
- подготовлены TLS-сертификаты и ключи;
- определены `api_addr`, `cluster_addr` и `node_id` для каждого узла;
- подготовлен unit-файл `systemd`;
- бинарный файл Deckhouse Stronghold доступен на целевых серверах;
- определено место хранения конфигурационного файла.
