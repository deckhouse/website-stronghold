## Первичная настройка

После установки и первого запуска **Deckhouse Stronghold** рекомендуется выполнить базовую настройку сервиса: проверить конфигурационный файл, включить необходимые параметры безопасности, настроить веб-интерфейс, логирование и убедиться, что узлы корректно работают в выбранной топологии.

В этом разделе рассматривается первичная настройка Standalone-развёртывания с backend-хранилищем `Raft`.

> Примечание  
> Подробное описание всех параметров конфигурации будет приведено в разделе **«Руководство администратора → Конфигурация»**.

## Где хранится конфигурация

Серверы Stronghold настраиваются с помощью конфигурационного файла в формате **HCL** или **JSON**. В примерах этого руководства используется файл:

```bash
/opt/stronghold/config.hcl
```

После изменения конфигурации примените изменения одним из способов:

- перезапустите сервис:

  ```bash
  systemctl restart stronghold
  ```

- либо отправьте процессу сигнал `SIGHUP`, если изменение поддерживает reload без полного перезапуска:

  ```bash
  systemctl reload stronghold
  ```

> Важно  
> Не все параметры применяются без перезапуска процесса. Если после `reload` изменения не вступили в силу, выполните полный `restart`.

## Базовая структура конфигурации

Минимальная конфигурация Standalone-инстанса с `Raft` обычно включает:

- включение веб-интерфейса;
- настройки `api_addr` и `cluster_addr`;
- секцию `listener`;
- секцию `storage`;
- параметры TLS;
- базовые параметры логирования и безопасности.

Пример стартовой конфигурации:

```hcl
ui = true
api_addr = "https://10.20.30.10:8200"
cluster_addr = "https://10.20.30.10:8201"
disable_mlock = true
log_level = "info"
log_format = "standard"

listener "tcp" {
  address       = "0.0.0.0:8200"
  tls_cert_file = "/opt/stronghold/tls/node-1-cert.pem"
  tls_key_file  = "/opt/stronghold/tls/node-1-key.pem"
  tls_min_version = "tls12"
}

storage "raft" {
  path    = "/opt/stronghold/data"
  node_id = "raft-node-1"
}
```

## Настройка веб-интерфейса

Чтобы включить встроенный веб-интерфейс, установите параметр:

```hcl
ui = true
```

После этого интерфейс будет доступен на том же адресе и порту, что и API, по пути `/ui/`.

Например:

```text
https://10.20.30.10:8200/ui/
```

Если доступ к Stronghold осуществляется по DNS-имени, используйте соответствующий URL.

> Примечание  
> Для корректной работы веб-интерфейса TLS должен быть настроен правильно, а сертификат должен соответствовать адресу, по которому открывается UI.

## Настройка listener

Секция `listener` определяет адрес и порт, на которых Stronghold принимает подключения.

Для базового Standalone-сценария обычно используется TCP-слушатель:

```hcl
listener "tcp" {
  address       = "0.0.0.0:8200"
  tls_cert_file = "/opt/stronghold/tls/node-1-cert.pem"
  tls_key_file  = "/opt/stronghold/tls/node-1-key.pem"
}
```

Рекомендуется:

- использовать TLS для всех подключений;
- не отключать TLS в production-окружении;
- проверять, что сертификат соответствует DNS-именам и IP-адресам узла;
- использовать только необходимые сетевые интерфейсы.

Если сервис должен быть доступен только локально или через reverse proxy, настройте `address` соответствующим образом.

## Настройка backend-хранилища

Для Standalone-развёртывания с высокой доступностью рекомендуется использовать интегрированное backend-хранилище `Raft`.

Базовая конфигурация:

```hcl
storage "raft" {
  path    = "/opt/stronghold/data"
  node_id = "raft-node-1"
}
```

При использовании `Raft` необходимо:

- задать уникальный `node_id` для каждого узла;
- использовать корректный каталог данных;
- проверить права доступа к каталогу;
- убедиться, что между узлами доступен `cluster_addr`.

Если используется кластерная конфигурация, в `storage "raft"` дополнительно настраиваются блоки `retry_join`.

Пример:

```hcl
storage "raft" {
  path    = "/opt/stronghold/data"
  node_id = "raft-node-1"

  retry_join {
    leader_tls_servername   = "raft-node-1.demo.tld"
    leader_api_addr         = "https://10.20.30.10:8200"
    leader_ca_cert_file     = "/opt/stronghold/tls/stronghold-ca.pem"
    leader_client_cert_file = "/opt/stronghold/tls/node-1-cert.pem"
    leader_client_key_file  = "/opt/stronghold/tls/node-1-key.pem"
  }
}
```

## Настройка api_addr и cluster_addr

Параметры `api_addr` и `cluster_addr` важны для корректной работы кластера, перенаправления клиентов и сервер-серверного взаимодействия.

Пример:

```hcl
api_addr = "https://10.20.30.10:8200"
cluster_addr = "https://10.20.30.10:8201"
```

Рекомендации:

- используйте реальные адреса, доступные другим узлам и клиентам;
- не указывайте адреса, которые недоступны из целевой сети;
- следите, чтобы значения совпадали с используемой сетевой схемой и TLS-сертификатами.

> Важно  
> Неправильно заданные `api_addr` и `cluster_addr` могут привести к ошибкам присоединения узлов к кластеру, проблемам с переадресацией запросов и некорректной работе HA.

## Настройка логирования

Для базовой эксплуатации рекомендуется явно указать параметры логирования.

Пример:

```hcl
log_level  = "info"
log_format = "standard"
```

Поддерживаются уровни логирования:

- `trace`
- `debug`
- `info`
- `warn`
- `error`

Рекомендации:

- для обычной эксплуатации используйте `info`;
- для диагностики временно переключайте уровень на `debug`;
- не оставляйте `trace` или `debug` включёнными в production без необходимости.

При необходимости можно настроить вывод логов в файл, но на этапе первичной настройки обычно достаточно журналов `systemd`.

Проверить журналы можно командой:

```bash
journalctl -u stronghold -f
```

## Настройка mlock и swap

Параметр `disable_mlock` управляет использованием системного вызова `mlock`, который предотвращает выгрузку памяти на диск.

Для конфигураций с `Raft` обычно используется:

```hcl
disable_mlock = true
```

Это связано с тем, что при использовании встроенного backend-хранилища включённый `mlock` может привести к нежелательному росту потребления памяти.

Рекомендации:

- для `Raft` используйте `disable_mlock = true`;
- по возможности отключите swap;
- если swap используется, оцените риски с точки зрения безопасности;
- не меняйте этот параметр без понимания влияния на целевую конфигурацию.

## Проверка прав доступа к файлам

Для повышения безопасности рекомендуется проверить:

- владельца конфигурационного файла;
- владельца каталога данных;
- владельца TLS-сертификатов и ключей;
- права доступа к закрытым ключам.

Типовая рекомендация:

- сервис запускается от отдельного системного пользователя;
- только этот пользователь имеет доступ к критичным файлам;
- группа и остальные пользователи не имеют прав на запись в конфигурацию и TLS-материалы.

Пример:

```bash
chown -R stronghold:stronghold /opt/stronghold
chmod 0700 /opt/stronghold/data
chmod 0700 /opt/stronghold/tls
```

> Предупреждение  
> Некорректные права доступа к конфигурации, данным или TLS-ключам могут привести как к проблемам запуска сервиса, так и к рискам компрометации чувствительных данных.

## Базовые рекомендации по безопасности

После установки рекомендуется проверить следующие настройки:

- включён TLS для внешних подключений;
- используются корректные сертификаты;
- сервис запущен не от `root`, а от отдельного системного пользователя;
- каталог данных защищён от доступа посторонних пользователей;
- закрытые ключи TLS доступны только сервисному пользователю;
- swap отключён или его использование осознанно контролируется;
- уровень логирования соответствует среде эксплуатации;
- веб-интерфейс включён только если он действительно требуется.

## Применение изменений

После обновления `config.hcl`:

1. проверьте синтаксис и корректность изменений;
2. примените конфигурацию;
3. убедитесь, что сервис успешно запустился;
4. проверьте состояние узла или кластера.

Базовые команды:

```bash
systemctl restart stronghold
systemctl status stronghold
stronghold status -ca-cert /opt/stronghold/tls/stronghold-ca.pem
```

Если узел входит в кластер `Raft`, дополнительно проверьте, что он видит актуальное состояние HA и участвует в работе кластера.
