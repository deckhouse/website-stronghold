## Проверка работоспособности

После установки и первичной настройки **Deckhouse Stronghold** рекомендуется выполнить базовую проверку работоспособности. Это позволит убедиться, что сервис запущен, принимает запросы, хранилище инициализировано, а операции чтения и записи выполняются корректно.

В этом разделе приведён минимальный набор проверок для двух вариантов развёртывания:

- **Standalone**;
- **в DKP**.

## Что считается успешной проверкой

Установку можно считать успешной, если выполняются следующие условия:

- сервис Stronghold запущен;
- API отвечает без ошибок;
- хранилище инициализировано и распечатано;
- веб-интерфейс доступен, если он используется;
- аутентификация проходит успешно;
- тестовая операция записи и чтения секрета выполняется без ошибок.

## Проверка в Standalone

### Шаг 1. Проверьте состояние сервиса

Убедитесь, что сервис `stronghold` запущен:

```bash
systemctl status stronghold
```

Если используется `systemd`, сервис должен находиться в состоянии `active (running)`.

При необходимости проверьте журналы:

```bash
journalctl -u stronghold -f
```

### Шаг 2. Проверьте состояние хранилища

Проверьте статус Stronghold:

```bash
stronghold status -ca-cert /opt/stronghold/tls/stronghold-ca.pem
```

Пример ожидаемого результата:

```text
Key                     Value
---                     -----
Seal Type               shamir
Initialized             true
Sealed                  false
Storage Type            raft
HA Enabled              true
```

Проверьте, что:
- `Initialized` равно `true`;
- `Sealed` равно `false`;
- тип хранилища соответствует ожидаемому;
- при кластерной установке включён HA.

### Шаг 3. Проверьте доступность API

Проверьте доступность health-эндпоинта:

```bash
curl --cacert /opt/stronghold/tls/stronghold-ca.pem \
  https://127.0.0.1:8200/v1/sys/health
```

Если обращение выполняется к удалённому узлу, используйте его фактический адрес или DNS-имя.

Ожидаемый результат:
- сервер отвечает;
- возвращается JSON-ответ;
- в ответе отсутствуют критические ошибки соединения или TLS.

### Шаг 4. Проверьте вход в Stronghold

Выполните вход с использованием root-токена или другого административного токена.

Пример с переменной окружения:

```bash
export STRONGHOLD_TOKEN=<ROOT_TOKEN>
export STRONGHOLD_ADDR=https://127.0.0.1:8200
```

После этого повторно проверьте состояние:

```bash
stronghold status -ca-cert /opt/stronghold/tls/stronghold-ca.pem
```

Если вход выполнен успешно, команды CLI должны выполняться без ошибок авторизации.

### Шаг 5. Выполните тестовую запись секрета

Создайте тестовый секрет в `KV`:

```bash
stronghold kv put -ca-cert /opt/stronghold/tls/stronghold-ca.pem secret/check value=test
```

Ожидаемый результат:
- команда завершается успешно;
- секрет сохраняется без ошибок.

### Шаг 6. Выполните тестовое чтение секрета

Прочитайте ранее сохранённый секрет:

```bash
stronghold kv get -ca-cert /opt/stronghold/tls/stronghold-ca.pem secret/check
```

Убедитесь, что:
- команда выполняется успешно;
- в выводе присутствует значение `test`.

### Шаг 7. При необходимости проверьте веб-интерфейс

Если веб-интерфейс включён, откройте адрес:

```text
https://<STRONGHOLD_ADDRESS>:8200/ui/
```

Проверьте, что:
- страница открывается;
- сертификат соответствует ожидаемому сценарию доступа;
- вход в интерфейс выполняется успешно.

## Проверка в DKP

### Шаг 1. Проверьте состояние namespace и подов

Убедитесь, что namespace `d8-stronghold` существует:

```bash
d8 k get ns d8-stronghold
```

Проверьте состояние подов:

```bash
d8 k -n d8-stronghold get po
```

Убедитесь, что:
- поды созданы;
- отсутствуют состояния `CrashLoopBackOff`, `ImagePullBackOff`, `ErrImagePull`;
- поды находятся в состоянии `Running` и готовы к работе.

### Шаг 2. Проверьте состояние Ingress и сервиса

Если Stronghold публикуется через Ingress, проверьте соответствующие ресурсы:

```bash
d8 k -n d8-stronghold get ing,svc
```

Если используется внешний Ingress-контроллер, дополнительно проверьте его состояние:

```bash
d8 k -n d8-ingress-nginx get po
```

### Шаг 3. Проверьте доступность API

Если Stronghold опубликован через Ingress, проверьте health-эндпоинт:

```bash
curl -kL https://stronghold.<ваш-домен>/v1/sys/health
```

Ожидаемый результат:
- эндпоинт доступен;
- возвращается корректный JSON-ответ;
- отсутствуют ошибки TLS, DNS или маршрутизации.

### Шаг 4. Проверьте доступность Dex

Если используется OIDC-аутентификация через Dex, проверьте его доступность:

```bash
curl -kL https://dex.<ваш-домен>/healthz
```

Ожидаемый ответ:

```text
Health check passed
```

### Шаг 5. Проверьте первый вход

Откройте веб-интерфейс Stronghold по адресу вашей инсталляции и выполните вход с административной учётной записью.

Убедитесь, что:
- открывается форма входа;
- аутентификация через OIDC выполняется без ошибок;
- после входа доступны административные разделы интерфейса.

Если используется CLI-доступ, выполните аутентификацию через `d8 stronghold login`:

```bash
export STRONGHOLD_ADDR=https://stronghold.<ваш-домен>
d8 stronghold login -method=oidc -path=oidc_deckhouse
```

Либо выполните вход с использованием токена:

```bash
d8 stronghold login -method=token
```

### Шаг 6. Выполните тестовую запись секрета

После успешной аутентификации создайте тестовый секрет:

```bash
d8 stronghold kv put secret/check value=test
```

Ожидаемый результат:
- операция выполняется успешно;
- секрет сохраняется в хранилище.

### Шаг 7. Выполните тестовое чтение секрета

Проверьте, что сохранённый секрет читается:

```bash
d8 stronghold kv get secret/check
```

Убедитесь, что:
- команда выполняется без ошибок;
- в выводе присутствует ожидаемое значение `test`.

## Очистка тестовых данных

После проверки удалите тестовый секрет:

### Standalone

```bash
stronghold kv delete -ca-cert /opt/stronghold/tls/stronghold-ca.pem secret/check
```

### DKP

```bash
d8 stronghold kv delete secret/check
```

## Что делать, если проверка не пройдена

Если один из шагов завершается ошибкой, проверьте:

- состояние сервиса или подов;
- журналы Stronghold;
- корректность TLS-сертификатов;
- доступность DNS-имён;
- сетевую связность;
- корректность аутентификации;
- наличие прав доступа на выполнение операций с секретами.

Дополнительные материалы по диагностике и устранению проблем приведены в разделе **«Руководство администратора → Диагностика»**.
