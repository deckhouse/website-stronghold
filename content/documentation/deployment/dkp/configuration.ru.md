После установки **Deckhouse Stronghold** в **Deckhouse Kubernetes Platform** рекомендуется выполнить базовую проверку доступа, настроить администраторов и убедиться, что веб-интерфейс и интеграции работают корректно.

В этом разделе рассматриваются действия, которые рекомендуется выполнить сразу после включения модуля `stronghold`.

## Шаг 1. Проверьте доступ к кластеру DKP

После завершения установки убедитесь, что у вас есть доступ к кластеру DKP.

Подключиться к платформе можно:
- напрямую с master-узла;
- удалённо с предварительно настроенного рабочего компьютера.

### Подключение с master-узла

Подключитесь к master-узлу по SSH:

```bash
ssh <USER_NAME>@<MASTER_IP>
```

Проверьте, что ресурсы кластера доступны:

```bash
d8 k get nodes
```

Если команда возвращает список узлов, базовый доступ к кластеру настроен корректно.

### Удалённое подключение

При необходимости настройте удалённый доступ к кластеру с рабочего компьютера. Для этого используйте механизмы доступа, принятые в вашей инсталляции DKP.

> Примечание  
> Для повседневной работы с Stronghold в DKP рекомендуется использовать удалённый доступ с корректно настроенным kubeconfig и правами пользователя.

## Шаг 2. Убедитесь, что модуль Stronghold включён

Модуль `stronghold` можно включить несколькими способами:

- декларативно, через `ModuleConfig`;
- через CLI;
- через веб-интерфейс DKP.

### Включение через CLI

Чтобы включить модуль, выполните команду:

```bash
d8 system module enable stronghold
```

По умолчанию модуль запускается в режиме `Automatic` с инлетом `Ingress`.

### Включение через веб-интерфейс DKP

Модуль можно включить в веб-интерфейсе DKP:

1. Откройте раздел **Модули**.
2. Найдите модуль **stronghold**.
3. Включите переключатель **Модуль включен**.

### Включение через ModuleConfig

Если требуется декларативное управление конфигурацией, используйте `ModuleConfig`:

```yaml
apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: stronghold
spec:
  enabled: true
  version: 1
  settings:
    management:
      mode: Automatic
```

Примените конфигурацию:

```bash
d8 k apply -f stronghold-moduleconfig.yaml
```

> Примечание  
> Если в вашей инфраструктуре используется GitOps- или декларативный подход к управлению платформой, предпочтительно включать модуль через `ModuleConfig`.

После включения модуля проверьте, что namespace `d8-stronghold` создан:

```bash
d8 k get ns d8-stronghold
```

Проверьте состояние подов Stronghold:

```bash
d8 k -n d8-stronghold get po
```

Убедитесь, что поды созданы и перешли в состояние `Running` или `Ready`.

При необходимости проверьте связанные ресурсы:

```bash
d8 k -n d8-stronghold get svc,ingress,secret
```

## Как Stronghold работает в DKP

В DKP Stronghold разворачивается как платформенный компонент и по умолчанию работает в режиме управления `Automatic`.

В этом режиме:

- хранилище автоматически инициализируется при первом запуске;
- root-токен и unseal-ключи сохраняются в секрете `stronghold-keys` в namespace `d8-stronghold`;
- узлы Stronghold автоматически распечатываются после запуска;
- для администраторов настраивается доступ через OIDC-аутентификацию с использованием **Dex**;
- текущий кластер DKP автоматически подготавливается для интеграции с `secrets-store-integration`.

> Важно  
> Несмотря на автоматизацию, рекомендуется заранее определить, кто будет иметь административный доступ к Stronghold, и проверить, как в вашей организации хранится и защищается доступ к чувствительным данным.

## Шаг 3. Настройте доступ к веб-интерфейсу

Доступ к веб-интерфейсу Stronghold в DKP обычно организуется через **Ingress**.

Адрес Stronghold формируется на основе шаблона `publicDomainTemplate`: в нём значение `%s` заменяется на `stronghold`.

Например, если:

```text
%s.kube.example.com
```

то веб-интерфейс Stronghold будет доступен по адресу:

```text
https://stronghold.kube.example.com
```

### Что нужно проверить

Убедитесь, что:

- Ingress-контроллер работает;
- доменное имя Stronghold корректно разрешается;
- сертификат для Ingress успешно выпущен или подключён;
- адрес Dex также доступен, если используется вход через OIDC.

Проверьте Ingress-контроллер:

```bash
d8 k -n d8-ingress-nginx get po
```

Проверьте Ingress Stronghold:

```bash
d8 k -n d8-stronghold get ing
```

Если используется внешний балансировщик, проверьте его адрес:

```bash
d8 k -n d8-ingress-nginx get svc nginx-load-balancer
```

### Быстрая проверка доступа

Если доменное имя уже настроено, проверьте доступность health-эндпоинта Stronghold:

```bash
curl -kL https://stronghold.<ваш-домен>/v1/sys/health
```

Если используется OIDC-аутентификация через Dex, также проверьте доступность Dex:

```bash
curl -kL https://dex.<ваш-домен>/healthz
```

Ожидаемый результат для Dex:

```text
Health check passed
```

> Примечание  
> Если при обращении к адресу Stronghold вы получаете ошибку `404`, это может означать, что Ingress уже обрабатывает запрос, но приложение или маршрут ещё не готовы. Для первичной проверки это допустимый промежуточный результат.

## Шаг 4. Назначьте администраторов Stronghold

В режиме `Automatic` Stronghold создаёт роль `deckhouse_administrators`, для которой настраивается доступ через OIDC-аутентификацию с использованием **Dex**.

Чтобы выдать административные права пользователям или группам, добавьте их в массив `administrators` ресурса `ModuleConfig` `stronghold`.

### Пример: назначение группы администраторов

Если пользователи входят в группу `admins`, передаваемую из внешнего identity-провайдера или LDAP через **Dex**, используйте следующую конфигурацию:

```yaml
apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: stronghold
spec:
  enabled: true
  version: 1
  settings:
    management:
      mode: Automatic
      administrators:
      - type: Group
        name: admins
```

Примените изменения:

```bash
d8 k apply -f stronghold-moduleconfig.yaml
```

### Пример: назначение отдельных пользователей

Если нужно выдать права конкретным пользователям:

```yaml
apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: stronghold
spec:
  enabled: true
  version: 1
  settings:
    management:
      mode: Automatic
      administrators:
      - type: User
        name: manager@mycompany.tld
      - type: User
        name: securityoperator@mycompany.tld
```

> Важно  
> Даже если доступ назначается отдельным пользователям, для OIDC-аутентификации через Dex они должны корректно разрешаться в используемой системе идентификации и, как правило, входить хотя бы в одну группу.

После обновления `ModuleConfig` дождитесь применения изменений:

```bash
d8 system queue main
```

## Шаг 5. Проверьте секрет stronghold-keys

В автоматическом режиме Stronghold сохраняет root-токен и unseal-ключи в секрете `stronghold-keys` в namespace `d8-stronghold`.

Проверьте наличие секрета:

```bash
d8 k -n d8-stronghold get secret stronghold-keys
```

> Предупреждение  
> Секрет `stronghold-keys` содержит критически важные данные доступа. Убедитесь, что доступ к нему ограничен, а порядок его резервного копирования и хранения соответствует требованиям безопасности вашей организации.

## Шаг 6. Определите, нужен ли secrets-store-integration

Если Stronghold в DKP используется только как хранилище секретов, API или административный сервис, модуль [`secrets-store-integration`](/modules/secrets-store-integration/stable/) не является строго обязательным.

Однако если ваша цель — безопасная и автоматизированная доставка секретов в приложения Kubernetes, этот модуль необходим.

Он обеспечивает:

- доставку секретов в поды через CSI-драйвер;
- инъекцию секретов через Mutating Webhook;
- синхронизацию данных из Stronghold в native Secrets Kubernetes.

> Важно  
> Если секреты должны автоматически доставляться в приложения без их хранения в стандартных секретах Kubernetes, `secrets-store-integration` должен быть включён.

Если модуль уже был включён на этапе установки, дополнительно ничего делать не нужно. Если нет — включите его отдельно в соответствии с вашими сценариями.

## Шаг 7. Проверьте первый вход

После назначения администраторов и настройки доступа к Ingress откройте веб-интерфейс Stronghold по адресу вашей инсталляции.

На первом входе убедитесь, что:

- открывается страница аутентификации;
- вход через OIDC выполняется корректно;
- пользователь получает административный доступ;
- доступны основные разделы веб-интерфейса.

Если доступ через веб-интерфейс не требуется, проверьте возможность обращения к Stronghold через API или CLI.

## Что делать, если доступ не работает

Если после установки Stronghold не открывается, проверьте по порядку:

- состояние namespace `d8-stronghold`;
- состояние подов Stronghold;
- наличие и состояние Ingress;
- наличие TLS-секрета для Ingress;
- доступность доменного имени;
- доступность Dex;
- корректность конфигурации `publicDomainTemplate`;
- назначение администраторов в `ModuleConfig`.

Дополнительно проверьте журналы подов:

```bash
d8 k -n d8-stronghold logs <имя-пода>
```
